## A. 假設校準（原有設計的前提重新檢視）

### A1. Context Rot 嚴重度 `HOLD`

**原假設：** 全自動有 context rot + 品質漂移風險，所以 P3 採半自動 + 高密度 stop gate。

**審視：** Opus 4.6 支援 1M context，內建 compaction 機制，長 session 指令遵循度已大幅提升。SG2「不可省略」在 20 task 的功能中 = 20 次中斷，trivial task（DTO 欄位、i18n key）幾乎是純摩擦。

**決定：** 先維持半自動觀察。在第一個實際專案試用時，記錄每次 SG2 中斷是否真的抓到問題。若連續 N 次 SG2 都直接放行，再回來調整密度策略。

**待觀察指標：**
- SG2 實際攔截率（PG 真的修改了測試清單的比例）
- SG2 攔截的問題嚴重度分布
- session 長度與品質漂移的相關性

### A2. 「讀規範、不掃 code」的邊界 `DISCUSS`

**原假設：** 不掃 code 歸納 pattern（AI 行為會漂移），改讀 CLAUDE.md 索引揭露的規範文件（deterministic）。

**審視：** 原則正確，但太二元。CLAUDE.md 不可能涵蓋所有細節（API response 格式、component prop 介面、test fixture 結構）。P3 實作時如果不能讀既有程式碼作為參考，就只能「猜」或要求 CLAUDE.md 寫到極度詳盡（成本太高）。

**但有一個現實問題：** 目前專案大家都用 AI 寫 code，實作風格已經不一致。如果隨便讀既有程式碼作為 reference，可能傳播壞 pattern。「讀哪支檔案」本身就是一個需要管控的決策。

**方向（待討論細節）：** 在 CLAUDE.md 索引中增加「canonical example」機制 —

```markdown
## Canonical Examples（P3 實作時的參考範例）

以下為經過 code review 確認的標準寫法範例，P3 實作時應以這些檔案為 reference：

| 分類 | 範例檔案 | 說明 |
|------|---------|------|
| Vue 頁面 | src/views/IM001/IM001Main.vue | 標準 CRUD 主頁面 |
| API Service | src/services/IM001Service.ts | 標準 API 呼叫封裝 |
| Entity | src/main/.../entity/IM001Entity.java | 標準 Entity 定義 |
| REST Controller | src/main/.../controller/IM001Controller.java | 標準 REST 端點 |
| 單元測試 | src/test/.../IM001ServiceTest.java | 標準測試寫法 |
```

**待討論：**
1. 誰負責 review 和指定 canonical example？（PG？Tech Lead？）
2. 如果專案所有現有 code 都不夠好怎麼辦？是否需要先寫一個 golden reference 再跑 P3？
3. canonical example 的粒度 — 按「檔案類型」指定就夠，還是要到「特定場景」？例如有 LOV 的頁面 vs 沒有的頁面可能差很多。
4. 讀 reference 後的行為規則怎麼寫？初步構想：

```
三層優先級：
1. CLAUDE.md 規範 = authority（命名、結構、慣例）
2. canonical example = reference（讀指定檔案仿照寫法，標記來源）
3. AI 推斷 = forbidden（不可自行發明慣例）

若 canonical example 與 CLAUDE.md 規範衝突 → 以 CLAUDE.md 為準，並標記「⚠️ 範例與規範不一致，待確認」。
```

### A3. Subagent 禁令範圍 `DECIDED`

**原假設：** 禁止使用 subagent 寫程式碼（session-791acadc 前例教訓）。

**審視：** 根因是「subagent prompt 與模板矛盾」，不是 subagent 架構本身的問題。現代 Claude Code Agent tool 已大幅改善。

**決定：** 把禁令從「禁止 subagent 寫程式碼」改為「禁止 subagent 產出會進 git 的檔案」。

| 場景 | 可否委託 subagent | 理由 |
|------|:-----------------:|------|
| 寫實作程式碼 | ✗ | 需要完整 context + 回饋迴圈 |
| 寫測試程式碼 | ✗ | 同上 |
| 跑 lint / typecheck / test | ✓ | 純指令執行，無 context 依賴 |
| 搜尋既有程式碼（含 canonical example）| ✓ | 探索性質，不影響主 session |
| P4b 分析失敗原因歸類 | ✓ | 判斷歸類，主 session 只需結果 |

**實作位置：** 更新 P3 SKILL.md 設計原則第 2 點、04 架構設計 D5 決策記錄。

### A4. P4a / P4b 的關係 `DISCUSS`

**原假設：** D8 — 選項式二擇一，PG 在 session 層級選路。

**審視：** P4b 的 workflow 本身會產出「不適合自動化的 case」清單交給 P4a，所以跑了 P4b 通常也需要 P4a 收尾。但如果改為「預設 P4b → P4a」，會強迫所有專案都要有 E2E 環境。

**修正方向：** 維持二擇一選項，但明確定義兩條路徑的適用條件和銜接：

```
路徑 α — 專案無 E2E 設施（或不值得建置）
  → 只跑 P4a（完整 UAT）

路徑 β — 專案有 E2E 設施
  → P4b（自動化 case）→ P4a（P4b 殘餘的不適合自動化 case）
  → P4a 此時只跑 P4b 產出的「補充 UAT 項目」，非完整 UAT
```

**待討論：**
1. 路徑 β 的 P4a 輸入來源改為讀 P4b 的「補充 UAT 項目」而非從頭產 checklist — 這需要修改 P4a SKILL.md 的輸入規格。
2. 是否需要在 P4b 彙總報告中加一個欄位標記「建議 P4a 後續項目」？目前已有「補充 UAT 項目」區塊，格式是否足夠給 P4a 直接使用？

## B. 缺失機制（需新增）

### B1. Checkpoint / Resume 機制（P3 容錯）`DECIDED`

**問題：** P3 如果在第 12 個 task 時 session 斷掉或 context 耗盡，沒有 checkpoint 記錄哪些 task 完成了，只能靠 git log 人工判斷。

**決定：** P3 每完成一個 task 後，更新 `{程式編號}_progress.md`。SG1 偵測到 progress 檔時，自動提出 resume。

**設計：**

```markdown
# {程式編號} — P3 Implementing Progress

> 最後更新：{YYYY-MM-DD HH:mm}

## Task 清單來源
- frontend: {path}
- backend: {path}

## 進度

| Task ID | 狀態 | Commit | 備註 |
|---------|:----:|--------|------|
| F01 | done | a1b2c3d | |
| F02 | done | e4f5g6h | SG2 增加了空值邊界測試 |
| F03 | skip | — | PG 指示跳過（待 SA 確認規格）|
| B01 | wip | — | Red 階段，測試已寫完 |
| B02 | todo | — | |

## 未解決 Warning
- F02 實作時發現 API response 格式與規格統計不一致，已標記待確認
```

**SG1 Resume 流程：**
1. 偵測到 `{程式編號}_progress.md` → 詢問「是否從上次進度繼續？」
2. PG 確認 → 從第一個 `todo` 或 `wip` 的 task 開始
3. PG 否認 → 從頭開始（不刪除 progress 檔，覆寫）

**實作位置：** P3 SKILL.md 新增 checkpoint 區塊 + SG1 步驟。

### B2. Context 管理策略（P3 長 session 存活）`DISCUSS`

**問題：** P3 如果有 20 個 task，每個 task 產生的程式碼 + 測試 + diff 會快速堆積 context。

**現實限制：**
- Claude Code 有自動 compaction（壓縮舊訊息），但 PG 觀察到 200k 左右效果開始變差
- Claude **無法自行執行 /clear** — 這是 harness 的權限邊界，只有人能觸發
- compaction 會丟失細節，可能導致後面的 task 遺失前面建立的 context

**目前已有的保障：**
- B1 的 checkpoint 檔 → 即使 /clear 也能 resume
- Git commits → 實作結果不受 context 影響

**剩餘問題：** compaction 或 /clear 後，P3 需要重新建立的 context 有哪些？

| Context 類型 | 是否需要重建 | 來源 |
|-------------|:----------:|------|
| CLAUDE.md 規範 | 是 | 重讀 CLAUDE.md 索引 |
| 任務清單 | 是 | 重讀 task list 檔案 |
| 已完成 task 的進度 | 是 | 讀 progress.md（B1） |
| 已完成 task 的實作細節 | **否** | 已 commit，不需要記在 context 中 |
| 當前 wip task 的中間狀態 | **是** | ← 這是唯一無法靠檔案恢復的 |

**待討論的方向：**

1. **被動策略（低成本）：** P3 SKILL 加入規則 — 「若 PG 在任何時候執行 /clear，AI 重啟時應：讀 progress.md → 讀 CLAUDE.md → 讀 task list → 從 wip task 繼續」。依賴 PG 自行判斷何時 /clear。

2. **主動提醒策略（中成本）：** P3 SKILL 加入規則 — 「每完成 N 個 task 後，主動建議 PG 考慮 /clear 重啟（附帶：已有 progress.md 可 resume）」。N 的預設值待定（5？8？依 task 複雜度？）。

3. **Hooks 自動化策略（高成本，未來探索）：** 利用 Claude Code hooks 偵測 context 使用量，自動觸發 checkpoint 寫入 + 提醒 PG。但目前 hooks 能否讀取 statusline 數據尚未確認。

**傾向：** 先實作方向 1 + 2（被動 + 主動提醒），hooks 自動化列為未來探索。

### B3. P4 → P2 回饋迴路（進化機制）`DECIDED`

**問題：** 如果 P4 反覆發現同類問題（例如 P2 的 test_cases.md 總是漏掉空值邊界），沒有結構化的方式回饋到 P2。

**決定：** 建立進化機制 — P4 彙總報告增加「系統性回饋」區塊（留在各專案內），SKILL 維護者定期蒐集回 AiCoding 專案集中審視。

**設計：**

P4a/P4b 彙總報告新增區塊（留在各專案內作為紀錄）：

```markdown
## 系統性回饋（供 P2 演進）

| 回饋對象 | 類型 | 描述 |
|---------|------|------|
| P2 test_cases.md | 遺漏 | 空值邊界測試案例缺失 — 所有可空欄位都應有空值 case |
| P2 frontend_tasks | 粒度 | LOV 連動 task 應拆為「觸發條件」和「畫面更新」兩個子 task |
| P3 實作 | 品質 | 錯誤處理統一用 try-catch 但未對齊 CLAUDE.md 規範的 error handler |
```

進化流程：
```
各專案 P4 彙總報告
  └─ 「系統性回饋」區塊（留在專案內）
        ↓ SKILL 維護者定期蒐集
AiCoding/lessons_learned.md（集中版）
        ↓ 審視後納入
P2 prompt 自檢清單 / P3 SKILL 規則 更新
```

**職責分離：**
- **專案內 PG：** 跑 P4 時如實填寫「系統性回饋」區塊，不需要知道 AiCoding 專案
- **SKILL 維護者：** 定期從各專案蒐集回饋，在 AiCoding 專案內審視並更新 SKILL

**`lessons_learned.md` 位置：** AiCoding 專案根目錄（集中收集點），按 SKILL 階段分類。

**實作位置：** 更新 P4a / P4b SKILL.md 彙總報告格式，新建 `lessons_learned.md` 骨架。

### B4. Session 紀錄集中歸檔 `DECIDED`

**問題：** Claude Code 原生將 session 存在 `~/.claude/projects/{project-path}/{uuid}.jsonl`，UUID 命名、JSONL 格式、跨專案散落。回頭找「上次跑 P3 的 session」幾乎不可能，也無法有效驅動進化機制。

**決定：** 每個 SKILL（P3/P4a/P4b）結束時，自動將當前 session 的完整紀錄複製到集中目錄，改為人類可讀的檔名，並附上結構化摘要。

**集中目錄位置：** `~/.soetek-ai-coding/`（使用者層級，跨專案共用）

**目錄結構：**

```
~/.soetek-ai-coding/
  ├── index.md                                              ← 全域索引
  └── {專案名}/
      ├── {編號}-{SKILL}-{簡述}_{yyyyMMdd}_{HHmm}.jsonl     ← 完整 session 複製
      ├── {編號}-{SKILL}-{簡述}_{yyyyMMdd}_{HHmm}/
      │   └── subagents/                                     ← subagent 紀錄複製
      │       ├── agent-{id}.jsonl
      │       └── agent-{id}.meta.json
      └── {編號}-{SKILL}-{簡述}_{yyyyMMdd}_{HHmm}.summary.md ← 結構化摘要
```

**檔名範例：**
- `IM004-P3-implementing_20260420_1430.jsonl`
- `IM004-P4b-e2e_20260421_0900.jsonl`
- `IM004-P3-implementing_20260420_1430.summary.md`

**Session 定位方式：** SKILL 結束時，掃描 `~/.claude/projects/{current-project-path}/` 下最近修改的 `.jsonl` 檔（即當前 session），連同其同名子目錄（subagents/）一起複製。

**結構化摘要格式（`.summary.md`）：**

```markdown
# Session Log — {編號} {SKILL} {簡述}

> 日期：{YYYY-MM-DD HH:mm}
> 專案：{專案名}
> SKILL：{skill-name} v{version}
> 原始 Session：~/.claude/projects/{path}/{uuid}.jsonl

## 完成的 Task
| Task ID | 結果 | 備註 |
|---------|------|------|
| F01 | done | |
| F02 | done | SG2 增加空值測試 |

## 關鍵決策
- {session 中做出的重要選擇和理由}

## 問題與教訓
- {遇到的問題、workaround、值得注意的現象}

## 系統性回饋（供進化機制）
- {與 P4 彙總報告的「系統性回饋」同步}
```

**觸發時機：** 每個 SKILL 的最後一步（P3 全部 task 完成後 / P4a 彙總報告後 / P4b 彙總報告後），自動執行歸檔。若使用者中途結束 session，下次 resume 時補歸檔。

**與進化機制（B3）的關係：**
- `summary.md` 的「系統性回饋」區塊 = 進化機制的原始素材
- SKILL 維護者回到 AiCoding 審視時，掃描 `~/.soetek-ai-coding/*/` 下的 summary 即可
- 需要深入追查時，打開對應的 `.jsonl` 看完整對話

**實作位置：** P3 / P4a / P4b SKILL.md 各新增「歸檔」步驟（最後一步）。
