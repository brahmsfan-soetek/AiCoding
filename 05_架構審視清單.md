## A. 假設校準（原有設計的前提重新檢視）

### A1. Context Rot 嚴重度 `HOLD`

**原假設：** 全自動有 context rot + 品質漂移風險，所以 P3 採半自動 + 高密度 stop gate。

**審視：** Opus 4.7 支援 1M context，內建 compaction 機制，長 session 指令遵循度已大幅提升。SG2「不可省略」在 20 task 的功能中 = 20 次中斷，trivial task（DTO 欄位、i18n key）幾乎是純摩擦。

**v2 進展：** 類型 tag 分流後，只有「有測試的 tag」才停 SG2（`[validator]` / `[processor]` / `[service]` / `[store-map]`），其他類型直接跳過。實際 SG2 中斷次數大幅降低。

**決定：** 維持半自動觀察。累積 v2 的實戰數據後回顧。

**待觀察指標：**
- SG2 實際攔截率（PG 真的修改了測試清單的比例）
- SG2 攔截的問題嚴重度分布
- `[processor]` 選填欄位空值覆蓋的實際 bug 防護次數

---

### A2. 「讀規範、不掃 code」的邊界 `DISCUSS`

**原假設：** 不掃 code 歸納 pattern（AI 行為會漂移），改讀 CLAUDE.md 索引揭露的規範文件（deterministic）。

**審視：** 原則正確，但太二元。CLAUDE.md 不可能涵蓋所有細節（API response 格式、component prop 介面、test fixture 結構）。

**方向（待討論細節）：** 在 CLAUDE.md 索引中增加「canonical example」機制：指定經過 code review 的範例檔案清單，作為 P3 實作的 reference。

**待討論：**
1. 誰負責 review 和指定 canonical example？
2. 如果專案所有現有 code 都不夠好怎麼辦？
3. canonical example 的粒度（檔案類型 vs 特定場景）？

初步構想三層優先級：
```
1. CLAUDE.md 規範 = authority（命名、結構、慣例）
2. canonical example = reference（讀指定檔案仿照寫法，標記來源）
3. AI 推斷 = forbidden（不可自行發明慣例）
```

---

### A3. Subagent 禁令範圍 `DONE`

**決定：** 把禁令從「禁止 subagent 寫程式碼」改為「禁止 subagent 產出會進 git 的檔案」。

| 場景 | 可否委託 subagent |
|------|:-----------------:|
| 寫實作 / 測試程式碼 | ✗ |
| 跑 lint / typecheck / test | ✓ |
| 搜尋既有程式碼 | ✓ |

**v2 狀態：** 三個 P3 SKILL 均已遵循此原則。

---

### A4. P4a / P4b 的關係 `OBSOLETE`

**v2 廢除：** P4a / P4b 均已廢除。test_cases.md 改為手測 checklist，PG 完工後開瀏覽器逐條勾選，不需要 SKILL 包裝。

**廢除理由：** AR003 實戰顯示 P4b 第一次導入陣痛（基建、seed、auth fixture、sessionStorage 踩坑）+ P3/P4b 回合制成本 > Playwright 帶來的回歸保護。PG 手測 1 秒看出 UI 問題遠快於寫 E2E。

## B. 缺失機制

### B1. Checkpoint / Resume 機制（P3 容錯）`DONE`

**v2 狀態：** 三個 P3 SKILL 共用 `Docs/spec/{程式編號}/log/{程式編號}_progress.md`，SG1 偵測存在時自動提出 resume。

**檔名位置變更（v2）：** 從 `{程式編號}_progress.md` 改為 `Docs/spec/{程式編號}/log/{程式編號}_progress.md`（對齊 F5 feedback — plan/log 子目錄結構）。

**狀態值：** `done` / `wip` / `skip` / `todo`（task prefix `B*` / `F*` 區分前後端）。

---

### B2. Context 管理策略（P3 長 session 存活）`DISCUSS`

**v2 進展：** 多了 `session_log.md`（repo 內）累積跨 session 決策與教訓（F2/F3 feedback）。compaction 或 /clear 後，P3 重建 context 的來源：

| Context | 來源 |
|---|---|
| CLAUDE.md 規範 | 重讀 CLAUDE.md 索引 |
| 任務清單 | 重讀 task list |
| 已完成 task 進度 | progress.md |
| 已完成 task 細節 | git log / 不需重建 |
| 當前 wip task 中間狀態 | **session_log.md** |
| 跨 session 決策演化 | **session_log.md** |

**待討論方向：**
1. 被動策略（已採用）：PG /clear 後 AI 重讀 progress + session_log + CLAUDE.md 繼續
2. 主動提醒策略：P3 每完成 N task 建議 /clear（N 待定）
3. Hooks 自動化：未來探索

---

### B3. P4 → P2 回饋迴路（進化機制）`DONE / 部分 OBSOLETE`

**原設計：** P4 彙總報告有「系統性回饋」區塊，SKILL 維護者定期從各專案蒐集到 `lessons_learned.md`。

**v2 調整：** P4 廢除後，回饋來源改為：
- **P3 SKILL 的 `session_log.md`**（repo 內）— 本專案教訓
- **`~/.soetek-ai-coding/skill-feedback/{skill-name}.md`**（repo 外）— 跨專案對 SKILL 的改進建議

`lessons_learned.md`（AiCoding 根目錄）仍是 SKILL 維護者的集中審視點，但來源從「P4 彙總報告」改為「skill-feedback + session_log 摘要」。

---

### B4. Session 紀錄集中歸檔 `DONE`

**v2 狀態：** 三個 P3 SKILL 均在結束時執行歸檔流程：
- 檔名：`{程式編號}-P3-{backend|frontend|data}_{yyyyMMdd}_{HHmm}.jsonl`
- 位置：`~/.soetek-ai-coding/{專案名}/`
- 附帶 `.summary.md`（從 progress.md + session_log.md 取內容）

## D. v2 新增待觀察項目

### D1. `[processor]` 選填欄位空值覆蓋實用性 `DISCUSS`

**背景：** v2 的 SG2 強制規則要求每個 Processor 選填欄位必須產 null / `""` / 空白三種測試案例。

**待觀察：**
- 實際防住幾個 bug（AR003 踩過的 SQL `:param IS NULL OR col = :param` pattern）
- 是否過度嚴格（例如某些欄位在業務邏輯上永遠不為空）
- P2 產清單時是否能正確列出選填欄位清單

---

### D2. 契約測試覆蓋度是否足夠防 contract mismatch `DISCUSS`

**背景：** AR003 BUG-P4b-R4-CONTRACT（後端 `creditAmt` vs 前端 `creditLimit`）是 v2 前端 `[store-map]` 契約測試的主要動機。

**待觀察：**
- 契約測試是否真的在 P3 階段就抓到 rename / shape 問題
- `[service]` 與 `[store-map]` 的測試範圍是否重複或漏

---

### D3. 手測 checklist 是否足以取代 P4a `DISCUSS`

**背景：** v2 廢除 P4a 後，PG 照 `test_cases.md` 的「狀態」欄勾選即可。

**待觀察：**
- 40-50 條 case 的手測體感（是否太累）
- 「執行步驟」與「預期結果」是否具體到人眼可對
- 手測失敗的 bug 派工流程是否順暢（ad hoc 對話）

---

### D4. 前後端分 session 的 progress.md 共用 `DISCUSS`

**背景：** v2 的 p3-backend / p3-frontend 共用同一份 progress.md，task prefix `B*` / `F*` 區分。

**待觀察：**
- 兩個 session 更新同一檔的衝突問題（通常不會同時跑但仍需留意）
- `F*` task 在 backend session 中出現時的顯示是否清晰
