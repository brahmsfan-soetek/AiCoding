## A. 假設校準（原有設計的前提重新檢視）

### A1. Context Rot 嚴重度 `HOLD`

**原假設：** 全自動有 context rot + 品質漂移風險，所以 P3 採半自動 + 高密度 stop gate。

**審視：** Opus 4.7 支援 1M context，內建 compaction 機制，長 session 指令遵循度已大幅提升。SG2「不可省略」在 20 task 的功能中 = 20 次中斷，trivial task（DTO 欄位、i18n key）幾乎是純摩擦。

**v2 進展：** 類型 tag 分流後，只有「有測試的 tag」才停 SG2（`[validator]` / `[processor]` / `[service]` / `[store-map]`），其他類型直接跳過。實際 SG2 中斷次數大幅降低。

**決定：** 維持半自動觀察。累積 v2 的實戰數據後回顧。

**待觀察指標（v2.x 更新後）：**
- SG2 實際攔截率（`[validator]` PG 修改測試清單、`[processor]` PG 修改對照表的比例）
- SG2 攔截的問題嚴重度分布
- `[processor]` SG2 雙對照表（api_contract A## + current_schema）對偏離的攔截次數
- 前端 `[service]` / `[store-map]` SG2 對照 api_contract A## 對跨層 rename / shape 漂移的攔截次數

---

### A2. 「讀規範、不掃 code」的邊界 `DECIDED`（2026-05-29 G2-05 / G3 優先序 7）

**原假設：** 不掃 code 歸納 pattern（AI 行為會漂移），改讀 CLAUDE.md 索引揭露的規範文件（deterministic）。

**審視：** 原則正確，但太二元。CLAUDE.md 不可能涵蓋所有細節（API response 格式、component prop 介面、test fixture 結構）。三方審查同向最強的一條：研究 §2 支持工具式檢索、實戰（SO0062 / AR003）必需「看一眼既有 code」。

**決定：「grep 開門 + 輕量指路」（已落地至 `subagent-boundary.md`）**
- **grep 開門**：容許 grep/glob 工具式查證既有 code 落地細節作交叉驗證；禁止的是把歷史 anti-pattern 當權威規範盲目複製。
- **輕量指路**：PG 需要時即時指一個「已知結構良好的既有檔」當參考（一句指路）；**不建 canonical-example 清單 apparatus**（清單有 silent drift + 維護債，原「三層優先級」構想作廢）。
- **兩條紅線不鬆**：schema（current_schema）與契約（api_contract）兩個 SSOT 不可從 code 推。

### A3. Subagent 禁令範圍 `DONE`

| 搜尋既有程式碼 | ✓ |

**v2 進展：** 多了 `session_log.md`（repo 內）累積跨 session 決策與教訓（F2/F3 feedback）。compaction 或 /clear 後，P3 重建 context 的來源：

| Context | 來源 |
|---|---|
| CLAUDE.md 規範 | 重讀 CLAUDE.md 索引 |

### D5. SG2 對照表攔截率（取代 D1 / D2）`DISCUSS`

**背景：** v2.x 拔 mock test 改 SG2 對照表後，所有 contract/schema 對齊防護集中在 SG2 對照表（雙對照表 / 契約對照表）。
