## 1. 核心觀念

### 1.1 根本原則

```
AI 做執行，人做判斷。
所有需要判斷的點，都必須在 AI 介入之前由人類完成。
```

Agentic Coding 的本質是**人機協作的新形態**。Agent（如 Claude Code）具備自主讀取檔案、執行命令、跑測試、多輪修正的能力，但它的判斷品質完全取決於人類架構的 Context。

```
傳統開發：      需求 → 工程師理解 → 工程師寫代碼 → 測試
Prompt 時代：   需求 → 工程師整理 prompt → LLM 生成代碼 → 工程師逐行驗證
Agentic 時代：  需求 → 工程師架構 context → Agent 自主探索/實作/驗證 → 工程師審查決策點
```

工程師的角色轉變為 **Context 架構師**——決定 Agent 在什麼範圍內工作、讀取哪些資訊、遵守哪些約束、用什麼回饋迴路自我修正。

與 Prompt 時代的關鍵差異：

| | Prompt 時代 | Agentic 時代 |
|---|---|---|
| **資訊傳遞** | 人工複製貼上到 prompt | Agent 自主讀取專案檔案 |
| **驗證方式** | 人工逐行檢查產出 | Agent 跑測試自我修正，人工審查業務邏輯 |
| **錯誤修正** | 人工描述錯誤，重新生成 | Agent 讀取錯誤訊息，自主迭代修正 |
| **Context 管理** | 手動控制 prompt 內容 | 透過 CLAUDE.md、planning 文件、sub-agent 架構控制 |
| **核心技能** | Prompt Engineering | Context Engineering |

> 「找到最小的高信號 token 集合，最大化期望結果的可能性。」— Anthropic

### 1.2 基本公式

```
Agent 產出品質 = f(Context 品質, 任務邊界, 回饋迴路強度)
```

- **Context 品質**：高信噪比、無矛盾、結構化——不是「給越多越好」，而是「精準給到」
- **任務邊界**：明確的範圍與介面契約，Agent 知道「做什麼」和「不做什麼」
- **回饋迴路強度**：測試、lint、型別檢查——Agent 能自主驗證並修正的機制越強，產出越可靠

三者缺一，Agent 的自主能力反而會**放大錯誤**——它會自信地在錯誤的方向上越跑越遠。

**量化證據——為什麼需要方法論：**

| 指標 | 數值 | 來源 |
|------|------|------|
| AI 生成 PR 被拒率 vs 人類 | 67.3% vs 15.6% | LinearB |
| AI 代碼含重大問題倍率 | 1.7× | CodeRabbit 2025/12 |
| 有經驗開發者使用 AI 後的速度變化 | −19%（但自認為 +20%） | METR 2025/07 |
| 實際淨開發週期改善 | 8-13%（非行銷宣稱的 50%） | Thoughtworks |
| AI 代碼未通過安全測試 | 45% | OWASP 相關研究 |

結論不是「不該用 Agent」，而是**無紀律的 Agent 使用比不用更糟**。

### 1.3 源頭鐵則

```
所有衍生物都必須從源頭長出來，絕不從下游反推。
```

需求驅動代碼，代碼不反推需求。這條規則在 Agentic 時代更重要——因為 Agent 有能力自主產出大量看似合理的代碼，讓「反推」變得更誘人也更危險。

- 需求清單 → 驅動代碼生成（正確方向）
- 代碼 → 反推需求清單（錯誤方向，會合理化錯誤）
- 業務規則 → 驅動測試案例（正確方向）
- 測試結果 → 反推業務規則（錯誤方向，會遺漏未測場景）

**Agentic 環境的特殊風險**：Agent 能自主產出完整的測試套件和文件，這些產出看起來很專業，但如果是從代碼反推而非從需求衍生，它們只是代碼的鏡像——無法發現遺漏，只能證明「代碼做了什麼」。

## 2. LLM 的運作特性與限制

理解 LLM 的底層原理，才能設計出有效的 Context Engineering 策略。

### 2.1 LLM 的本質：從機率預測到自主行動

LLM 的運作本質是根據輸入的 token 序列，預測最可能的下一個 token。改變的是在這個基礎上演化出的能力層次：

```
┌─────────────────────────────────────────────────────────┐
│  Layer 3: Agentic Loop（自主行動層）◀ 本文件的核心層     │
│    ├─ 讀檔、寫檔、搜尋專案                               │
│    ├─ 執行命令、跑測試、跑 lint                           │
│    ├─ Sub-agent 分工（context 隔離）                      │
│    ├─ 持久記憶（CLAUDE.md、memory 系統）                  │
│    └─ 多輪自主迭代（生成 → 驗證 → 修正 → 再驗證）         │
│  Layer 2: Extended Thinking（推理層）                     │
│    └─ 自我反駁、回溯、多步推導；thinking 不佔後續 context  │
│  Layer 1: In-Context Learning（上下文學習層）             │
│    └─ 從 CLAUDE.md 規則、專案範例中學習模式               │
│  Layer 0: Next-Token Prediction（基礎層）                 │
│    └─ 機率預測，一切的底層                                │
└─────────────────────────────────────────────────────────┘
```

每一層繼承下層的能力也繼承其限制：
- Layer 2 的推理能力讓模型能自我修正，但修正的依據仍是 context 中的資訊——源頭資訊有矛盾，推理層也會被誤導
- Layer 3 的工具使用讓模型能驗證假設，但選擇「驗證什麼」和「如何解讀結果」仍受 Layer 0 的機率傾向影響
- Layer 3 的 sub-agent 能隔離 context，但主 agent 仍需根據 sub-agent 的摘要做決策——摘要品質決定決策品質

**核心認知：Layer 3 讓 Agent 具備了自我修正的能力，但這個能力的天花板取決於回饋迴路的品質。** 能自動失敗的測試 > 只靠推理的自省 > 沒有回饋的盲目生成。

**Claude Code 的 Context 機制：**

| 機制 | 作用 | Context Engineering 的意義 |
|------|------|---------------------------|
| **CLAUDE.md** | 每次 session 自動載入，作為持久指令 | 專案規範、約束條件的 single source of truth |
| **Sub-agent** | 獨立 context 執行探索/實作，返回精煉摘要 | 防止主 agent 被大量探索結果污染 |
| **Compaction** | ~95% context 使用率時自動壓縮歷史對話 | 延長有效工作時間，但壓縮是有損的 |
| **Memory 系統** | 跨 session 持久記憶 | 累積專案知識，但不應取代源頭文件 |
| **Extended Thinking** | 前次 thinking block 自動移除 | 推理不佔 context，但結論仍受 context 品質影響 |
| **KV-cache** | 相同前綴的 prompt 可重用計算結果 | 保持 CLAUDE.md/planning 文件結構穩定，可大幅降低延遲與成本 |

**各機制的具體限制（2026/03，以官方文檔為準）：**

| 類別 | 限制 | 數值 | 工程意義 |
|------|------|------|---------|
| **CLAUDE.md** | 建議行數上限 | **~200 行/檔** | 超過後 Agent 遵從度下降；拆分到 `.claude/rules/` 的多個 `.md` 檔 |
| | 匯入遞迴深度 | **最多 5 層** | `@path/to/import` 的連鎖匯入上限 |
| | Token 膨脹 | **~1.7×** 原始 token | 行號格式化造成的額外消耗 |
| | Compaction 後行為 | 從磁碟重新讀取 | CLAUDE.md 的指令不會因 compaction 丟失 |
| | 子目錄 CLAUDE.md | 惰性載入 | 只在 Agent 讀取該目錄檔案時才載入 |
| **MEMORY.md** | 啟動時載入行數 | **前 200 行** | 超過 200 行的內容不會被自動載入 |
| **Context Window** | Opus 4.6 | **1M tokens**（128K 輸出上限） | 中大型功能可在單一 session 完成 |
| | Sonnet 4.6 | **1M tokens**（64K 輸出上限） | MRCR 準確率僅 18.5%（vs Opus 76%） |
| | Haiku 4.5 | **200K tokens**（64K 輸出上限） | 適合 sub-agent 的探索任務 |
| **Compaction** | 自動觸發閾值 | **~95% context** | 可用 `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE` 調低 |
| | 手動觸發 | `/compact` 指令 | 可附帶自訂摘要指示 |
| **Sub-agent** | 巢狀限制 | **不可巢狀** | Sub-agent 不能再生成 sub-agent |
| | Agent hook 回合上限 | **50 回合** | 超過後自動停止 |
| **Extended Thinking** | 最小 budget | **1,024 tokens** | API 層級限制 |
| | `max` 努力等級 | **僅 Opus 4.6** | Sonnet 不支援最高推理等級 |
| **檔案讀取** | 預設行數上限 | **2,000 行** | 超過需指定 offset/limit |
| | 預設 token 上限 | **~25,000 tokens** | 可用 `CLAUDE_CODE_FILE_READ_MAX_OUTPUT_TOKENS` 調整 |
| **KV-cache** | 預設 TTL | **5 分鐘** | Prompt 結構穩定時可重複命中 |
| | 最小可快取 tokens | Opus 4,096 / Sonnet 2,048 | 太短的 CLAUDE.md 無法觸發快取 |
| | 快取讀取價格 | **0.1× 基礎輸入價** | 命中快取時成本僅為未命中的 1/10 |

### 2.2 語義漂移（Semantic Drift）

每一次 LLM 處理都是一次「轉換」。與有損壓縮（如 JPG）不同，LLM 的轉換是**有損且隨機的**——更像是**傳話遊戲**：每一輪不只丟失資訊，還可能**添加**不存在的細節（幻覺）。

```
路徑 A：每次從源頭載入（錨定路徑）

planning/requirements.md ──→ Session 1 Agent 讀取 → 產出代碼
planning/requirements.md ──→ Session 2 Agent 讀取 → 產出代碼
planning/requirements.md ──→ Session 3 Agent 讀取 → 產出代碼
  ↑ 每次回到相同的源頭文件，漂移被歸零

路徑 B：以前次產出為輸入（漂移路徑）— 傳話遊戲

原始需求 → Session 1 產出 → Session 2 基於 S1 產出 → Session 3 基於 S2 ...
   100%        ~95%              ~85%                      ~70%
                          ↑ 每輪不只丟失，還可能添加錯誤細節
```

**不是「用 Agent 越多次品質越差」，而是「不回到源頭就會越用越差」。**

**走上路徑 B 的常見原因：**

| 原因 | 發生什麼 |
|------|---------|
| **沒有錨點** | 只有代碼沒有需求清單，Agent 從代碼推測意圖，推測引入偏差 |
| **口頭修改不落文件** | 「上次跟你說那個改一下」→ 新 Session 的 Agent 不知道「那個」是什麼 |
| **跨 Session 失去上下文** | 新 Session 只看到結果代碼，不知道前次的設計取捨 |
| **衍生文件取代源頭** | 用 Agent 對代碼的「理解」取代原始需求 |

**Agentic 環境特有的漂移來源：**

| 來源 | 機制 | 對策 |
|------|------|------|
| **Compaction 壓縮** | 對話達 ~95% context 時自動摘要，早期精確指令可能被壓縮成模糊描述 | 關鍵約束放在 CLAUDE.md（不被壓縮），而非僅在對話中提及 |
| **Sub-agent 摘要** | Sub-agent 返回精煉結果，細節被過濾 | 讓 sub-agent 輸出結構化結果（表格、JSON），非散文摘要 |
| **Memory 記憶偏差** | 跨 session 的 memory 是 Agent 的「理解」，非原始事實 | Memory 存參考指引，不存業務規則本身 |
| **代碼即 context** | Agent 讀取既有代碼，錯誤代碼成為「事實」影響新代碼 | 測試作為獨立錨點，代碼的正確性由測試驗證而非 Agent 的判斷 |

**確保留在路徑 A 的方法：**

| 策略 | 做法 |
|------|------|
| **CLAUDE.md 錨定** | 將不可妥協的約束寫入 CLAUDE.md，每次 session 自動載入 |
| **Planning 文件在專案內** | Context Package 存放需求/規則/測試清單，Agent 直接讀取源頭文件 |
| **編號追溯** | 代碼中標註對應的 REQ/BR 編號，偏差可被發現 |
| **測試錨定** | 測試是不會被 compaction 壓縮的固定錨點，代碼偏離時測試會失敗 |
| **變更回源** | 任何需求變更都更新 planning/ 文件，不只口頭告知 Agent |
| **Session 間歸零** | 新 session 讓 Agent 重新讀取源頭文件，不依賴前次對話的記憶 |

**核心認知：Agent 的對話記憶是有損的，Compaction 後更是如此。檔案系統才是無損的持久記憶。** 一切決策和變更都必須落地到文件。

### 2.3 Context Window 的現實

即使 Context Window 很大（如 Opus 4.6 的 1M tokens），也不代表「全部丟進去」是好策略。

#### 位置偏見：改善但未消失

早期的「Lost in the Middle」問題已大幅改善，但未完全消失。根源是架構性的——RoPE 天然讓模型更關注序列首尾的 token。

**NIAH 不是好的衡量標準。** 更有意義的 benchmark 是 **MRCR v2**（Multi-needle Retrieval）：

| 模型 | MRCR v2 @ 1M tokens | 意義 |
|------|---------------------|------|
| Claude Opus 4.6 | 76% | 大型任務可在單一 session 完成 |
| Claude Sonnet 4.5 | 18.5% | 相同任務需要更積極地拆分 session |

**模型等級的影響遠大於 prompt 技巧的差異。**

#### Context Rot：真正的瓶頸

Chroma Research（2025）發現：**每個模型都隨著 context 增加而品質下降**，下降速度取決於：

1. **語義相似度**：問題與答案的語義距離越遠，檢索越困難——比位置效應影響更大
2. **干擾密度**：單一無關資訊降低準確率；四個干擾項會加劇惡化
3. **注意力預算**：每個新 token 都消耗有限的注意力資源，有效 context 遠小於名義容量

```
Context 的實際有效性（結合位置、語義競爭、注意力預算）：

  [精準指令 10K tokens]                        →  高品質輸出
  [精準指令 + 乾淨參考 50K tokens]              →  良好輸出
  [三份矛盾文件 100K tokens]                    →  不可預測的輸出（語義競爭）
  [大量無關代碼 500K tokens]                    →  關鍵指令被稀釋，品質下降（context rot）
```

**核心原則：信噪比（Signal-to-Noise Ratio）比總量更重要。** 載入 50K 精準內容的效果，優於載入 500K 含噪訊的內容。

#### Claude Code 的 Context 管理

| 機制 | 觸發條件 | 效果 | 注意事項 |
|------|---------|------|---------|
| **Compaction** | context 使用率 ~95% | 自動摘要早期對話；CLAUDE.md 從磁碟重新讀取 | 有損壓縮——結構化摘要品質 > 激進壓縮 |
| **Extended Thinking 自動清理** | 每一輪結束 | 前次 thinking block 移除，不佔 context | 推理過程不累積，只保留結論 |
| **Sub-agent context 隔離** | 手動或自動分派 | 子任務在獨立 context 中執行 | 主 agent 只看到摘要 |
| **CLAUDE.md 優先載入** | 每次 session 開始 | 持久指令在 context 前段（高注意力區域） | 官方建議 ~200 行/檔 |

### 2.4 各層能力邊界

#### Layer 0 — Next-Token Prediction 帶來的固有特性

| 特性 | 具體表現 | Agentic 工程對策 |
|------|---------|-----------------|
| 模式複製 | 輸入中出現的模式會被放大。專案中的錯誤模式若出現多次，Agent 傾向複製 | CLAUDE.md 中明確標註「不要做的事」；確保專案範例本身正確 |
| 語義競爭 | 多份矛盾資訊共存時，模型混合而非裁決 | Quality Gate 1 消除矛盾；CLAUDE.md 規則控制在 20 條以內 |
| 頻率偏見 | 訓練資料中常見模式優先於罕見但正確的做法 | 少見的業務邏輯在 planning 文件中給明確指令 |
| 自信填補 | 資訊不足時生成看似合理的內容（幻覺） | 待確認項目標記「暫不實作」；要求 Agent 遇到不確定時詢問 |

#### Layer 1 — In-Context Learning

| 能力 | 可靠場景 | 不可靠場景 |
|------|---------|-----------|
| 遵循指令格式 | CLAUDE.md 中的結構化規則（表格、清單） | 長篇散文中夾帶的隱式要求 |
| 學習專案慣例 | Agent 自主讀取 2-3 個一致的範例檔案 | 範例之間風格不一致 |
| 套用命名規則 | 規則明確且寫在 CLAUDE.md | 規則有例外且未說明例外條件 |
| 遵守約束條件 | 約束集中在前段，數量 < 20 條 | 50+ 條規則散布在多個文件中 |

#### Layer 2 — Extended Thinking

| 能力 | 可靠場景 | 不可靠場景 |
|------|---------|-----------|
| 多步推導 | 有明確前提和推導規則的邏輯鏈 | 需要業務直覺或組織脈絡的判斷 |
| 自我修正 | 推導過程中發現與已知事實矛盾 | 矛盾來自 context 中的錯誤資訊 |
| 方案比較 | 技術方案的 trade-off 分析 | 涉及主觀偏好或政治考量的選擇 |
| 邊界推理 | 從規則推導邊界案例 | 規則本身未涵蓋的業務邊界 |

#### Layer 3 — Agentic Loop ◀ 核心

| 能力 | 可靠場景 | 不可靠場景 |
|------|---------|-----------|
| **檔案探索** | 定向搜尋特定檔案或模式 | 需要理解整個專案的架構意圖和歷史脈絡 |
| **執行與驗證** | 有明確的測試案例、編譯檢查、lint 規則 | 需要判斷「行為是否符合使用者預期」 |
| **迭代修正** | 錯誤訊息明確指向問題 | 靜默的邏輯錯誤（代碼能跑但結果不對） |
| **多檔案協調** | 介面契約明確的跨檔案修改 | 隱式耦合的檔案間連動 |
| **Sub-agent 分工** | 獨立的探索/實作任務 | 需要在子任務間共享狀態 |
| **自主決策鏈** | 每一步都有回饋信號 | 連續多步無回饋的決策 |

**回饋迴路品質決定自主行動的可靠性：**

```
可靠性排序（由高到低）：

  自動化測試失敗 → 明確知道哪裡錯    → Agent 可自主修正    ✅ 可靠
  編譯/lint 錯誤  → 明確知道語法問題  → Agent 可自主修正    ✅ 可靠
  型別檢查失敗    → 知道介面不匹配    → Agent 可自主修正    ✅ 可靠
  測試全過但邏輯錯 → 沒有失敗信號     → Agent 認為已完成   ⚠️ 需要人工 review
  無測試無檢查     → 完全無回饋       → Agent 盲目自信     ❌ 不可靠
```

**投資在回饋迴路上的 ROI，在 Agentic 時代遠高於傳統開發。**

## 3. 反模式：常見的錯誤做法

### 3.1 反模式一：Raw Dump（原始傾倒）

```
做法：把 SA 規格書、HTML 原型、會議紀錄全部丟給 AI，說「照這個做」
結果：AI 在矛盾中隨機選擇，產出包含錯誤的代碼，且難以定位問題
```

AI 遇到矛盾時不會報錯，而是**靜默地選擇其中一個**，甚至**混合兩者**，產出看似完整但邏輯不一致的代碼。

### 3.2 反模式二：Over-Prompting（過度提示）

```
做法：用一個超長的 prompt 包含所有規則、所有例外、所有邊界條件
結果：AI 遺漏後段指令，或在相互矛盾的規則間產生幻覺
```

5000 行的 prompt 中，第 4000 行的指令被遵守的機率明顯低於第 100 行。

### 3.3 反模式三：One-Shot Everything（一次完成全部）

```
做法：讓 AI 一次生成整個功能的前端 + 後端 + 測試
結果：各部分的介面不一致，錯誤發生時難以定位，修改一處牽動全局
```

單次生成的代碼量越大，AI 在後段「忘記」前段設定的機率越高。

### 3.4 反模式四：No Verification（不驗證就合併）

```
做法：AI 生成的代碼看起來合理，直接合併進主分支
結果：隱含的邏輯錯誤在後期暴露，修復成本倍增
```

語法正確≠邏輯正確，看起來能跑≠行為符合需求。

### 3.5 反模式五：Downstream Backfill（下游反推）

```
做法：AI 寫完代碼後，再根據代碼反推需求清單或測試案例
結果：需求清單變成代碼的描述，而非代碼的依據；錯誤被合理化而非被發現
```

```
正確：需求清單 → 代碼 → 用需求清單驗證代碼
                ↑ 源頭            ↑ 錨點

錯誤：代碼 → 反推需求清單 → 「驗證通過」
                    ↑ 只是代碼的鏡像，不是獨立的驗證依據
```

## 4. 混合架構總覽

### 4.1 從五階段到三層架構

傳統五階段（Phase 0→4）是線性流程——先做完前一步才進下一步。混合架構保留所有內容，但重構為**品質維度**，每個維度在不同時機觸發：

```
Layer 3 — 知識管理（持續維護）
  Context Package：維護無矛盾、可按需組裝的 context 模組
  ↓ 提供乾淨的輸入

Layer 2 — 任務品質（每個功能/修改）
  Quality Gate：人類守門，確保必要的判斷不被跳過
  ↓ 確認可以開始

Layer 1 — 互動品質（每次 AI 對話）
  Execution Checklist：確保這次互動是「純執行」而非要求 AI 判斷
```

### 4.2 核心原則

| 原則 | 說明 |
|------|------|
| **源頭鐵則** | 所有衍生物從源頭長出來，絕不從下游反推 |
| **Single Source of Truth** | 每個事實只在一個地方定義，消除矛盾 |
| **Structured Over Prose** | 結構化清單優於散文式描述，AI 解析更準確 |
| **Small Batch** | 小批量、可驗證的任務，優於大範圍一次性生成 |
| **Test First** | 先定義預期行為，再讓 AI 實作，有錨點可對照 |
| **Human Decides, AI Executes** | 業務決策由人類做，代碼生成由 AI 做 |

### 4.3 與原五階段的對應關係

| 原階段 | 混合架構中對應的位置 | 核心差異 |
|-------|-------------------|---------|
| Phase 0 文件整備 | Gate 1（矛盾消除）+ Context Package 維護 | 不是「先做完才進下一步」，而是「發現矛盾隨時觸發」 |
| Phase 1 需求定義 | Gate 2（需求可驗證）+ `requirements/` + `rules/` + `tests/` | 產出物相同，但按功能域分檔而非單一大文件 |
| Phase 2 技術設計 | Gate 3（架構對齊）+ `contracts/` + `standards/` | 產出物相同，但強調定向取樣而非全面設計 |
| Phase 3 AI Coding | Layer 1 Checklist + Context 組裝 | 從「載入全部 Phase 1/2」變為「按需載入相關模組」 |
| Phase 4 驗證收斂 | Gate 5（追溯回填）+ 測試粒度分工 | 從「最後統一驗證」變為「邊做邊驗 + 最終人工驗收」 |

**核心變化：內容沒丟，從「先後順序」重構為「品質維度」。**

### 4.4 工作環境策略

```
知識管理（Layer 3）：獨立資料夾
  理由：純粹思考「要做什麼」，不被「現在怎麼做的」干擾
  AI 的 context 只有規格相關文件，信噪比最高

任務品質（Layer 2）：獨立資料夾 + 定向取樣專案
  理由：需要對齊專案架構，但不要被全部代碼淹沒
  做法：指定讀取特定檔案（共用元件、既有 API 模式），不掃描整個專案

互動執行（Layer 1）：專案內
  理由：必須在真實環境中執行和驗證
```

**定向取樣的做法：**

```
在獨立資料夾的 AI Session 中：

「請讀取 /project/src/components/TablesComponent.vue，
  我需要了解它的 props 和 events 介面」

「請讀取 /project/src/api/ 目錄下任一支既有 API，
  我需要了解路由、Controller、Service 的分層方式」
```

**定向取樣的方式選擇：**

| 情境 | 建議方式 | 理由 |
|------|---------|------|
| 你知道要看哪 1-3 個檔案 | 手動複製貼上 | 比 AI 搜尋更快、零額外 token 消耗 |
| 你知道要看什麼類型但不確定在哪 | 給 AI 明確搜尋指令 | 範圍受控，避免大規模掃描 |
| 你不確定專案中有什麼慣例 | 讓 AI 搜尋，但限定範圍和數量 | 探索性質，但需要設上限 |

避免的做法：讓 AI 無限制地「了解整個專案架構」——這會消耗大量 token 且引入噪訊。

**Agentic 工具下的調整：**

現代工具具備自主讀取檔案的能力，「獨立資料夾」的物理隔離不再是唯一手段。隔離的目的不變（控制信噪比），但實現方式可以是：

- 傳統方式：物理上獨立的資料夾，手動複製需要的參考檔案
- Agentic 方式：在專案目錄內工作，但透過 prompt 限定 scope

選擇依據不是「哪個更先進」，而是哪個在當下更高效。

## 5. Layer 3：知識管理 — Context Package

### 5.1 目的

維護一組**無矛盾、可按需組裝**的 context 模組。這是所有 AI 互動的基礎——context 品質決定 Agent 產出品質。

### 5.2 目錄結構

```
{project}-planning/
├── rules/                    ← 業務規則（按功能域分檔）
│   ├── validation.md         ← 驗證規則 BR-VL*
│   ├── calculation.md        ← 計算規則 BR-CA*
│   ├── conditional.md        ← 條件顯示 BR-CD*
│   └── posting.md            ← 過帳規則 BR-PO*
│
├── requirements/             ← 需求清單（帶追溯欄位）
│   ├── query.md              ← REQ-Q* 查詢功能
│   ├── edit.md               ← REQ-E* 新增編輯
│   └── posting.md            ← REQ-P* 過帳
│
├── contracts/                ← 介面契約
│   ├── api.md                ← API endpoint 定義
│   ├── components.md         ← 元件 props/events
│   └── data-model.md         ← 資料表 schema
│
├── tests/                    ← 測試錨點
│   ├── unit-specs.md         ← 單元測試規格（人類審核過）
│   └── acceptance.md         ← 人工驗收腳本（TC 清單）
│
├── standards/                ← 專案規範
│   └── CLAUDE.md             ← Agent 原生格式
│
├── decisions/                ← 決策紀錄
│   └── clarifications.md     ← 釐清紀錄、待確認項目
│
└── _sources/                 ← 原始文件（僅保留，不餵 AI）
    ├── SA_IM004.md
    └── IM004.html
```

**與單檔結構的對比：**

| 單檔結構（舊） | 模組化結構（新） | 優勢 |
|---------------|----------------|------|
| `01_需求清單.md`（所有需求） | `requirements/query.md`、`requirements/edit.md` | 按需載入，減少噪訊 |
| `02_業務規則.md`（所有規則） | `rules/validation.md`、`rules/calculation.md` | 修改一個域不影響其他域 |
| `04_資料模型.md` + `05_API契約.md` | `contracts/` 目錄統一管理 | 介面契約集中，易於交叉檢查 |

### 5.3 維護鐵則

| 鐵則 | 說明 | 違反後果 |
|------|------|---------|
| **單一定義** | 每個事實只在一個檔案中定義 | 修改時漏改其中一份，產生矛盾 |
| **無矛盾** | 模組之間不能有互相矛盾的定義 | AI 在矛盾中隨機選擇（反模式一） |
| **變更回源** | 任何需求變更先更新對應模組，再寫碼 | AI 下次 Session 讀到的還是舊需求 |
| **原始保留** | `_sources` 只看不餵，所有 AI 消費的內容都在模組中 | 原始文件的矛盾汙染 AI context |
| **結構化** | 用表格、清單，不用散文 | AI 解析不準確，人類也難以核對 |
| **帶追溯** | 需求/規則清單帶追溯欄位，不另開文件 | 另開文件會誘發下游反推 |
| **可引用** | 每個規則/需求有唯一編號 | 無法精準追溯，「如前所述的那個驗證」 |
| **可驗證** | 每條需求都能轉成 true/false 判斷 | 「介面應友善好用」無法驗證 |
| **載入原文** | AI Session 載入原始文件，不載入上個 Session 的產出 | 語義漂移累積 |

### 5.4 組裝原則

每次 AI 互動只載入任務需要的模組，不多不少：

```
任務：實作儲存驗證
載入：rules/validation.md + contracts/api.md + standards/CLAUDE.md
不載入：其他所有

任務：修 D01 條件顯示 bug
載入：rules/conditional.md + contracts/components.md + decisions/clarifications.md
不載入：validation, calculation, api, tests
```

## 6. Layer 2：任務品質 — Quality Gate

### 6.1 Gate 定義

```
Gate 1 — 矛盾消除
  問題：我要用的 context 模組之間有沒有矛盾？
  誰守：人類（AI 可協助比對，但人類做最終判斷）
  觸發：新功能、需求變更、規格更新
  跳過：Bug fix（已有明確的現狀 vs 預期）
  產出：更新後的 context 模組 + decisions/clarifications.md

Gate 2 — 需求可驗證
  問題：這次任務的每條需求能不能轉成 true/false？
  誰守：人類
  觸發：新功能、需求變更
  跳過：Bug fix、純樣式調整
  產出：requirements/*.md 中的新增/更新條目

Gate 3 — 架構對齊
  問題：技術方案會不會跟專案既有架構衝突？
  誰守：人類（定向取樣專案，不全盤掃描）
  觸發：新增 API、新元件、新資料表
  跳過：修改既有元件內部邏輯
  產出：contracts/*.md 中的新增/更新條目

Gate 4 — 測試先行
  問題：可執行的單元測試規格是否存在且由人類審核？
  誰守：人類寫規格 → AI 產生測試碼 → 人類審核測試碼
  觸發：任何涉及業務邏輯的程式碼生成
  跳過：純 UI 樣式調整
  產出：tests/unit-specs.md 中的新增條目

Gate 5 — 追溯回填
  問題：產出的程式碼是否對應到具體的 REQ/BR 編號？
  誰守：AI 回填 + 人類抽查
  觸發：每次 AI 產出後，立即執行
  跳過：探索性 prototype（標記為不進主分支）
  產出：requirements/*.md 中追溯欄位的更新
```

### 6.2 Gate 詳細指引

#### Gate 1：矛盾消除（對應原 Phase 0）

**工作內容：**

```
步驟 1：比對所有文件，列出矛盾點
步驟 2：與 SA/PM 逐項確認，產出釐清紀錄
步驟 3：將確認結果記錄（不修改原始文件，產出獨立的釐清紀錄）
步驟 4：標記仍待確認的項目，不讓 AI 對這些部分做假設
```

**AI 的角色：**
- 交叉比對多份文件找出矛盾（AI 擅長）
- 列出需要確認的問題清單（AI 擅長）
- 決定哪個版本是正確的（人類決策）
- 解決業務層面的模糊地帶（人類決策）

#### Gate 2：需求可驗證（對應原 Phase 1）

**關鍵特徵：**
- **不碰技術**：不討論用什麼框架、什麼元件
- **不碰專案**：不需要知道專案現在怎麼做的
- **純業務語言**：SA/PM 應該能看懂每一條內容
- **帶追溯欄位**：為後續回填預留空間，一份文件貫穿全程

#### Gate 3：架構對齊（對應原 Phase 2）

**取樣目標：**
- 共用元件的 props/events 介面
- 既有 API 的分層模式（Controller → Service → Repository）
- 資料庫連線與 ORM 用法
- 認證/授權的取得方式
- 目錄結構與命名慣例

**關鍵特徵：**
- **必須參考專案**：但是定向取樣，不是全盤掃描
- **技術語言**：工程師之間的溝通
- **對齊既有架構**：不是從零設計，是融入現有系統
- **不寫代碼**：只定義結構和介面

#### Gate 4：測試先行

前提：單元測試規格由人類撰寫或審核，AI 不得自行修改已通過審核的測試。

#### Gate 5：追溯回填

每完成一個任務，立即回填需求清單中的追溯欄位。驗證方向永遠是：需求清單 → 代碼（需求驅動驗證），絕不反向。

### 6.3 任務類型 vs Gate 觸發矩陣

| 任務類型 | Gate 1 | Gate 2 | Gate 3 | Gate 4 | Gate 5 |
|---------|--------|--------|--------|--------|--------|
| 大型新功能 | ✅ | ✅ | ✅ | ✅ | ✅ |
| 中型需求變更 | ✅ | ✅ | 視情況 | ✅ | ✅ |
| 小型邏輯修改 | — | — | — | ✅ | ✅ |
| Bug fix | — | — | — | 視情況 | ✅ |
| 純 UI 樣式 | — | — | — | — | — |
| 探索性 prototype | — | — | — | — | — |

### 6.4 預設全觸發規則

以下情況強制觸發所有 Gate，不允許自行判斷跳過：
- 工程師對該業務領域不熟悉
- 跨模組修改（改 A 可能影響 B）
- 第一次用 AI 做的功能類型

## 7. Layer 1：互動品質 — Execution Checklist

### 7.1 每次 AI 互動前

```
□ 範圍：這次任務能用一句話說清嗎？
  → 不能 = 任務太大，先拆分

□ Context：載入的檔案只包含這次任務需要的嗎？
  → 載了不相關的 = 移除，提高信噪比

□ 判斷：有沒有需要 AI「選擇」或「猜測」的地方？
  → 有 = 停下來，先由人類做完判斷，寫進 context 模組

□ 測試：有沒有可驗證的預期結果？
  → 沒有 = 先補（單元測試規格或人工驗收條件）

□ 邊界：有沒有「不要做的事」需要明確告知？
  → 有 = 寫進 prompt（例如「不要新增 API」「不要改 schema」）
```

### 7.2 任務拆分原則

```
錯誤：「請根據規格實作 IM004 庫存異動單的全部功能」
正確：「請根據以下規格實作查詢功能，需滿足 REQ-Q01~Q05」
```

| 原則 | 說明 |
|------|------|
| **邏輯邊界** | 按功能模組拆分（查詢、新增編輯、過帳），而非按單一元件 |
| **明確介面** | 告訴 AI 本次任務與其他模組的介面契約 |
| **附帶測試** | 每個任務附帶對應的測試案例編號 |
| **參考規範** | 附帶專案規範中相關的段落 |

**粒度選擇依據：**

| 條件 | 建議粒度 |
|------|---------|
| 模組間耦合低、有明確測試案例 | 較大粒度（一個功能模組，含前後端） |
| 模組間有複雜互動、業務規則多 | 較小粒度（拆開前端/後端，或拆開核心邏輯/邊界處理） |
| 使用 Agentic 工具（可自主跑測試迭代） | 可嘗試較大粒度，AI 能自行驗證並修正 |
| 使用純對話模式（無 tool use） | 維持較小粒度，人工驗證每步產出 |

### 7.4 執行順序建議

```
1. 資料模型層（Entity / Model / Migration）
   ↓ 驗證 schema 正確
2. API 層（Controller / Service / Repository）
   ↓ 驗證 API 可用
3. 前端元件（由內而外）
   3.1 純展示元件（無狀態依賴）
   3.2 表單元件（有驗證邏輯）
   3.3 容器元件（組合子元件、管理狀態）
   3.4 頁面元件（路由、權限）
   ↓ 驗證前後端整合
4. 業務邏輯（過帳、傳票、成本更新）
   ↓ 驗證核心流程
5. 邊界情境（錯誤處理、極端案例）
```

### 7.5 Session 管理

1M context 模型下，中等複雜度的功能可以在單一 session 中完成，不必預設拆分。

**人工關注 context 品質的做法：**

- 在對話過程中主動檢查 AI 的產出是否仍與原始指令一致
- 發現偏差時，直接在對話中重申關鍵規則或重新載入源頭文件片段
- 當對話累積過長、感覺 AI 開始「漫不經心」時，主動開新 session

**需要開新 session 的判斷由人類做，而非依賴固定規則。**

如果需要拆分，按邏輯邊界拆（前端/後端/業務邏輯），而非按元件粒度。每個新 session 載入源頭原文，不載入前次對話。

每個 Session 開始時載入：
1. 專案規範（固定，來自 `standards/`）
2. 本次任務相關的需求/規則/測試（精選，來自源頭模組）
3. 已完成的介面定義（供參考，來自 `contracts/`）

**不要**載入：原始規格書、HTML 原型、AI 上個 Session 的對話紀錄、無關的代碼。

**Context 管理的最佳策略會隨模型演進而改變。** 使用任何模型前，應查閱該模型當下的 context 研究與官方建議。

**Claude 模型的 Context 研究現況（2026/03）：**

| 主題 | 發現 | 來源 |
|------|------|------|
| Context rot | 隨 token 增加準確率漸進衰減。Opus 4.6 @ 1M MRCR v2 準確率 76-78% | Anthropic, Chroma |
| 位置偏見 | 仍然存在，前段資訊準確率較高，但已比 2023 年大幅改善 | Chroma |
| Compaction | ~95% context 自動 compact；CLAUDE.md 從磁碟重新讀取 | Anthropic |
| Context editing | 自動清除過時 tool call 結果，29% 複雜任務改善、84% token 節省 | Anthropic |
| Extended thinking | 前次 thinking block 自動移除，不佔 context | Anthropic |

### 7.6 每次 AI 產出後

```
□ 幻覺：有沒有呼叫不存在的 API / 使用不存在的 props？
□ 命名：與專案慣例一致嗎？
□ 作弊：AI 有沒有刪除或修改既有測試？
□ 殘留：有沒有留下除錯用的變體檔案或 console.log？
□ 追溯：對應的 REQ/BR 編號已回填到 requirements/*.md？
□ 多餘 fallback：AI 是否加入過度的防禦性代碼（?.、?? ''、try-catch everywhere）？
□ 命名漂移：同一個概念在不同檔案中是否用不同名稱？
□ 硬編碼：把應該查表的值寫死在前端？
□ 樣板殘留：從範例複製後忘記改的殘留（如註解中的舊名稱）？
```

## 8. 測試與驗證

### 8.1 測試粒度分工

```
AI 負責小粒度（單元測試）
  驗證規則 → AI 寫單元測試 + AI 寫實作
  計算規則 → AI 寫單元測試 + AI 寫實作
  條件邏輯 → AI 寫單元測試 + 人工確認畫面

人工負責大粒度（業務驗收）
  過帳流程 → 人工按 TC 清單操作
  查詢功能 → 人工按 TC 清單操作
  跨元件互動 → 人工操作確認
  視覺正確性 → 人工目視確認
```

前提：單元測試規格由人類撰寫或審核，AI 不得自行修改已通過審核的測試。

### 8.2 驗證層次

| Level | 驗證內容 | 驗證方式 | AI 自主驗證 |
|-------|---------|---------|------------|
| 1 | 語法正確 | lint、build | ✅ Agentic 工具可自主執行並修正 |
| 2 | 介面對齊 | 整合測試 | ✅ 可自主跑測試、比對介面契約 |
| 3 | 邏輯正確 | 單元測試 + Review | ⚠️ 測試可自主跑，但邏輯審查需人工 |
| 4 | 需求覆蓋 | 需求清單回填 | ❌ 需人工逐條確認實作符合需求意圖 |
| 5 | 體驗正確 | 人工測試 | ❌ 必須人工操作 |

Level 1-2 可交由 Agentic 工具即時完成（生成 → 跑測試 → 修正），不必等到最終驗證。最終驗證的核心價值在 Level 3-5——這些是 AI 無法自我驗證的層次。

### 8.3 最終核對

回到需求清單，逐條確認：

```
檢查方向：需求清單 → 代碼（需求驅動驗證）

對每一條 REQ/BR：
  1. 實作位置欄是否已填？ → 沒填 = 遺漏
  2. 對應測試是否通過？   → 沒過 = 邏輯錯誤
  3. 實作是否符合需求描述？→ 不符 = 語義漂移

絕對不做：代碼 → 反推需求（下游反推）
```

### 8.4 完成標準

- [ ] 需求清單所有 REQ 的狀態欄為 ✅
- [ ] 業務規則清單所有 BR 的狀態欄為 ✅
- [ ] 測試案例清單所有 TC 的狀態欄為 ✅
- [ ] 無未回填的追溯欄位
- [ ] 人工操作測試通過

## 9. 流程規模選擇

流程的重量應匹配任務的風險，且取決於文件體系的現況。

### 9.1 已有完整文件體系

文件體系越健康，後續變更的流程越輕。因為源頭文件已存在，變更只需要「更新源頭 → 執行 → 回填」。

| 任務類型 | 做法 |
|---------|------|
| Bug fix | 定位對應的 REQ/BR → 修代碼 → 更新 TC 狀態。不需重走任何 Gate |
| 小需求變更（改欄位、調邏輯、加選項） | 更新 requirements/ → 執行 → 回填追溯 |
| 中等需求變更（新增子功能、新 API） | 新增 REQ/BR/TC → 評估是否需更新 contracts/ → 執行 |
| 大型新功能 | Gate 1-5 完整流程 |

關鍵：**任何變更都先回到源頭文件更新，再往下執行。**

### 9.2 尚無文件體系（legacy、中途導入）

| 任務類型 | 做法 |
|---------|------|
| Bug fix | 直接修正，不需補文件 |
| 小改動 | 直接執行 |
| 中大型功能 | 走完整 Gate 流程，同時建立該功能的文件體系 |

不建議為整個 legacy 系統補齊文件——按需建立，在每次做中大型功能時逐步擴充文件覆蓋率。

### 9.3 文件體系已存在但失真

這是最危險的狀態：團隊以為有文件可依賴，但文件內容已不反映實際代碼。

在此狀態下，源頭鐵則暫時失效——因為文件已不是可信的源頭。必須先校正，才能恢復源頭的權威性。

| 步驟 | 做法 |
|------|------|
| 1. 判斷失真範圍 | 比對 REQ/BR 與實際代碼行為，評估偏差程度 |
| 2. 決定校正範圍 | 優先全面校正。僅在系統過大時才按模組分批 |
| 3. 校正文件 | 將文件更新為代碼的實際行為——此時允許從代碼反推文件（唯一合法的下游反推場景） |
| 4. 標記已知偏差 | 如果代碼行為本身就是錯的，標記「現狀：X，應為：Y」 |
| 5. 恢復源頭 | 校正完成後，文件重新成為源頭，後續回到正常流程 |

判斷文件是否已失真的信號：
- 追溯欄位大量空白或過時
- 需求清單中的描述與實際行為不符
- 團隊成員說「文件不準，看代碼比較快」

**判斷原則：當「搞錯需求」的代價高於「更新文件」的成本時，先更新文件。**

## 10. 實務案例：IM004 庫存異動單

### 10.1 Gate 1 實際執行

對 IM004 規格進行矛盾消除，過程中發現：

| 類型 | 數量 | 舉例 |
|------|------|------|
| 規格內部矛盾 | 2 處 | 資料表名稱不一致、D02/D03 欄位寫反 |
| 規格遺漏 | 6 處 | 編號規則、利潤中心顯示條件、D01 交易欄位 |
| HTML 與規格不符 | 10 處 | 代碼格式、狀態值、多餘欄位、缺少選項 |
| 需外部文件補充 | 2 處 | SA_CO003-1、SA_CO003-2（已取得） |
| 待 SA 確認 | 7 項 | 批次控制、庫存檢查、刪除方式等 |

如果跳過 Gate 1 直接將三份文件丟給 AI，上述 27 個問題中的每一個都可能導致 AI 產出錯誤的代碼。而且這些錯誤會**靜默發生**——AI 不會告訴你它選擇了 `IM_DOC_LINE` 而非 `IM_DOC_DETAIL`。

### 10.2 投入產出比

```
Gate 1 投入：約 1 小時（問答釐清 + 文件整理）
避免的返工：預估 2~5 天（發現 AI 產出的代碼基於錯誤假設，需重新生成和修正）

ROI = 極高
```

### 10.3 關鍵學習

1. **規格書不等於 AI 的輸入**：規格書是寫給人看的，人可以用經驗補足模糊地帶；AI 不行
2. **HTML 原型是雙面刃**：提供視覺參考但包含實作錯誤，AI 會照抄錯誤
3. **釐清問答的價值**：不只是修正錯誤，更重要的是讓工程師建立對需求的完整理解
4. **待確認項目要明確標記**：告訴 AI「這個部分暫不實作」，好過讓它猜測
5. **知識管理與架構對齊必須分開**：在獨立環境定義「做什麼」，再回到專案決定「怎麼做」
