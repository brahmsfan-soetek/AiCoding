## 1. 核心觀念

### 1.1 Agentic Coding 不是「讓 Agent 自己搞定」

Agentic Coding 的本質是**人機協作的新形態**。Agent（如 Claude Code）具備自主讀取檔案、執行命令、跑測試、多輪修正的能力，但它的判斷品質完全取決於人類架構的 Context。

```
傳統開發：      需求 → 工程師理解 → 工程師寫代碼 → 測試
Prompt 時代：   需求 → 工程師整理 prompt → LLM 生成代碼 → 工程師逐行驗證
Agentic 時代：  需求 → 工程師架構 context → Agent 自主探索/實作/驗證 → 工程師審查決策點
```

工程師的角色轉變為 **Context 架構師**——決定 Agent 在什麼範圍內工作、讀取哪些資訊、遵守哪些約束、用什麼回饋迴路自我修正。

與 Prompt 時代的關鍵差異：

| | Prompt 時代 | Agentic 時代 |
|---|---|---|
| **資訊傳遞** | 人工複製貼上到 prompt | Agent 自主讀取專案檔案 |
| **驗證方式** | 人工逐行檢查產出 | Agent 跑測試自我修正，人工審查業務邏輯 |
| **錯誤修正** | 人工描述錯誤，重新生成 | Agent 讀取錯誤訊息，自主迭代修正 |
| **Context 管理** | 手動控制 prompt 內容 | 透過 CLAUDE.md、planning 文件、sub-agent 架構控制 |
| **核心技能** | Prompt Engineering | Context Engineering |

> 「找到最小的高信號 token 集合，最大化期望結果的可能性。」— Anthropic

### 1.2 基本公式

```
Agent 產出品質 = f(Context 品質, 任務邊界, 回饋迴路強度)
```

- **Context 品質**：高信噪比、無矛盾、結構化——不是「給越多越好」，而是「精準給到」
- **任務邊界**：明確的範圍與介面契約，Agent 知道「做什麼」和「不做什麼」
- **回饋迴路強度**：測試、lint、型別檢查——Agent 能自主驗證並修正的機制越強，產出越可靠

三者缺一，Agent 的自主能力反而會**放大錯誤**——它會自信地在錯誤的方向上越跑越遠。

**量化證據——為什麼需要方法論：**

| 指標 | 數值 | 來源 |
|------|------|------|
| AI 生成 PR 被拒率 vs 人類 | 67.3% vs 15.6% | LinearB |
| AI 代碼含重大問題倍率 | 1.7× | CodeRabbit 2025/12 |
| 有經驗開發者使用 AI 後的速度變化 | −19%（但自認為 +20%） | METR 2025/07 |
| 實際淨開發週期改善 | 8-13%（非行銷宣稱的 50%） | Thoughtworks |
| AI 代碼未通過安全測試 | 45% | OWASP 相關研究 |

結論不是「不該用 Agent」，而是**無紀律的 Agent 使用比不用更糟**。方法論的價值在於讓 Agent 的加速效果大於管理 Agent 的 overhead。

### 1.3 源頭鐵則

```
所有衍生物都必須從源頭長出來，絕不從下游反推。
```

需求驅動代碼，代碼不反推需求。這條規則在 Agentic 時代更重要——因為 Agent 有能力自主產出大量看似合理的代碼，讓「反推」變得更誘人也更危險。

- 需求清單 → 驅動代碼生成（正確方向）
- 代碼 → 反推需求清單（錯誤方向，會合理化錯誤）
- 業務規則 → 驅動測試案例（正確方向）
- 測試結果 → 反推業務規則（錯誤方向，會遺漏未測場景）

每多一份**不從源頭衍生**的文件，就多一個失真的入口。

**Agentic 環境的特殊風險**：Agent 能自主產出完整的測試套件和文件，這些產出看起來很專業，但如果是從代碼反推而非從需求衍生，它們只是代碼的鏡像——無法發現遺漏，只能證明「代碼做了什麼」。

## 2. LLM 的運作特性與限制

理解 LLM 的底層原理，才能設計出有效的 Context Engineering 策略。

### 2.1 LLM 的本質：從機率預測到自主行動

LLM 的運作本質是根據輸入的 token 序列，預測最可能的下一個 token。這個本質從 GPT-2 到 Opus 4.6 沒有改變。

改變的是在這個基礎上演化出的能力層次：

```
┌─────────────────────────────────────────────────────────┐
│  Layer 3: Agentic Loop（自主行動層）◀ 本文件的核心層     │
│    ├─ 讀檔、寫檔、搜尋專案                               │
│    ├─ 執行命令、跑測試、跑 lint                           │
│    ├─ Sub-agent 分工（context 隔離）                      │
│    ├─ 持久記憶（CLAUDE.md、memory 系統）                  │
│    └─ 多輪自主迭代（生成 → 驗證 → 修正 → 再驗證）         │
│  Layer 2: Extended Thinking（推理層）                     │
│    └─ 自我反駁、回溯、多步推導；thinking 不佔後續 context  │
│  Layer 1: In-Context Learning（上下文學習層）             │
│    └─ 從 CLAUDE.md 規則、專案範例中學習模式               │
│  Layer 0: Next-Token Prediction（基礎層）                 │
│    └─ 機率預測，一切的底層                                │
└─────────────────────────────────────────────────────────┘
```

每一層繼承下層的能力也繼承其限制：
- Layer 2 的推理能力讓模型能自我修正，但修正的依據仍是 context 中的資訊——源頭資訊有矛盾，推理層也會被誤導
- Layer 3 的工具使用讓模型能驗證假設，但選擇「驗證什麼」和「如何解讀結果」仍受 Layer 0 的機率傾向影響
- Layer 3 的 sub-agent 能隔離 context，但主 agent 仍需根據 sub-agent 的摘要做決策——摘要品質決定決策品質

**Agentic 開發的核心認知：Layer 3 讓 Agent 具備了自我修正的能力，但這個能力的天花板取決於回饋迴路的品質。** 能自動失敗的測試 > 只靠推理的自省 > 沒有回饋的盲目生成。

**Claude Code 的 Context 機制：**

| 機制 | 作用 | Context Engineering 的意義 |
|------|------|---------------------------|
| **CLAUDE.md** | 每次 session 自動載入，作為持久指令 | 專案規範、約束條件的 single source of truth |
| **Sub-agent** | 獨立 context 執行探索/實作，返回精煉摘要 | 防止主 agent 被大量探索結果污染 |
| **Compaction** | ~95% context 使用率時自動壓縮歷史對話（可透過環境變數調低） | 延長有效工作時間，但壓縮是有損的 |
| **Memory 系統** | 跨 session 持久記憶 | 累積專案知識，但不應取代源頭文件 |
| **Extended Thinking** | 前次 thinking block 自動移除 | 推理不佔 context，但結論仍受 context 品質影響 |
| **KV-cache** | 相同前綴的 prompt 可重用計算結果 | 保持 CLAUDE.md/planning 文件結構穩定，可大幅降低延遲與成本 |

**各機制的具體限制（2026/03，以官方文檔為準）：**

| 類別 | 限制 | 數值 | 工程意義 |
|------|------|------|---------|
| **CLAUDE.md** | 建議行數上限 | **~200 行/檔** | 超過後 Agent 遵從度下降；拆分到 `.claude/rules/` 的多個 `.md` 檔 |
| | 匯入遞迴深度 | **最多 5 層** | `@path/to/import` 的連鎖匯入上限 |
| | Token 膨脹 | **~1.7×** 原始 token | 行號格式化造成的額外消耗 |
| | Compaction 後行為 | 從磁碟重新讀取 | CLAUDE.md 的指令不會因 compaction 丟失 |
| | 子目錄 CLAUDE.md | 惰性載入 | 只在 Agent 讀取該目錄檔案時才載入 |
| **MEMORY.md** | 啟動時載入行數 | **前 200 行** | 超過 200 行的內容不會被自動載入，需 Agent 主動讀取 |
| **Context Window** | Opus 4.6 | **1M tokens**（128K 輸出上限） | 中大型功能可在單一 session 完成 |
| | Sonnet 4.6 | **1M tokens**（64K 輸出上限） | 同上，但 MRCR 準確率僅 18.5%（vs Opus 76%） |
| | Haiku 4.5 | **200K tokens**（64K 輸出上限） | 適合 sub-agent 的探索任務 |
| **Compaction** | 自動觸發閾值 | **~95% context** | 可用 `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE` 調低 |
| | 手動觸發 | `/compact` 指令 | 可附帶自訂摘要指示（如 `/compact 保留代碼範例`） |
| **Sub-agent** | 巢狀限制 | **不可巢狀** | Sub-agent 不能再生成 sub-agent |
| | Agent hook 回合上限 | **50 回合** | 超過後自動停止 |
| **Extended Thinking** | 最小 budget | **1,024 tokens** | API 層級限制 |
| | 前次 thinking block | **自動移除** | 推理過程不累積佔用 context |
| | `max` 努力等級 | **僅 Opus 4.6** | Sonnet 不支援最高推理等級 |
| **檔案讀取** | 預設行數上限 | **2,000 行** | 超過需指定 offset/limit |
| | 預設 token 上限 | **~25,000 tokens** | 可用 `CLAUDE_CODE_FILE_READ_MAX_OUTPUT_TOKENS` 調整 |
| **KV-cache** | 預設 TTL | **5 分鐘** | Prompt 結構穩定時，相同前綴可重複命中 |
| | 最小可快取 tokens | Opus 4,096 / Sonnet 2,048 | 太短的 CLAUDE.md 無法觸發快取 |
| | 快取讀取價格 | **0.1× 基礎輸入價** | 命中快取時成本僅為未命中的 1/10 |

> **來源**：https://code.claude.com/docs/en/memory, https://code.claude.com/docs/en/sub-agents, https://code.claude.com/docs/en/model-config, https://platform.claude.com/docs/en/build-with-claude/prompt-caching

### 2.2 語義漂移（Semantic Drift）

每一次 LLM 處理都是一次「轉換」。與有損壓縮（如 JPG）不同，LLM 的轉換是**有損且隨機的**——更像是**傳話遊戲**（telephone game）：每一輪不只丟失資訊，還可能**添加**不存在的細節（幻覺）。

```
路徑 A：每次從源頭載入（錨定路徑）

planning/01_需求清單.md ──→ Session 1 Agent 讀取 → 產出代碼
planning/01_需求清單.md ──→ Session 2 Agent 讀取 → 產出代碼
planning/01_需求清單.md ──→ Session 3 Agent 讀取 → 產出代碼
  ↑ 每次回到相同的源頭文件，漂移被歸零

路徑 B：以前次產出為輸入（漂移路徑）— 傳話遊戲

原始需求 → Session 1 產出 → Session 2 基於 S1 產出 → Session 3 基於 S2 ...
   100%        ~95%              ~85%                      ~70%
                          ↑ 每輪不只丟失，還可能添加錯誤細節
```

**不是「用 Agent 越多次品質越差」，而是「不回到源頭就會越用越差」。**

**走上路徑 B 的常見原因：**

| 原因 | 發生什麼 |
|------|---------|
| **沒有錨點** | 只有代碼沒有需求清單，Agent 從代碼推測意圖，推測引入偏差 |
| **口頭修改不落文件** | 「上次跟你說那個改一下」→ 新 Session 的 Agent 不知道「那個」是什麼 |
| **跨 Session 失去上下文** | 新 Session 只看到結果代碼，不知道前次的設計取捨 |
| **衍生文件取代源頭** | 用 Agent 對代碼的「理解」取代原始需求 |

**Agentic 環境特有的漂移來源：**

| 來源 | 機制 | 對策 |
|------|------|------|
| **Compaction 壓縮** | 對話達 ~95% context 時自動摘要，早期的精確指令可能被壓縮成模糊描述 | 關鍵約束放在 CLAUDE.md（不被壓縮），而非僅在對話中提及 |
| **Sub-agent 摘要** | Sub-agent 返回精煉結果，細節被過濾 | 讓 sub-agent 輸出結構化結果（表格、JSON），非散文摘要 |
| **Memory 記憶偏差** | 跨 session 的 memory 是 Agent 的「理解」，非原始事實 | Memory 存參考指引，不存業務規則本身 |
| **代碼即 context** | Agent 讀取專案中的既有代碼，錯誤代碼成為「事實」影響新代碼 | 測試作為獨立錨點，代碼的正確性由測試驗證而非 Agent 的判斷 |

**確保留在路徑 A 的方法（Claude Code 實踐）：**

| 策略 | 做法 |
|------|------|
| **CLAUDE.md 錨定** | 將不可妥協的約束寫入 CLAUDE.md，每次 session 自動載入 |
| **Planning 文件在專案內** | `planning/` 目錄存放需求/規則/測試清單，Agent 直接讀取源頭文件 |
| **編號追溯** | 代碼中標註對應的 REQ/BR 編號，偏差可被發現 |
| **測試錨定** | 測試是不會被 compaction 壓縮的固定錨點，代碼偏離時測試會失敗 |
| **變更回源** | 任何需求變更都更新 planning/ 文件，不只口頭告知 Agent |
| **session 間歸零** | 新 session 讓 Agent 重新讀取 planning/ 源頭文件，不依賴前次對話的記憶 |

**核心認知：Agent 的對話記憶是有損的，Compaction 後更是如此。檔案系統才是無損的持久記憶。** 一切決策和變更都必須落地到文件。

### 2.3 Context Window 的現實

即使 Context Window 很大（如 Opus 4.6 的 1M tokens），也不代表「全部丟進去」是好策略。

#### 位置偏見：改善但未消失

早期的「Lost in the Middle」問題（context 中段資訊被遺漏）已大幅改善，但未完全消失。根源是架構性的——RoPE（Rotary Position Embedding）天然讓模型更關注序列首尾的 token。實務影響：前段資訊的檢索準確率仍略高於中後段。

**NIAH（Needle-in-a-Haystack）不是好的衡量標準。** 它只測單一事實的精確檢索，不反映真實的多步推理場景。更有意義的 benchmark 是 **MRCR v2**（Multi-needle Retrieval）：

| 模型 | MRCR v2 @ 1M tokens | 意義 |
|------|---------------------|------|
| Claude Opus 4.6 | 76% | 大型任務可在單一 session 完成 |
| Claude Sonnet 4.5 | 18.5% | 相同任務需要更積極地拆分 session |

**模型等級的影響遠大於 prompt 技巧的差異。** 這是選擇工具時最重要的決策。

#### Context Rot：真正的瓶頸

Chroma Research（2025）測試 18 個主流模型後發現：**每個模型都隨著 context 增加而品質下降**，但下降的速度取決於：

1. **語義相似度**：問題與答案的語義距離越遠，檢索越困難——比位置效應影響更大
2. **干擾密度**：單一無關資訊降低準確率；四個干擾項會加劇惡化
3. **注意力預算**：每個新 token 都消耗有限的注意力資源，有效 context 遠小於名義容量

```
Context 的實際有效性（結合位置、語義競爭、注意力預算）：

  [精準指令 10K tokens]                        →  高品質輸出
  [精準指令 + 乾淨參考 50K tokens]              →  良好輸出
  [三份矛盾文件 100K tokens]                    →  不可預測的輸出（語義競爭）
  [大量無關代碼 500K tokens]                    →  關鍵指令被稀釋，品質下降（context rot）
```

#### Claude Code 的 Context 管理

| 機制 | 觸發條件 | 效果 | 注意事項 |
|------|---------|------|---------|
| **Compaction** | context 使用率 ~95%（可用 `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE` 調低） | 自動摘要早期對話，釋放空間；CLAUDE.md 從磁碟重新讀取注入 | 有損壓縮——結構化摘要品質 > 激進壓縮（Factory.ai 評測） |
| **Extended Thinking 自動清理** | 每一輪結束 | 前次 thinking block 移除，不佔 context | 推理過程不累積，只保留結論 |
| **Sub-agent context 隔離** | 手動或自動分派 | 子任務在獨立 context 中執行，返回精煉結果 | 主 agent 只看到摘要，不看到探索細節 |
| **CLAUDE.md 優先載入** | 每次 session 開始 | 持久指令始終在 context 前段（高注意力區域）；compaction 後從磁碟重新讀取 | 官方建議 ~200 行/檔；超過後遵從度下降 |

**核心原則：信噪比（Signal-to-Noise Ratio）比總量更重要。** 載入 50K 精準內容的效果，優於載入 500K 含噪訊的內容。Anthropic 的官方建議與此一致：「找到最小的高信號 token 集合。」

### 2.4 各層能力邊界

#### Layer 0 — Next-Token Prediction 帶來的固有特性

| 特性 | 具體表現 | Agentic 工程對策 |
|------|---------|-----------------|
| 模式複製 | 輸入中出現的模式會被放大。專案中的錯誤模式若出現多次，Agent 傾向複製 | CLAUDE.md 中明確標註「不要做的事」；確保專案範例本身正確 |
| 語義競爭 | 多份矛盾資訊共存時，模型混合而非裁決 | Phase 0 消除矛盾；CLAUDE.md 中的規則數量控制在 20 條以內 |
| 頻率偏見 | 訓練資料中常見的模式優先於罕見但正確的做法 | 少見的業務邏輯在 planning 文件中給明確指令，不假設 Agent 會推斷 |
| 自信填補 | 資訊不足時生成看似合理的內容（幻覺） | 待確認項目標記「暫不實作」；在 CLAUDE.md 中要求 Agent 遇到不確定的業務問題時詢問而非假設 |

#### Layer 1 — In-Context Learning 的能力與邊界

| 能力 | 可靠場景 | 不可靠場景 |
|------|---------|-----------|
| 遵循指令格式 | CLAUDE.md 中的結構化規則（表格、清單） | 長篇散文中夾帶的隱式要求 |
| 學習專案慣例 | Agent 自主讀取 2-3 個一致的範例檔案 | 範例之間風格不一致（Agent 會混合多種風格） |
| 套用命名規則 | 規則明確且寫在 CLAUDE.md（如「API 用 kebab-case」） | 規則有例外且未說明例外條件 |
| 遵守約束條件 | 約束集中在 CLAUDE.md 前段，數量 < 20 條 | 50+ 條規則散布在多個文件中互相競爭 |

#### Layer 2 — Extended Thinking 的能力與邊界

| 能力 | 可靠場景 | 不可靠場景 |
|------|---------|-----------|
| 多步推導 | 有明確前提和推導規則的邏輯鏈 | 需要業務直覺或組織脈絡的判斷 |
| 自我修正 | 推導過程中發現與已知事實矛盾 | 矛盾來自 context 中的錯誤資訊（會修正成錯的） |
| 方案比較 | 技術方案的 trade-off 分析（有客觀標準） | 涉及主觀偏好或政治考量的選擇 |
| 邊界推理 | 從規則推導邊界案例（如 null、空陣列） | 規則本身未涵蓋的業務邊界（需領域知識） |

#### Layer 3 — Agentic Loop 的能力與邊界 ◀ 核心

這是 Agentic Coding 的關鍵層。Agent 在這一層的表現決定了人類需要介入的程度。

| 能力 | 可靠場景 | 不可靠場景 |
|------|---------|-----------|
| **檔案探索** | 定向搜尋特定檔案或模式（grep/glob） | 需要理解整個專案的架構意圖和歷史脈絡 |
| **執行與驗證** | 有明確的測試案例、編譯檢查、lint 規則 | 需要判斷「行為是否符合使用者預期」 |
| **迭代修正** | 錯誤訊息明確指向問題（編譯錯誤、測試失敗） | 靜默的邏輯錯誤（代碼能跑但結果不對） |
| **多檔案協調** | 介面契約明確的跨檔案修改 | 隱式耦合的檔案間連動（改 A 壞 B） |
| **Sub-agent 分工** | 獨立的探索/實作任務，結果可被結構化驗證 | 需要在子任務間共享狀態或保持一致性 |
| **自主決策鏈** | 每一步都有回饋信號（測試通過/失敗） | 連續多步無回饋的決策（累積不確定性） |

**Layer 3 的工程鐵律：回饋迴路品質決定自主行動的可靠性。**

```
可靠性排序（由高到低）：

  自動化測試失敗 → 明確知道哪裡錯    → Agent 可自主修正    ✅ 可靠
  編譯/lint 錯誤  → 明確知道語法問題  → Agent 可自主修正    ✅ 可靠
  型別檢查失敗    → 知道介面不匹配    → Agent 可自主修正    ✅ 可靠
  測試全過但邏輯錯 → 沒有失敗信號     → Agent 認為已完成   ⚠️ 需要人工 review
  無測試無檢查     → 完全無回饋       → Agent 盲目自信     ❌ 不可靠
```

**實務意義：投資在回饋迴路上（寫測試、設 lint 規則、定義介面型別）的 ROI，在 Agentic 時代遠高於傳統開發。** 因為這些機制不只服務人類，更直接決定 Agent 能自主走多遠。

## 3. 反模式：常見的錯誤做法

### 3.1 反模式一：Raw Dump（原始傾倒）

```
做法：把 SA 規格書、HTML 原型、會議紀錄全部丟給 AI，說「照這個做」
結果：AI 在矛盾中隨機選擇，產出包含錯誤的代碼，且難以定位問題
```

**為什麼會失敗：**

以 IM004 為例，三份文件中存在的矛盾：

| 議題 | 文件 A 說 | 文件 B 說 |
|------|----------|----------|
| 異動類型代碼 | D01~D05（規格書） | 321、551、201（HTML） |
| 資料表名稱 | IM_DOC_DETAIL（規格前段） | IM_DOC_LINE（規格後段） |
| 文件狀態 | A、P 兩種（規格書） | DRAFT、POSTED、CANCELLED 三種（HTML） |
| D02 額外欄位 | 成本中心、報廢原因（規格書） | 成本中心、領用申請（應為正確邏輯） |

AI 遇到這些矛盾時不會報錯，而是**靜默地選擇其中一個**，甚至**混合兩者**，產出看似完整但邏輯不一致的代碼。

### 3.2 反模式二：Over-Prompting（過度提示）

```
做法：用一個超長的 prompt 包含所有規則、所有例外、所有邊界條件
結果：AI 遺漏後段指令，或在相互矛盾的規則間產生幻覺
```

**為什麼會失敗：**

LLM 的注意力機制對長文本的處理有衰減效應。一個 5000 行的 prompt 中，第 4000 行的指令被遵守的機率明顯低於第 100 行。

### 3.3 反模式三：One-Shot Everything（一次完成全部）

```
做法：讓 AI 一次生成整個功能的前端 + 後端 + 測試
結果：各部分的介面不一致，錯誤發生時難以定位，修改一處牽動全局
```

**為什麼會失敗：**

單次生成的代碼量越大，AI 在後段「忘記」前段設定的機率越高，且工程師難以逐行驗證大量生成的代碼。

### 3.4 反模式四：No Verification（不驗證就合併）

```
做法：AI 生成的代碼看起來合理，直接合併進主分支
結果：隱含的邏輯錯誤在後期暴露，修復成本倍增
```

**為什麼會失敗：**

AI 生成的代碼語法正確率很高，但**業務邏輯正確率取決於輸入品質**。語法正確≠邏輯正確，看起來能跑≠行為符合需求。

### 3.5 反模式五：Downstream Backfill（下游反推）

```
做法：AI 寫完代碼後，再根據代碼反推需求清單或測試案例
結果：需求清單變成代碼的描述，而非代碼的依據；錯誤被合理化而非被發現
```

**為什麼會失敗：**

這違反了源頭鐵則。當需求清單是從代碼反推出來的，它只能證明「代碼做了什麼」，不能證明「代碼做對了什麼」。遺漏的功能不會被發現，因為反推出來的清單裡根本沒有它。

```
正確：需求清單(Phase 1) → 代碼(Phase 3) → 用需求清單驗證代碼(Phase 4)
                ↑ 源頭                            ↑ 錨點

錯誤：代碼(Phase 3) → 反推需求清單 → 「驗證通過」
                            ↑ 這份清單只是代碼的鏡像，不是獨立的驗證依據
```

## 4. 五階段開發流程總覽

### 4.1 階段定義

```
Phase 0        Phase 1         Phase 2          Phase 3         Phase 4
文件整備        需求定義         技術設計          AI Coding        驗證收斂
──────        ──────         ──────          ──────         ──────
消除矛盾        做什麼           怎麼做            執行             做對了嗎
```

| Phase | 核心問題 | 工作環境 | AI 角色 | 產出 |
|-------|---------|---------|--------|------|
| 0 | 規格說了什麼？有沒有矛盾？ | 獨立資料夾 | 協助分析 | 規格釐清紀錄 |
| 1 | 要做什麼？規則是什麼？ | 獨立資料夾 | 協助討論 | 需求清單、業務規則、測試案例 |
| 2 | 怎麼做？用什麼結構？ | 獨立 + 定向取樣專案 | 協助設計 | 資料模型、API 契約、元件拆解、專案規範 |
| 3 | 寫代碼 | 專案內 | 主力執行 | 可執行的代碼 |
| 4 | 做對了嗎？ | 專案內 | 協助測試 | 測試結果、需求清單回填 |

### 4.2 為什麼分五個階段

Phase 1 和 Phase 2 在舊版方法論中是合併的，拆開的原因：

| | Phase 1（需求定義） | Phase 2（技術設計） |
|---|---|---|
| **關注點** | 業務面：做什麼 | 技術面：怎麼做 |
| **所需輸入** | 規格書、釐清紀錄 | Phase 1 產出 + 專案現有架構 |
| **工作環境** | 獨立資料夾（隔離噪訊） | 需要參考專案（對齊架構） |
| **決策者** | SA/PM/工程師（業務判斷） | 工程師（技術判斷） |
| **風險** | 需求遺漏、規則矛盾 | 與既有架構不一致 |

如果合併在一起，會出現兩個問題：
1. 在獨立資料夾做技術設計 → 無法對齊專案架構，等到 Phase 3 才發現設計不合
2. 在專案資料夾做需求定義 → 被既有代碼影響判斷，失去業務獨立性

### 4.3 工作環境策略

```
Phase 0-1：獨立資料夾
  理由：純粹思考「要做什麼」，不被「現在怎麼做的」干擾
  AI 的 context 只有規格相關文件，信噪比最高

Phase 2：獨立資料夾 + 定向取樣專案
  理由：需要對齊專案架構，但不要被全部代碼淹沒
  做法：指定讀取特定檔案（共用元件、既有 API 模式），不掃描整個專案

Phase 3-4：專案內
  理由：必須在真實環境中執行和驗證
```

**定向取樣的做法：**

```
在獨立資料夾的 AI Session 中：

「請讀取 /project/src/components/TablesComponent.vue，
  我需要了解它的 props 和 events 介面」

「請讀取 /project/src/api/ 目錄下任一支既有 API，
  我需要了解路由、Controller、Service 的分層方式」

「請讀取 /project/src/composables/useAuth.js，
  我需要了解 COMPANY_CODE 的取得方式」
```

這樣既保持了獨立資料夾的乾淨 context，又取得了對齊專案所需的關鍵資訊。

**Agentic 工具下的調整：**

現代 AI Coding 工具（如 Claude Code、Cursor）具備自主讀取檔案的能力，「獨立資料夾」的物理隔離不再是唯一手段。隔離的目的不變（控制信噪比），但實現方式可以是：

- 傳統方式：物理上獨立的資料夾，手動複製需要的參考檔案
- Agentic 方式：在專案目錄內工作，但透過 prompt 限定 scope

選擇依據不是「哪個更先進」，而是哪個在當下更高效：手動複製 3 個檔案可能比等 AI 搜尋 30 秒更快。Agentic 工具的優勢在探索未知時才顯現。

### 4.4 核心原則

| 原則 | 說明 |
|------|------|
| **源頭鐵則** | 所有衍生物從源頭長出來，絕不從下游反推 |
| **Single Source of Truth** | 每個事實只在一個地方定義，消除矛盾 |
| **Structured Over Prose** | 結構化清單優於散文式描述，AI 解析更準確 |
| **Small Batch** | 小批量、可驗證的任務，優於大範圍一次性生成 |
| **Test First** | 先定義預期行為，再讓 AI 實作，有錨點可對照 |
| **Human Decides, AI Executes** | 業務決策由人類做，代碼生成由 AI 做 |

### 4.5 流程規模選擇

流程的重量應匹配任務的風險，且取決於文件體系的現況。

**已有完整文件體系（從 Phase 0 開始建立）：**

文件體系越健康，後續變更的流程越輕。因為源頭文件已存在，變更只需要「更新源頭 → 執行 → 回填」。

| 任務類型 | 做法 |
|---------|------|
| Bug fix | 定位對應的 REQ/BR → 修代碼 → 更新 TC 狀態。不需重走任何 Phase |
| 小需求變更（改欄位、調邏輯、加選項） | 更新 Phase 1 文件（需求或規則）→ Phase 3 執行 → 回填追溯 |
| 中等需求變更（新增子功能、新 API） | 在 Phase 1 新增 REQ/BR/TC → 評估是否需更新 Phase 2 → Phase 3 執行 |
| 大型新功能 | Phase 0-4 完整流程 |

關鍵：**任何變更都先回到 Phase 1 源頭文件更新，再往下執行。** 這就是源頭鐵則在維護階段的體現。

**尚無文件體系（legacy、中途導入）：**

| 任務類型 | 做法 |
|---------|------|
| Bug fix | 直接修正，不需補文件 |
| 小改動 | 直接執行 |
| 中大型功能 | 走 Phase 0-4，同時建立該功能的文件體系 |

不建議為整個 legacy 系統補齊文件——按需建立，在每次做中大型功能時逐步擴充文件覆蓋率。

**文件體系已存在但失真（文件與代碼脫節）：**

這是最危險的狀態：團隊以為有文件可依賴，但文件內容已不反映實際代碼。常見原因：口頭改需求未更新文件、多人同時修改、追溯欄位未維護。

在此狀態下，源頭鐵則暫時失效——因為文件已不是可信的源頭。必須先校正，才能恢復源頭的權威性。

| 步驟 | 做法 |
|------|------|
| 1. 判斷失真範圍 | 比對 REQ/BR 與實際代碼行為，評估偏差程度 |
| 2. 決定校正範圍 | 優先全面校正。僅在系統過大、無法一次完成時才按模組分批 |
| 3. 校正文件 | 將文件更新為代碼的實際行為——此時允許從代碼反推文件（這是唯一合法的下游反推場景，僅限於源頭已失真時） |
| 4. 標記已知偏差 | 如果代碼行為本身就是錯的，在文件中標記「現狀：X，應為：Y」 |
| 5. 恢復源頭 | 校正完成後，文件重新成為源頭，後續變更回到正常流程 |

如分批校正，必須明確標記哪些模組已校正、哪些尚未校正，避免「部分可信、部分不可信卻分不清楚」的狀態。

判斷文件是否已失真的信號：
- 追溯欄位大量空白或過時
- 需求清單中的描述與實際畫面/API 行為不符
- 團隊成員說「文件不準，看代碼比較快」

**判斷原則：當「搞錯需求」的代價高於「更新文件」的成本時，先更新文件。**

## 5. Phase 0：文件整備 — 消除矛盾

### 5.1 目的

將所有原始輸入（規格書、HTML 原型、會議紀錄）整合成**無矛盾的事實基礎**。

### 5.2 工作內容

```
步驟 1：比對所有文件，列出矛盾點
步驟 2：與 SA/PM 逐項確認，產出釐清紀錄
步驟 3：將確認結果記錄（不修改原始文件，產出獨立的釐清紀錄）
步驟 4：標記仍待確認的項目，不讓 AI 對這些部分做假設
```

### 5.3 AI 的角色

Phase 0 中 AI 的角色是**協助分析**，而非做決策：

- 交叉比對多份文件找出矛盾（AI 擅長）
- 列出需要確認的問題清單（AI 擅長）
- 決定哪個版本是正確的（人類決策）
- 解決業務層面的模糊地帶（人類決策）

### 5.4 產出

- **規格釐清紀錄**：已確認事項 + 待確認備註 + HTML 需修正項目

### 5.5 完成檢查

- [ ] 所有文件中的資料表名稱一致
- [ ] 所有欄位名稱、型態、約束有唯一定義
- [ ] 所有業務規則無互相矛盾
- [ ] 所有待確認項目已標記處理方式（暫不實作 / 用預設值 / 等回覆）
- [ ] HTML 原型的已知錯誤已記錄

## 6. Phase 1：需求定義 — 做什麼

### 6.1 目的

從整合後的規格中，提煉出**純業務面的結構化定義**。這個階段只關心「做什麼」，不碰「怎麼做」。

### 6.2 工作環境

**獨立資料夾**，不在專案目錄中。隔離專案代碼的噪訊，確保需求定義來自規格而非既有實作。

### 6.3 產出文件

#### 6.3.1 需求清單（Requirements List）

每條需求獨立編號、可測試、無歧義。**末尾保留追溯欄位**，Phase 3-4 時回填。

```markdown
| 編號 | 需求描述 | 實作位置 | 測試 | 狀態 |
|------|---------|---------|------|------|
| REQ-Q01 | 查詢頁面提供篩選條件：文件編號、異動類型、文件狀態、文件日期起迄 | | | ☐ |
| REQ-Q02 | 異動類型下拉選單來源：PF_TYPE = 'INV_MVT'，包含 D01~D05 | | | ☐ |
| REQ-Q03 | 文件狀態下拉為固定值：A(暫存)、P(已過帳) | | | ☐ |
| REQ-Q04 | 查詢結果欄位：文件編號、異動類型、文件狀態、參考文件、項目數、總金額、文件日期、過帳時間、操作 | | | ☐ |
| REQ-Q05 | 已過帳僅顯示「檢視」；暫存顯示「編輯」與「刪除」 | | | ☐ |
```

**為什麼帶追溯欄位：**
- Phase 1 時後三欄是空的，Phase 3-4 逐步回填
- 一份文件兩個用途：需求定義 + 驗收核對
- 避免另開「追溯矩陣」導致下游反推的風險（見反模式五）

#### 6.3.2 業務規則清單（Business Rules）

逐條列出，標明觸發時機、條件、動作。同樣帶追溯欄位。

```markdown
## 條件顯示規則
```

## 7. Phase 2：技術設計 — 怎麼做

### 7.1 目的

將 Phase 1 的業務需求翻譯成**可直接驅動 AI Coding 的技術方案**，並確保與專案既有架構對齊。

### 7.2 工作環境

**獨立資料夾 + 定向取樣專案**。不搬進專案目錄，但需要讀取專案中的關鍵檔案。

取樣目標：
- 共用元件的 props/events 介面
- 既有 API 的分層模式（Controller → Service → Repository）
- 資料庫連線與 ORM 用法
- 認證/授權的取得方式
- 目錄結構與命名慣例

**定向取樣的執行方式選擇：**

| 情境 | 建議方式 | 理由 |
|------|---------|------|
| 你知道要看哪 1-3 個檔案 | 手動複製貼上 | 比 AI 搜尋更快、零額外 token 消耗 |
| 你知道要看什麼類型但不確定在哪 | 給 AI 明確搜尋指令（如「找 src/api/ 下的 Controller」） | 範圍受控，避免大規模掃描 |
| 你不確定專案中有什麼慣例 | 讓 AI 搜尋，但限定範圍和數量（如「找 2 個典型的 API 範例」） | 探索性質，但需要設上限 |

避免的做法：讓 AI 無限制地「了解整個專案架構」——這會掃描大量檔案，消耗 token 且引入噪訊，違反信噪比原則。

## 8. Phase 3：Agentic Coding — 執行

### 8.1 工作環境

**專案內**。此時 AI 需要存取完整的專案環境來生成、編譯、測試代碼。

### 8.2 任務拆分原則

```
錯誤：「請根據規格實作 IM004 庫存異動單的全部功能」
正確：「請根據以下規格實作查詢功能，需滿足 REQ-Q01~Q05」
```

| 原則 | 說明 |
|------|------|
| **邏輯邊界** | 按功能模組拆分（查詢、新增編輯、過帳），而非按單一元件 |
| **明確介面** | 告訴 AI 本次任務與其他模組的介面契約 |
| **附帶測試** | 每個任務附帶對應的測試案例編號 |
| **參考規範** | 附帶專案規範中相關的段落 |

**粒度選擇依據：**

| 條件 | 建議粒度 |
|------|---------|
| 模組間耦合低、有明確測試案例 | 較大粒度（一個功能模組，含前後端） |
| 模組間有複雜互動、業務規則多 | 較小粒度（拆開前端/後端，或拆開核心邏輯/邊界處理） |
| 使用 Agentic 工具（可自主跑測試迭代） | 可嘗試較大粒度，AI 能自行驗證並修正 |
| 使用純對話模式（無 tool use） | 維持較小粒度，人工驗證每步產出 |

### 8.4 執行順序建議

```
1. 資料模型層（Entity / Model / Migration）
   ↓ 驗證 schema 正確
2. API 層（Controller / Service / Repository）
   ↓ 驗證 API 可用
3. 前端元件（由內而外）
   3.1 純展示元件（無狀態依賴）
   3.2 表單元件（有驗證邏輯）
   3.3 容器元件（組合子元件、管理狀態）
   3.4 頁面元件（路由、權限）
   ↓ 驗證前後端整合
4. 業務邏輯（過帳、傳票、成本更新）
   ↓ 驗證核心流程
5. 邊界情境（錯誤處理、極端案例）
```

**為什麼這個順序：**
- 先建立穩定的地基（資料模型），再往上蓋
- 前端由內而外：內層元件穩定後，外層組合時不會因內層變動而連鎖出錯
- 業務邏輯放後面：確保基礎設施正確後，再處理複雜的業務流程

### 8.5 Session 管理與 Context 品質

1M context 模型下，中等複雜度的功能可以在單一 session 中完成，不必預設拆分。Session 拆分是一個工具，不是必要儀式。

**人工關注 context 品質的做法：**

- 在對話過程中主動檢查 AI 的產出是否仍與原始指令一致
- 發現偏差時，直接在對話中重申關鍵規則或重新載入源頭文件片段
- 當對話累積過長、感覺 AI 開始「漫不經心」時，主動開新 session

**需要開新 session 的判斷由人類做，而非依賴固定規則。**

如果需要拆分，按邏輯邊界拆（前端/後端/業務邏輯），而非按元件粒度。每個新 session 載入 Phase 1/2 原文，不載入前次對話。

**Context 管理的最佳策略會隨模型演進而改變。** 使用任何模型前，應查閱該模型當下的 context 研究與官方建議。以下為 2026/03 時間點的參考：

**Claude 模型的 Context 研究現況（2026/03）：**

| 主題 | 發現 | 來源 |
|------|------|------|
| Context rot | 隨 token 增加，檢索準確率漸進衰減（非硬斷崖）。Opus 4.6 在 1M tokens MRCR v2 準確率 76-78%，Sonnet 4.5 僅 18.5%——模型等級差異巨大 | Anthropic, Chroma Research |
| 位置偏見 | 仍然存在——前段資訊的檢索準確率較高，但已比 2023 年大幅改善 | Chroma "Context Rot" study |
| Compaction | Claude Code 在 ~95% context 使用率時自動 compact（可透過 `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE` 調整）。Compaction 後 CLAUDE.md 從磁碟重新讀取注入 | Anthropic Engineering |
| Context editing | 自動清除過時的 tool call 結果，可帶來 29% 複雜任務改善、84% token 節省 | Anthropic Blog |
| Extended thinking | 前次的 thinking block 會自動從後續回合中移除，不佔用 context | Anthropic API Docs |

**核心原則（Anthropic 官方）：** 「找到最小的高信號 token 集合，最大化期望結果的可能性。」

**延伸閱讀：**
- Effective Context Engineering for AI Agents: https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents
- Context Rot 研究: https://research.trychroma.com/context-rot
- Claude Context Windows 官方文件: https://platform.claude.com/docs/en/build-with-claude/context-windows

每個 Session 開始時載入：
1. 專案規範（固定，來自 Phase 2）
2. 本次任務相關的需求/規則/測試（精選，來自 Phase 1 原文）
3. 已完成的介面定義（供參考，來自 Phase 2 原文）

**不要**載入：原始規格書、HTML 原型、AI 上個 Session 的對話紀錄、無關的代碼。

## 9. Phase 4：驗證收斂 — 做對了嗎

### 9.1 驗證層次

| Level | 驗證內容 | 驗證方式 | AI 自主驗證 |
|-------|---------|---------|------------|
| 1 | 語法正確 | lint、build | ✅ Agentic 工具可自主執行並修正 |
| 2 | 介面對齊 | 整合測試 | ✅ 可自主跑測試、比對介面契約 |
| 3 | 邏輯正確 | 單元測試 + Review | ⚠️ 測試可自主跑，但邏輯審查需人工 |
| 4 | 需求覆蓋 | 需求清單回填 | ❌ 需人工逐條確認實作符合需求意圖 |
| 5 | 體驗正確 | 人工測試 | ❌ 必須人工操作 |

Level 1-2 可交由 Agentic 工具在 Phase 3 中即時完成（生成 → 跑測試 → 修正），不必等到 Phase 4。Phase 4 的核心價值在 Level 3-5——這些是 AI 無法自我驗證的層次。

### 9.2 用需求清單做最終核對

Phase 4 的核心動作是回到 Phase 1 的需求清單，逐條確認：

```
檢查方向：需求清單 → 代碼（需求驅動驗證）

對每一條 REQ/BR：
  1. 實作位置欄是否已填？ → 沒填 = 遺漏
  2. 對應測試是否通過？   → 沒過 = 邏輯錯誤
  3. 實作是否符合需求描述？→ 不符 = 語義漂移

絕對不做：代碼 → 反推需求（下游反推）
```

### 9.3 AI 生成代碼的 Review 重點

AI 生成的代碼有特定的常見問題模式，Review 時應特別注意：

| 檢查項目 | 說明 |
|---------|------|
| **幻覺 API** | AI 可能呼叫不存在的 API 或使用不存在的 props |
| **多餘的 fallback** | AI 傾向加入過度的防禦性代碼（?.、?? ''、try-catch everywhere） |
| **命名漂移** | 同一個概念在不同檔案中用不同名稱（mvtType / movementType / moveType） |
| **業務邏輯硬編碼** | 把應該查表的值（如異動類型清單）寫死在前端 |
| **忽略邊界** | 空陣列、null 值、併發情境的處理不完整 |
| **樣板複製** | 從範例複製後忘記改的殘留（如註解中的舊名稱） |

### 10.2 文件流向

```
原始規格 ──→ Phase 0 釐清 ──→ Phase 1 需求/規則/測試
                                      │
                                      ↓
               專案取樣 ──→ Phase 2 技術設計
                                      │
                                      ↓
                              Phase 3 載入 Phase 1 + Phase 2 原文 → 生成代碼
                                      │
                                      ↓
                              Phase 4 回到 Phase 1 需求清單 → 回填驗證

箭頭方向 = 資訊流向，永遠從上游到下游，不反向。
```

### 10.3 文件鐵則

| 鐵則 | 說明 | 違反後果 |
|------|------|---------|
| **源頭鐵則** | 所有文件從源頭衍生，不從下游反推 | 錯誤被合理化，遺漏不被發現 |
| **單一定義** | 每個事實只在一個文件中定義 | 修改時漏改其中一份，產生矛盾 |
| **結構化** | 用表格、清單，不用散文 | AI 解析不準確，人類也難以核對 |
| **帶追溯** | 需求/規則清單帶追溯欄位，不另開文件 | 另開文件會誘發下游反推 |
| **載入原文** | AI Session 載入 Phase 1/2 的原始文件，不載入上個 Session 的產出 | 語義漂移累積 |
| **變更回源** | 需求變更時更新 Phase 1 文件，不只口頭告知或改代碼 | AI 下次 Session 讀到的還是舊需求 |

### 10.4 文件撰寫原則

| 原則 | 做法 | 反例 |
|------|------|------|
| 結構化 | 用表格、清單、代碼區塊 | 用段落散文描述欄位定義 |
| 可引用 | 每個規則/需求有唯一編號 | 「如前所述的那個驗證」 |
| 無矛盾 | 每個事實只定義在一個地方 | 在三個文件中重複寫欄位型態 |
| 最小化 | 只包含 AI 執行任務所需的資訊 | 附帶整份規格書「以防萬一」 |
| 可驗證 | 每條需求都能轉成 true/false 判斷 | 「介面應友善好用」 |

## 11. 實務案例：IM004 庫存異動單

### 11.1 Phase 0 實際執行

本次對 IM004 規格進行了完整的 Phase 0 文件整備，過程中發現：

| 類型 | 數量 | 舉例 |
|------|------|------|
| 規格內部矛盾 | 2 處 | 資料表名稱不一致、D02/D03 欄位寫反 |
| 規格遺漏 | 6 處 | 編號規則、利潤中心顯示條件、D01 交易欄位 |
| HTML 與規格不符 | 10 處 | 代碼格式、狀態值、多餘欄位、缺少選項 |
| 需外部文件補充 | 2 處 | SA_CO003-1、SA_CO003-2（已取得） |
| 待 SA 確認 | 7 項 | 批次控制、庫存檢查、刪除方式等 |

如果跳過 Phase 0 直接用方式 A（三份文件丟給 AI），上述 27 個問題中的每一個都可能導致 AI 產出錯誤的代碼。而且這些錯誤會**靜默發生**——AI 不會告訴你它選擇了 `IM_DOC_LINE` 而非 `IM_DOC_DETAIL`。

### 11.2 投入產出比

```
Phase 0 投入：約 1 小時（問答釐清 + 文件整理）
避免的返工：預估 2~5 天（發現 AI 產出的代碼基於錯誤假設，需重新生成和修正）

ROI = 極高
```

### 11.3 關鍵學習

1. **規格書不等於 AI 的輸入**：規格書是寫給人看的，人可以用經驗補足模糊地帶；AI 不行
2. **HTML 原型是雙面刃**：提供視覺參考但包含實作錯誤，AI 會照抄錯誤
3. **釐清問答的價值**：不只是修正錯誤，更重要的是讓工程師建立對需求的完整理解，這個理解會體現在 Phase 1 文件的品質中
4. **待確認項目要明確標記**：告訴 AI「這個部分暫不實作」，好過讓它猜測後產出錯誤的實作
5. **Phase 1 和 Phase 2 必須分開**：在獨立資料夾定義「做什麼」，再回到專案決定「怎麼做」，避免被既有實作帶歪需求定義
