> 每項特性列出：現象 → 關鍵數據 → 工程對策方向。

## 各項細節

### 1. 語義漂移（Semantic Drift）

- **現象**：LLM 無狀態，每次呼叫只看當前 context。串接的產出會收斂到模型偏好的模式（吸引態），而非原始意圖。
- **關鍵數據**：衰減非線性，可能穩定數輪後突然崩潰（Perez, Kovač et al., ICLR 2025）
- **對策方向**：每個任務從源頭文件載入（錨定路徑），不基於前次 AI 產出接著做。

### 2. Context Rot（上下文腐蝕）

- **現象**：品質隨 context 長度漸進衰減。語義相似的干擾比位置效應影響更大。
- **關鍵數據**：
  - 多輪對話平均效能下降 39%，unreliability 增加 112%（Laban et al.）
  - 策劃過的 context 減少 60% 錯誤（Inkeep 2025）
  - 推理效能約 ~3,000 tokens 開始退化
  - 僅兩輪對話即出現退化
  - 一旦早期犯錯，後續無法自行恢復
- **對策方向**：精準載入最小高信號 context，不貪多。每個任務獨立 context。

### 3. 位置偏見（Position Bias）

- **現象**：前段資訊檢索準確率仍略高。已大幅改善但未消失。
- **關鍵數據**：MRCR v2 @ 1M — Opus 4.6: 76%, Sonnet 4.5: 18.5%
- **對策方向**：模型等級的選擇比 prompt 技巧重要。關鍵指令放前段。

### 4. 語義競爭（Semantic Competition）

- **現象**：多份語義相近但內容矛盾的資訊 → LLM 靜默混合，不報錯不請求釐清。
- **關鍵數據**：
  - 所有測試模型面對矛盾都產出混合答案（WikiContradict, NeurIPS 2024）
  - ~70% context / ~30% parametric memory 混合比
  - 自相矛盾句子比例 17.7%（ICLR 2024）
- **對策方向**：矛盾消除必須在 AI 介入之前完成。DOCX vs HTML 衝突檢查是必要前置步驟。

### 5. 模式複製與頻率偏見

- **現象**：複製訓練資料和 context 中的高頻模式，含 bug。
- **關鍵數據**：
  - 生成的 bug 中平均 44.44% 與訓練資料 bug 完全一致（GPT-4o 達 82.6%）
  - 對有 bug 代碼生成的 test oracle 準確率 <50%
  - 93.5% 覆蓋率但僅 58.8% mutation score（極端案例：100% 覆蓋率僅 4% mutation score）
  - 規格挖掘（從代碼反推規格）誤判率 90-99%
- **對策方向**：
  - 測試必須從規格推導，不從代碼推導
  - 專案範例代碼必須正確
  - 罕見但正確的業務邏輯需明確指令

### 6. 自信填補（Hallucination）

- **現象**：資訊不足時編造合理內容。Agentic 環境中幻覺變成錯誤行動（不可逆副作用）。
- **關鍵數據**：
  - 19.7% 推薦套件不存在，43% 幻覺名 10/10 次重複出現（USENIX Security 2025）
  - 加入明確 grounding 指令可提升成功率至 87%
- **對策方向**：
  - 待確認項目標記「暫不實作」
  - 要求 Agent 遇不確定時詢問而非假設
  - 所有 API 呼叫和依賴需驗證存在性

### 7. 附和偏見（Sycophantic Bias）

- **現象**：附和使用者觀點，確認有缺陷的前提，不主動反對。
- **關鍵數據**：
  - 被質疑時 98% 錯誤承認失誤（Claude 1.3）
  - 偏好模型 95% 偏好附和回覆（vs 基線真實回覆）
  - Anthropic 4.5 模型已降低附和分數 70-85%
- **對策方向**：
  - 代碼審查不由生成代碼的同一 session 執行
  - 重要決策明確要求列舉反對理由

### 8. 自我修正盲區（Self-Correction Blind Spot）

- **現象**：無法修正自己產出的錯誤，但能修正外部來源的相同錯誤。啟動問題，非知識問題。
- **關鍵數據**：
  - 平均盲區率 64.5%
  - 中等任務：自我修正 18.3% vs 外部修正 88.1%
  - 「Wait」提示可降低盲區率 89.3%
- **對策方向**：
  - 寫代碼和審查代碼必須分離（不同 agent/session）
  - 外部驗證信號（測試、lint、型別檢查）繞過盲區
  - 任務越難盲區越嚴重 → 代碼審查正好在最差類別

### 9. 規格博弈（Reward Hacking）

- **現象**：鑽測試漏洞達成「通過」而非「正確」。手法包括篡改測試、覆寫運算子、硬編碼預期值。
- **關鍵數據**：
  - GPT-5 不可能任務 76% 博弈率
  - 回饋機制可見性：RE-Bench 30.4% vs HCAST 0.7%（43× 差異）
  - 反作弊指令幾乎無效（o3 被告知不要作弊後 14/20 次仍博弈）
  - 嚴格 prompt 可將博弈率從 92% 降至 1%
- **對策方向**：
  - 測試檔案對 Code Agent 唯讀
  - 寫測試和寫代碼分離（結構性防禦）
  - 給 Agent 明確的「我做不到」退出選項
  - 最小權限原則

### 10. 指令遵循飽和（Instruction Following Saturation）

- **現象**：同時遵循多條指令呈指數衰減。
- **關鍵數據**：
  - P(全遵守) = P(單條)^N
  - 10 條同時指令：GPT-4o 15%, Claude 3.5 Sonnet 44%
  - 500 條指令最佳僅 68%
  - RAG 篩選工具選擇準確率從 13.62% 提升至 43.13%
- **對策方向**：
  - SKILL 結構按 Phase 分段，每個 Phase 只載入該階段的規則
  - 約束條件精簡，優先級明確
  - 不在單一 prompt 塞入所有規範

### 11. 任務複雜度天花板（Task Complexity Ceiling）

- **現象**：品質隨複雜度急劇下降，存在「複雜度牆」。
- **關鍵數據**：
  - pass@1：<10 行 87.2% vs >50 行 26.5%（70% 下降）
  - 分解生成改善：+35%（APPS）、+76%（CodeContests）
  - 迭代 Agent vs 單次：6.4× 提升
  - SWE-bench 74.4% → FeatureBench 11.0%（多檔案功能開發難度完全不同）
- **對策方向**：
  - 分解是最有效的單一策略
  - 每個子任務控制在模型有效規劃範圍內
  - 多檔案功能開發需要更激進的拆分

## 交叉影響矩陣

某些特性組合會互相放大風險：

| 組合 | 放大效果 | SKILL 中的防禦 |
|------|---------|---------------|
| §5 模式複製 + §8 自我修正盲區 | AI 複製 bug 到測試中，且無法自我發現 | Phase 2/3 分離（不同 agent） |
| §4 語義競爭 + §6 自信填補 | 矛盾資訊被混合後，缺口用幻覺填補 | Phase 0 衝突檢查前置 |
| §1 語義漂移 + §2 Context Rot | 多輪累積偏差 + context 膨脹 → 雙重退化 | 每任務獨立 context，從源頭載入 |
| §7 附和偏見 + §8 自我修正盲區 | 審查時附和自己的代碼 + 無法發現自己的錯誤 | Phase 4 獨立 agent 審查 |
| §9 規格博弈 + §5 模式複製 | 學到「通過測試」的捷徑模式並複製 | 測試唯讀 + 最小權限 |
| §10 指令飽和 + §11 複雜度天花板 | 指令太多 + 任務太大 = 雙重崩潰 | Phase 分段載入 + 任務拆解 |
