## 1. 不足（判斷力外部化的洞）

### 缺-1【最尖銳】SG2 雙對照表只對「形狀」不對「語意」——D-01 會原樣再犯

D-01（`crossChartWarning` 把「跨 CHART_CODE」做成「比公司」）走一遍現行 SG2（`spec-p3-backend/SKILL.md:177-189`）：欄位名對、型別對、巢狀層對——**兩張表全過**。公式明明在 api_contract A07 的「業務說明」欄，但對照表格式沒有任何一欄強迫 Sonnet 複述「這個欄位的值是怎麼算出來的」。

更結構性的問題：`API契約清單模板.md:21` 規定計算欄位「對應 DB 欄位」填 `(計算)`——**欄位越是計算出來的（＝歧義越高），shape 對照越是空集合檢查**。保護強度與需要保護的程度成反比。

**改法**：`[processor]` / `[validator]` 含計算／比較／彙總規則者，SG2 加第三段「**業務規則複述**」：逐條列「規則原文（引 A## 業務說明或 task 規則原文欄）→ 預計實作（一句話：比什麼 key、算什麼式）」。PG 一眼可抓「比公司？錯，比 CHART_CODE」。成本每 task 幾行，是 D-01 的直接補丁。

### 缺-2 P2 task 缺「規則原文」欄（W-02 採納＋一個修正）

後端模板欄位只有 `# 類型 項目 說明 選填欄位 依賴 驗收條件`（`後端任務清單模板.md:43-45`），P2 prompt 對 `[processor]` 只要求選填欄位＋A## 指針（`任務清單_prompt.md:123-126`）——指了公式卻不抄公式，Sonnet 手上只有四個字。

- **W-02 全採納**：帶計算／業務規則的 task 加必填「規格出處＋關鍵原文**逐字**摘錄」欄；純 CRUD 不填。與缺-1 對接：SG2 第三段的「規則原文」直接取自此欄。
- **W-01 修正裁決**：「高歧義附虛擬碼」**降為選配**。虛擬碼是 P2 時刻的衍生 artifact——P2 寫錯，Sonnet 就忠實做錯，與 mock fixture 漂移同構。逐字原文才是 SSOT（同「模板逐字複製、禁改寫」的既有教訓）。
- **W-01 的「歧義 tag」不開新軸**：規則原文欄「有內容＝高歧義，SG2 走第三段；空＝低歧義，照舊」。一個欄位承載分流，避免 tag 通膨。

### 缺-3 W-03 落地：靶位是 P1 的敘述與 context，不是 P2 的機械段

1. **P1 Step 1 子 agent 化**：規格統計初稿由便宜模型子 agent 產出（規格書全文只活在會被丟棄的子 agent context），主 session 只收檔案、Opus 依自檢清單覆核。
2. **輸出紀律**：P1/P2 prompt 加一行「產出寫檔，console 只留 ≤10 行摘要；PG 審檔案不審 console」。326K 純敘述是全流程最無槓桿的 spend。
3. **反轉 session 指引**：刪 `spec-p1-digest-flow/SKILL.md:43`「同 session 繼續」；改為「各步驟以檔案為介面，S2/S4 前建議 /clear」。artifact 本來就是為此設計的，SKILL 卻鼓勵反著用。

### 缺-4（小）回饋迴路斷鏈

`session-archive.md:28-33` 指定回饋寫 `~/.soetek-ai-coding/skill-feedback/`，實際回饋落在 `serp_struct/.user/SPEC_WORKFLOW_IMPROVEMENTS.md`。兩條管道＝沒有管道，統一成一條（建議就用 target repo 的 `.user/`，因為它實際發生了）。

## 2. 多餘（注意力稅）

### 多-1【最大宗】出處引註污染執行檔

AR003 BUG-P4b / SO0062 / G2-xx / 切入點 N / 05 D9 / session-791acadc……是**維護者的 provenance，不是執行者的指令**。p3-backend 23K / p3-frontend 24.6K 估 30–40% 是 rationale／案例史／跨檔重複；且引用的 `review/2026-04-09` 不在 target 專案，執行 session 根本查不到。弱模型讀案例史不會更守規矩，只會稀釋對真指令的注意力。
→ 執行檔留一行規則；教訓史集中到 refs 的 rationale 檔一份（refs 已有雛形，但 SKILL 檔沒卸貨）。

### 多-2 SG1 是六件事的複合 gate

Scope Statement＋載入報告＋類型分佈＋前端硬守則清單（含逐條 grep pattern）＋hook 安裝三選一對話＋起始 task（`spec-p3-frontend/SKILL.md:134-158`）。弱模型對複合指令的典型行為是做四掉二。
→ hook 安裝改「每專案一次；已裝靜默；未裝一行問、預設裝」，spike (a)(b)(c) 選單留 README；SG1 收斂回「scope＋載入＋起點」三件。

### 多-3 同一規則多點全文重複

「不可從 sibling code 推 schema」含 rationale 出現 ≥10 處；「mock test 是套套邏輯」5+ 處。點用處一行提醒是對的（弱模型需要），**每處都帶完整案例史**是稅。

### 多-4 熱路徑上掛實驗品

`lightweight-review-prompt.md`（預設關）與 `hooks/README.md`（eap 抽救、介面未接線、明文「不可直接安裝」）都列在「共用規約（**動手前必讀**）」（`spec-p3-backend/SKILL.md:21-22`）——必讀清單裡放兩個讀了也不能用的檔。移出必讀清單，留 refs 即可。

### 多-5 session-archive 無消費者

每 session 收尾複製 jsonl＋產 summary 到 `~/.soetek-ai-coding/`。實際驅動改進的是手寫回饋檔；jsonl 原地永存（本次分析即直接讀原位）。砍，需要時 `/session-analyzer`。

### 多-6 G2-12「最強模型」一刀切與 W-03 經濟學矛盾

tag 系統的存在意義就是「哪些 task 便宜模型就夠」，四個 SKILL 卻各寫一條「建議最強可用模型」。把模型選擇也外部化成 tag→model 對照：`[validator]`/`[processor]`（含規則）→ 強；`[entity]`/`[sql]`/`[spi]`/`[i18n]`/`[page]` 等 → Sonnet 即可；P2 → 最強（維持，符合「Opus 定怎麼做」）。缺-1/缺-2 落地後 Sonnet 跑 P3 的可行性更高而非更低。
