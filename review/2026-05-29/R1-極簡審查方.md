## 二、eap / lite / serp 存廢初判（各一句 + 理由）

> 立場：極簡方傾向「降級/合併優於並存三套」。下為 R1 初判，R2 待與他方交鋒後可能修正。

- **eap → 降級為「主線 + eap 專案包」**：核心流程（P0-P3 + Agent 分離 + hooks）有真價值且 hooks 是全套裡唯一落實「結構性防禦＝確定性控制」的範例（block-test-edit / silent-test-pass，03 P6/P7），**但不該是獨立全量 skill**。理由：它與 serp 共享 80% 骨架，應抽成「通用 agentic-coding 主線」+「eap conventions/templates 專案包」，骨架只留一份。先決動作：刪 test01/（MN-06）、移除 Phase 日誌正文（MN-03）。

- **lite → 刪（或降級為 examples/ 下的「TM002 參考實作集」）**：它不是 skill，是一支程式的零錯誤施工說明書（MN-02）。理由：「Demo 給老闆看、模板零錯誤」是一次性交付目標，把整份正確程式碼 + 20 條硬編碼 gotcha 包成可觸發 skill，違反「skill＝可重用知識」的定義；Demo 跑完價值即耗盡。若要保留，應退為 `examples/tm002/` 純參考資產，不掛 description 不進 skill 清單。

- **serp → 刪（合併進 eap 降級後的主線）**：serp 是 eap 的 fork-and-rename，複製了展示性外殼卻沒複製實質（MN-07 + MN-01）。理由：唯一專案差異（MySQL vs MSSQL、模板清單）放進「serp 專案包」即可，獨立 skill 只是讓 Red Flags 要維護三份。

## 三、總評：整套防禦 apparatus 是否已過度工程？

**是——而且是以一種特定方式過度：防禦的「累積複雜度」本身，正在變成它聲稱要防的那個「複雜度天花板」（01-§11）。**

證據鏈與三條判讀：

1. **它最該防的風險，反被自己違反。** SKILL 體系反覆引用 §10 指令飽和、§11 複雜度天花板、§2 Context Rot 作為「拆檔、精簡載入」的理由——這部分主線做得很好（refs 抽離、類型 tag 分流、按需載入模板）。但特化三套各帶全量骨架（MN-01）、lite 塞 20 條硬編碼（MN-02）、eap 把度量基礎設施寫進正文（MN-03）、同一訓誡三段重複（MN-04）、§N 標註灌滿全篇（MN-05）——這些都是在**往 context 裡加東西**。一個「為了對抗指令飽和而存在的系統」，自己的指令總量在膨脹，這是過度工程最典型的自噬。

2. **複製是過度工程的稅，而非設計。** traceability 三份 MD5 相同、Red Flags 逐字相同（〇節鐵證）不是「重視一致性」，是「沒有單一權威來源」的維護債。03 Work#3 明說要「分發單一份實戰配置」。三份並存意味著每次研究更新（01 改一個數字）要改三處，且已可預見 silent drift（05-D8 自己標 HOLD 在擔心這件事，卻只擔心主線 refs，沒看到特化三套的複製早已是同一病的重症）。

3. **但要區分「主線」與「特化」——不能一竿子打翻。** 主線 5 SKILL + spec-workflow-refs/p3/ 其實是**反例中的正例**：refs 檔 27~118 行、漸進揭露落實、類型 tag 把 stop gate 從「每 task 都停」瘦成「只有該防的 tag 才停」（直接呼應 05-A1）。主線的問題只是尾巴（MN-10/11 的本體重述與強語氣密度）。**真正過度工程的是特化三套的存在方式**：把一個本該是「主線 + 三個薄專案包」的東西，做成了三個各自完整、各自複製骨架的巨型 skill。

**「下一個該拔的是什麼」——極簡方的優先序：**

| 序 | 動作 | 砍掉什麼 | 對應指控 | 預期收益 |
|:--:|---|---|---|---|
| 1 | 刪 `eap/test01/` | ~11000 行廢料 | MN-06 | 立即、零風險、純策展 |
| 2 | 把 eap/serp 骨架抽成「通用主線 + 專案包」，骨架留一份 | serp 整個獨立 skill + eap 重複骨架 | MN-01 / MN-07 | 消除三份複製的維護債 |
| 3 | lite 降為 `examples/` 參考資產，移出 skill 清單 | 一個鎖死單模組的「skill」 | MN-02 | 還原「skill＝可重用」定義 |
| 4 | eap Phase 日誌正文移成靜默 hook 或刪 | ~50 行 SKILL + 每 prompt 的 logger bash | MN-03 | 度量回到「成功靜默」 |
| 5 | 合併 Gotchas/Red Flags/硬性規則去重；§N 標註瘦身 | 三段語義重複 + 行內編號噪音 | MN-04 / MN-05 | 降指令飽和、省 context |

**一句話收束：** 這套體系的「思路」是對的（結構性防禦 > 指令性約束，主線已證明可行）；它的「實作」在特化層過度了——把一份好骨架複製成三份、把一次 Demo 凍成一個 skill、把度量工具當知識教。**該拔的不是防禦的思路，是防禦的冗餘副本。少即是多，下一刀先砍複製。**
