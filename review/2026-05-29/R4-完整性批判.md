### GAP-1 詳述（最重要的漏）— 沒人質疑「透鏡選對了嗎」

team-lead 的提示精準。我 Read `review/2026-03-23/2026-03-23-r1-審查報告.md:182`，原文鐵證：

> 「文件的中心組織架構——Layer 0（Next-Token Prediction）→ Layer 1（In-Context Learning）→ Layer 2（Extended Thinking）→ Layer 3（Agentic Loop）——**不是學術文獻中公認的框架**。沒有已發表的論文、教科書或會議教程使用此分層分類法。」（`r1:182`）
> 「核心公式 `Agent 產出品質 = f(Context 品質, 任務邊界, 回饋迴路強度)` 同樣是**作者自創**。」（`r1:184`）
> 「但文件密集的學術引用密度營造出一種印象：框架本身經過同儕審查——實際上它是編輯層面的綜合整理。」（`r1:186`）

**這是整場三方對抗的共同盲區。** 三方都把 `01_摘要.md` 的「11 風險 + 交叉矩陣」當**既定的、中立的判準尺**來用——辯護方拿它驗推導鏈、極簡方拿 02/03 驗過度工程、實戰方拿 log 驗 ROI。但**沒有任何一方退一步問：「11 風險」這個清單本身，是不是把 LLM 行為切成了一個方便防禦、卻不一定是最相關的透鏡？**

- 它的上位框架（Layer 0-3）r1 已明判「非學術公認、作者自創」。
- 11 風險全是**缺陷導向**（漂移/偏見/博弈/盲區…），這呼應 `01:644` 維護者「決定不加獨立正面框架」的刻意選擇（見挑戰二）。
- 三方唯一沾到邊的是 CS-09，但 CS-09 打的是「數字被誇大」（引用層），不是「框架選錯透鏡」（結構層）。裁決方還把 CS-09 後半駁掉了。

**為何 high：** 整個 G1 推導鏈驗證、整個 11 風險覆蓋表，都建立在「11 風險是對的分析單位」這個未經檢驗的前提上。如果透鏡本身偏（只照缺陷、分層自創），那麼「§7 附和是最重缺口」這類結論的**權重**就有系統性偏移——可能高估了「防缺陷」類規則的重要性，低估了「放手讓 agent 發揮」的價值（而後者正是極簡方一直在喊、卻沒有判準尺支撐的東西）。**這不是要推翻 11 風險，是要 G1 明確聲明「判準框架自身是作者綜合整理、非公認模型」，讓所有『兌現/缺口』判定都帶上這個限度。** 否則定稿會給人「我們拿一把校準過的尺量出了客觀結論」的錯覺，而那把尺 r1 自己說是手工做的。

### 挑戰一（high）｜CS-09 後半被**整個**駁掉，抹掉了「框架仍單向列險」這個仍成立的部分 — 位置：G1 §④ + §⑥ + G2 §③

**我的挑戰：裁決方把 CS-09 後半當成「只主張數字被誇大」來駁，但它還有一個維護者自己承認、且仍成立的內核被連帶抹掉了。**

我 Read `01:644`，鐵證：

> `- [x] 系統性悲觀偏誤：決定不加獨立正面框架，改以引言範圍聲明處理`

再 Read 那個「範圍聲明」`01:9-12`：

> 「本文件聚焦於 LLM 在 Agentic Coding 情境下的**已知行為風險**……LLM 在程式碼生成、重構、解釋等任務的正面效益已有廣泛共識，**不在本文件的範圍內**。」

**這兩段合起來說明一件裁決方沒處理的事：系統性悲觀偏誤在 v1.1 是「被聲明」而非「被消除」。** 維護者**主動決定不加正面框架**，只用一句範圍聲明把「只列風險」正當化為「刻意的範圍選擇」。也就是：

- 裁決方說「v1.1 已套用全部修正、CS-09 打到舊靶」——**對「歸屬錯誤/數字」這層成立**（r1/r2/r3 那些確實改了）。
- 但「**框架仍單向列險、偏誤僅被聲明未被中和**」這層，`01:644` 證明它**在 v1.1 仍然成立**。裁決方用「v1.1 已修」一刀切掉 CS-09 後半，把這個仍成立的部分一起埋了。

**這跟 GAP-1 是同一根：** 一個只列風險、不列正面框架的判準文件，拿來當「該不該加防禦」的尺，天然會偏向「多防一點」。CS-09 後半的真正價值不是「數字打折所以防禦打折」（這條裁決駁得對），而是「**判準尺本身是單向的，所以用它算出的『缺口』會系統性偏多**」——這條裁決沒駁、也駁不掉，卻被歸進「已駁回附錄」。

**該怎麼修：** G1 §④ 對 CS-09 的裁決，應從「後半整個駁回」改為「**駁回『數字打折→防禦打折』；但保留『判準框架單向列險（01:644 維護者自承不加正面框架）→ 所有缺口判定的權重需打折看待』**」。這條保留下來，正好給極簡方「別過度防禦」一個**有判準依據**的立足點（目前極簡方全靠 02/03，缺 01 這邊的支撐）。

### 挑戰二（high，最關鍵）｜裁決方對「輕量 review session」的紀律，寬於對所有舊防禦 — 位置：G2-02 + G3 優先序 9 vs G2 退場原則

**裁決方判定：** RD-16 是最重缺口，正解＝補「輕量獨立 review session」（G2-02 設限四條：限邏輯類 / context 防火牆 / 先試 1 專案 / 設退場條件）。

**我的挑戰：表面上 G2-02 設了「先試＋退場」，看起來有掛紀律。但對比裁決方對舊防禦的嚴苛，這帖新藥享受了隱性的優待——而且優待藏在「論證強度」而非「條文」裡。**

先確認共同紀律的存在：CS-01 + 實戰 R2 提案 5 + G2 §②結語確實要求「**所有 v2.x 無驗證攔截型新防禦設『試用 N 專案不達標即拔』**」。G2-02 第 3/4 條也照抄了「先試 1 專案 + 退場」。**所以條文層面，裁決方沒有明顯雙標。** 這點我必須誠實認給裁決方。

**但實質層面有三個不對稱，裁決方對舊防禦嚴、對自己開的新藥寬：**

1. **舉證門檻不對稱。** 對 SG2 對照表（G2-03）、commit-hook（G2-08）、Scope Statement，裁決方反覆強調「零 session 驗證、W-05~08 待驗、攔截率未知」，語氣是**懷疑性的**（DISCUSS + 退場）。但對輕量 review session，裁決方的論證是**辯護性的**——它花了 G1 §⑤一整段、§③交叉矩陣兩列、加「查證二補正」來論證「寫審分離有效」（引 eap Review Agent 抓 4 MUST-FIX）。**問題：eap Review Agent 的有效性，證明的是「eap 那種重型 general-purpose subagent review」有效，不是「裁決方新提的『開乾淨 session 餵 diff+規格』那種輕量 review」有效。** 後者**同樣零實戰驗證**——它跟 SG2 對照表一樣，是「反推自缺口的合理設計」，不是「跑過有效」。裁決方拿 eap（重型）的成功，給自己的輕量版**借了它沒資格借的信用**。

2. **「先試 1 專案」對 review session 是空頭支票。** SG2 對照表、commit-hook 至少已經寫進主線 SKILL，下個專案一跑就在驗證。但「輕量 review session」連**機制都還沒定**（G2-02 只說「加一個可選 SG + 提供 prompt 範本」）。對一個還沒成形的東西說「先試 1 專案」，紀律密度遠低於對已落地舊防禦的「試 N 專案不達標即拔」。

3. **它和它要補的缺口，論證上自我指涉。** RD-16 說「自檢落 §8 盲區（自我修正 18%）」。但「開乾淨 session 餵 diff」的 reviewer **也是 LLM**，它 review 時面對的雖是「外部來源的 code」（理論上落 88% 高修正區），可是——**這正是 §8 `01:366-369` 標 [待驗證] 的那兩個數字**（18.3%/88.1%）。裁決方一邊在挑戰一說「結構防禦不依賴數字大小」（淡化數字），一邊在 RD-16 又重度依賴「18% vs 88%」這個落差來論證 review session 的必要性（強調數字）。**同一份 [待驗證] 數字，淡化或強調取決於它幫哪一方——這是裁決方論證的內在不一致。**

**該怎麼修：** G2-02 + G3 優先序 9 應明確標註：「**輕量 review session 與 SG2 對照表/commit-hook 同屬零實戰驗證的新防禦，eap Review Agent 的有效性不可遷移為本機制的信用**；退場條件應與其他新防禦**同級**（試 N 專案、量攔截數 vs 手測發現數，不達標即拔），不得因『補最重缺口』而享受更鬆的舉證。」並在 G1 §⑤刪去/降權「eap Review 抓 4 MUST-FIX 反證輕量 review 有效」這個論證跳躍。

### 挑戰四（medium）｜RD-17 被「改用 grep」駁回，把兩個不衝突的東西當對立 — 位置：G2 §③ + G2-05 + G1 §②
