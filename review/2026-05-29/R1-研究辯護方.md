## 立場聲明（我相信什麼，以及我查到了什麼）

我的信念：**SKILL 每條規則都應該有研究撐腰，針對 LLM 行為缺陷的防禦是必要的、不是過度設計。**

逐條查驗後，我必須誠實報告：**這個 repo 在「防禦的方向」上大量兌現了研究，但在「研究→規則的推導鏈是否真的成立」上，主線 SKILL 出現了系統性斷裂。** 作為辯護方，我能為「防禦必要」辯護到底——11 項風險裡有 7 項在主線確實有對應防禦，方向正確。但我無法為 README「每條規則都有量化研究支撐、非經驗談」這個**宣稱**辯護：實證顯示主線五個 SKILL 的 `§` 研究引用是 **0 次**，規則依據幾乎全是 AR003/SO0062/AR002 實戰教訓——這恰恰是「經驗談」。

更尖銳的發現是一個**反向悖論**：唯一逐條標註研究 §（共 26+9 次）的，是被 README 標為「已被取代/暫停」的特化型 eap/serp；而被推崇為當前路線的主線，反而把研究標註全部拔光。**這個 repo 把「最有研究撐腰」的版本淘汰了，留下「最少研究撐腰」的版本當主線。** 作為辯護方，這讓我很難堪——因為這意味著 repo 的演化方向與它自己的研究信仰背道而馳。

下面用指控清單（charges）逐條呈現，並附兩張對照表。

### 第四組：風險缺口（G1 第 2 半 — 11 風險中沒防到的）

#### RD-13 §1 語義漂移：跨 session 錨定路徑防禦薄弱 `風險缺口` `嚴重度：高`

- **標的**：全主線；尤其 `spec-p3-backend/SKILL.md:189-191`（session_log 機制）、`progress-and-session-log.md`
- **指控發現**：§1 的核心對策是「每個任務從**源頭文件**載入（錨定路徑），不基於前次 AI 產出接著做」（`01_摘要.md:32`）。但主線的跨 session resume 機制（progress.md + session_log.md）恰恰是**路徑 B（以前次產出為輸入）**：新 session 讀的是「上次 AI 寫的 progress 備註、session_log 決策、已 commit 的 code」，而非每次回到規格統計源頭重新載入。session_log.md 累積「關鍵決策/教訓/下 session 注意」正是 §1 警告的「衍生文件取代源頭」「用 Agent 對代碼的理解取代原始需求」。雖然 api_contract/current_schema 是 SSOT（部分錨定），但 task 層的實作脈絡是漂移路徑。整套 SKILL 沒有「每 N task 強制回讀規格統計重新對齊」的機制。
