### eap-agentic-coding → **砍流程架構、抽救 2 hook + review prompt 設計、test01 歸檔**（分層處置）`DECIDED`

**結論：**
1. **砍**流程架構：Phase 0-3 + Test/Code/Review 三 Agent **編排** + subagent 寫 code + 整合測試（@QuarkusTest）路線 + SKILL.md 本體 + description frontmatter。
2. **抽救** 2 個 hook + 1 份 review prompt 設計：
   - `block-test-edit.sh`（SKILL.md:293）+ `silent-test-pass.sh`（SKILL.md:307）提取為主線可複用的 hook 範本。
   - **`phase-3-review-agent.md` 的 review prompt 設計（反附和指令：「主動找問題、不附和、引用具體規格條目/行號、發現的問題分 MUST/SHOULD/NOTE」）抽救為 G2-02 輕量 review session 的 prompt 範本**——**抽的是 prompt 設計（一份指令文字），不是三 Agent 編排**（編排砍掉）。
3. **歸檔** test01/（17780 行）：移到 repo 外審查歸檔區（`~/.soetek-ai-coding/` 或 review 歸檔）當「失敗案例存證」，不隨 plugin source 打包。

> **抽救清單三件與「砍架構」的界線（必補三）：** 抽救的是**可移植的零件**（2 hook 的確定性控制 + 1 份 review prompt 的反附和指令設計）；砍的是**綁定編排**（三 Agent 編排 + subagent 寫 code + 整合測試）。零件可移植到主線輕量使用，編排被 TM002 證偽 ROI 為負故砍。**注意：抽救 review prompt ＝抽「設計」，不抽「eap review 的 ROI 結論」**——輕量 review 用此 prompt 但其有效性仍須 G2-02 第 3 條試用自證（與 G1 §⑤、G2-02「不借 eap 信用」口徑一致）。

**理由（砍架構）：**
- **唯一實測即 ROI 災難**：TM002 112 分鐘、P2 Code Agent 獨佔 56 分鐘/67.4M token/實寫 code 11%（查證二）。整合測試基建 19m + Encoding 13m + API routing 6m＝66% 浪費全在這套架構的 P2 段。
- **架構被主線逐條淘汰**：subagent 寫 code 違反主線 session-791acadc 教訓；整合測試被主線拔除（CS-02/03）。
- **三 Agent 是 03 P4 角色扮演反模式 #5**（按職能命名的常駐角色分割，極簡 R2 核實 03 P4 line 67）。

**理由（抽救 hook — 這是 G3 最關鍵的「砍前先救」）：**
- 極簡 R2 + 實戰 R2 雙方都讓步承認：`block-test-edit.sh`（PreToolUse 結構性阻擋改測試檔，**不靠 agent 自律**）+ `silent-test-pass.sh`（吞通過測試輸出只留失敗，直擊 03 反模式 #8 冗長驗證輸出）是**全套標的裡唯一真正落實 03 P6（hooks 確定性）+ P7（成功靜默、失敗詳述）的範例**。
- 主線的 commit-hook（G1 #5）是這個思路的延伸，但主線**自己沒有「測試唯讀結構強制」這一塊**——只有 git diff 自檢（靠事後對照，非事前阻擋）。
- **裁決採極簡/實戰共同的「砍架構、留 hook」**：把 hook 抽成獨立可選 hook 範本（任何走 TDD 的流程都能裝），放到 `spec-workflow-refs/p3/hooks/`（與既有 commit-hook 並列），不連帶保留 eap 流程架構。

**承認張力：eap 的 review 階段是整套災難架構裡唯一 ROI 為正的部分（必補三，把「砍架構」講圓）：**
- 查證二的細項顯示：eap 的 Phase 3 Review Agent **只花 5m42s（5% 成本）就抓到 4 個 MUST-FIX**（6 Processor 未用 buildStandardResponse 等真實一致性問題），P4a/P4b 修完 37/37 通過。**這是 112 分鐘裡 ROI 最高的一段**——與 P2 整合測試那 56 分鐘（ROI 為負）形成鮮明對比。
- **所以「砍 eap 架構」砍的精確標的是 P2 整合測試 + subagent 寫 code + 三 Agent 編排（ROI 負的部分），不是 review 這個動作本身**。review 階段的 prompt 設計（反附和指令）反而要抽救（結論第 2 點）。
- **這不構成「保留 eap」的理由**：① review 的價值在「prompt 設計」這個可移植零件，已抽救；② eap 把 review 綁在三 Agent 編排 + .agentic 中間產物 + 整合測試裡賣，那個綁定 ROI 為負；③ 抽出 prompt 設計後，三 Agent 編排對 review 不是必要的（G2-02 證明開淨 session 單次 review 即可承載同樣的反附和指令）。**故：承認 review 階段有效、抽救其設計、砍掉承載它的重型編排——三者不矛盾。**

**eap 內部分歧的裁決（極簡「降級保留整套流程+hook」vs 實戰「全砍只留 hook 零件」）：**
- 極簡 R2：eap「降級為主線+專案包」，保留 P0-P3 流程架構。
- 實戰 R2：eap 流程架構（Agent 分離）正是 TM002 災難根源，**連流程一起砍**，只把 hook 概念抽成獨立零件。
- **裁決採實戰 R2（更精準）：流程架構（三 Agent + subagent 寫 code + 整合測試）有 TM002 實測證偽 ROI 為負，不保留為「可選路線」（留著＝留一套被證偽的架構當誘惑）；只抽可移植的 hook 零件。** 極簡方擔心的「丟掉唯一結構防禦範例」已由「抽救 2 hook」解決——hook 救出來了，流程不必留。

**遷移/封存建議（抽救順序＝砍 eap 的前置）：**
- 先 `cp` 兩 hook 到 `spec-workflow-refs/p3/hooks/`（連同 settings 註冊片段 + 一頁說明「測試唯讀結構強制 + 成功靜默」設計脈絡）。
- **再 `cp` `phase-3-review-agent.md` 的反附和 review prompt 段到 `spec-workflow-refs/p3/`（如 `lightweight-review-prompt.md`），去掉三 Agent 編排與 .agentic 中間產物引用，只留「開淨 session 餵 diff+規格條目、主動找問題、分 MUST/SHOULD/NOTE」的單次 review 指令——供 G2-02 輕量 review session 引用。**
- `git mv skills/eap-agentic-coding/test01` 到 review 歸檔區（或 `git rm` 後在 review/2026-05-29/ 留一份指標說明「失敗案例存證位置」）。
- `git rm -r skills/eap-agentic-coding`（**hook + review prompt + test01 三者都抽救/歸檔完後**）。

## ② 過度工程分區裁決（過度 / 適中 / 不足 分層）

> direction.md 要求「分區裁決，勿一刀切」。裁決方用 02/03 原則逐層判，並處理那個反諷（對抗 §10 指令飽和而生的系統，自己指令是否膨脹）。

### 分區一：主線 5 SKILL + spec-workflow-refs/p3/ → **適中偏「反例中的正例」**（局部不足）

**裁決：適中。是「反例中的正例」，過度工程的指控**不**成立於此層。**

- **正例證據（三方在 R2 罕見共識）**：refs 檔 27-118 行、漸進揭露落實（SKILL.md 指路、refs 放細節）、類型 tag 分流把 stop gate 從「每 task 都停」瘦成「只有該防的 tag 才停」（直接呼應 05 A1 + 兌現 §10）。共用規約抽離（D25）是 DRY 重構正例（極簡 R2 駁 CS-01 把它列待砍）。
- **局部過度（尾巴肥肉，非主病）**：MN-10 P3 三 SKILL「核心原則」段 refs 已抽離但本體又重述一遍（同規約讀兩次）；MN-11 強語氣密度（已部分自解）。→ 歸 G2-07 瘦身，非結構性過度。
- **局部不足（更該關注）**：§7 寫審分離缺口（G1 RD-16）、§3 模型選擇缺席（G1 RD-15）——**主線的真問題是「過度精簡」（拔掉有研究撐的硬防禦），不是過度工程**。這與極簡方總評一致：「主線思路對、特化層實作過度」。

### 分區二：特化三套的「存在方式」 → **過度（apparatus 冗餘複製）**

**裁決：過度。但過度發生在「存在方式」（三份各自完整 + 複製骨架），不在「防禦思路」。**

- **最強單一證據**：traceability 三份 MD5 相同、Red Flags 逐行零差異（MN-01 鐵證）——不是「重視一致性」，是「沒有單一權威來源」的維護債（03 Work#3「分發單一份實戰配置」）。每次研究更新要改三處，必然 silent drift。
- **lite 把一次 Demo 凍成 skill**（MN-02）、**eap 把度量基礎設施寫進正文**（MN-03 Phase 日誌/statusline）、**eap test01 17780 行廢料隨 plugin 打包**（MN-06）、**同一訓誡三段重複**（MN-04 Gotchas/Red Flags/硬性規則）——都是「往 context/repo 加冗餘」。
- **§N 標註**：對執行 agent 是噪音（MN-05），服務對外敘事非 agent 決策——但這是「該瘦」非「該補」（G1 §④裁決：改宣稱 + 不補正文 §N）。

### 分區三：v2.x 四大新防禦 → **「未驗證」而非「過度」**（區分關鍵）

**裁決：多數是「未驗證」（該設退場條件），不是「過度工程」——但需逐項分。**

- **SG2 對照表 / commit-hook / Scope Statement**：攔截型新防禦，方向有研究撐（G1 兌現/牽強），但零實戰驗證（W-05~W-07 DISCUSS）。**屬「未驗證」**——設「試用 N 專案不達標即拔」退場條件，從無限期 DISCUSS 變有截止實驗（CS-01 + 實戰 R2 提案 5）。
- **共用規約抽離（D25）**：**正例，不列待砍**（DRY 重構，非攔截型防禦，極簡 R2 駁 CS-01 此項，G2 已收）。
- **判別準則**：攔截型防禦（會中斷流程/加 gate）需「跑過驗證」→ 未驗證的設退場；重構型（去重/抽離）看「引用解析 + 語義一致」→ 不需跑專案驗證。

### 反諷的精準定位：指令膨脹發生在「特化層」，主線反而瘦身

direction.md 點的反諷——「一個為對抗 §10 指令飽和而生的系統，自己指令總量是否在膨脹」：

- **裁決：反諷成立於特化層，不成立於主線。**
  - **主線**：refs 抽離 + tag 分流是**真瘦身**（兌現 §10），指令按 Phase/tag 分段載入，不是膨脹。RD-06 三方公認主線正例。

- **對照主線 refs 抽離/tag 分流的反證**：反諷的解藥 repo 自己已有（主線就是怎麼做對的示範）；過度發生在「特化層沒套用主線已驗證的瘦身手法」。**精準定位：膨脹在特化層的複製副本與展示性標註，不在主線的防禦機制本身。**
