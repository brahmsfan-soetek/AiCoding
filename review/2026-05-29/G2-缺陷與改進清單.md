**背景：** `spec-p3-backend/SKILL.md` 審查環節只剩 line 79（git diff 自檢測試檔）、93-94（自檢 response shape）、166、180——全是同 session 自檢，無獨立 review（裁決方查證三證實）。§7+§8 是研究指出的結構性弱點，核心方向＝「自我修正率 < 外部修正率」（§8）+「附和傾向使 AI 不反對≠正確」（§7）。註：§8 具體數字（自我 18% vs 外部 88% / 64.5%）`01:366-369` 標 [待驗證]，故本條依賴**方向**而非未驗數字（與 G1 §⑤口徑一致）。

### G2-05　「讀規範、不掃 code」絕對化禁令，與研究 + 實戰雙向衝突 `DISCUSS`

**背景：** `subagent-boundary.md:22-33`、`spec-p3-backend/SKILL.md:131,223`、`spec-p3-frontend/SKILL.md:222`、README:139 把「不從掃 sibling code 歸納 pattern」絕對化為禁令。

**審視：** **三方同向最強的一條**（G1 推導鏈 #8 斷鏈）：研究 §2「工具式檢索優於預載入、鼓勵 glob/grep 按需讀 code」（RD-11）+ 實戰 SO0062「先讀 backend processor 落地 mapping，不能只看 SQL」+ AR003「參考 Ap008VoidProcessor.java 既有模式」（CS-07）+ 維護者自承 `05` A2 DISCUSS。規則把研究**支持**、實戰**必需**的行為禁掉了。極簡方視角：強行剝奪 agent「去看一眼既有 code」的判斷＝02 P4 railroading。

**審視（與 RD-17 canonical example 的關係，R4 polish 校準口徑）：** RD-17（沒範例可仿）的缺口真，但**正解不是「純用 grep」單選、也不是「維護一整套 canonical-example 清單 apparatus」單選——而是「grep 開門 + 輕量指路」兩者組合，非二選一**：
- **grep 開門**＝RD-11 的藥：鬆綁禁令，讓 agent 可 grep/glob 查證既有 code 落地點（這一步已解掉「完全無 code 可參考」的真空）。
- **輕量指路**＝保留 canonical example 的**有用內核、去掉其 apparatus**：不建「一整套指定清單 + 三層優先級 + 專人維護」（05 A2，極簡 R2 指其多餘＋silent drift），而是在**需要時**（如 G2-02 review 或 PG 起 P3 時）口頭/即時指一個「已知結構良好的既有檔」當參考（「仿 Ap008VoidProcessor.java 的 ApplicationScoped + NativeSqlService 模式」）。這是一句指路，不是一份要維護的清單。
- **為何非二選一**：純 grep 仍有「agent 不知道哪個既有檔是好範本、可能 grep 到 anti-pattern」的殘留風險；輕量指路補上「指哪個」，但不付出整套清單的維護債。極簡 R2 的「用 grep」+ 實戰 R2「canonical example 方向對但別建 apparatus」在此合為一解。

**決定（DISCUSS → 偏 DECIDED）：** 把絕對禁令鬆綁為「**容許 grep/glob 工具式查證既有 code 落地點，禁止的是『把歷史 anti-pattern 當權威規範盲目複製』**」；canonical example 採「**grep 開門 + 輕量指路（即時指一個好範本，不建清單 apparatus）**」。

**下一步：**
- 改 `subagent-boundary.md:22-33`：把「不掃 code 歸納 pattern」改為「**規範權威＝CLAUDE.md 索引文件；但容許用 grep/glob 查證既有 code 的落地細節（API response shape、Processor mapping 落地點、既有元件 prop、reference implementation）作交叉驗證**；禁止的是把無 review 的歷史 code 當權威規範整段複製」。
