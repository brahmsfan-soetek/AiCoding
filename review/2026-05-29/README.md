## 二、對抗審查怎麼跑的（方法與獨立性保證）

五個獨立角色，回合制對抗，刻意防止「附和自己」（§7 附和偏見 + §8 自我修正盲區是這套設計要防的核心）：

```
R1 獨立立場   三方並行各掃全標的，互不可見 ← 防附和偏見汙染獨立判斷
R2 交叉反駁   每方讀另兩方 R1，逐條 refute/agree/qualify ← 對抗核心交鋒
R3 裁決綜合   中立裁決方讀全部交鋒，產 G1/G2/G3，被駁倒的列「已駁回」附錄
R4 完整性批判 獨立 critic 揪漏 + 挑戰過寬裁決 → verdict（complete / needs_more）
```

**獨立性如何落實：**
- **R1 互不可見**：三方並行、各自只看自己的 prompt 與標的，無法互相揣測附和。
- **R3 裁決中立**：裁決方不是任一陣營，且被要求「爭議事實親自回源查證，不採信任一方」——它確實這麼做了，甚至**糾正了研究辯護方 R2 的一個事實錯誤**（見下）。
- **R4 critic 獨立**：critic 是與裁決方分開的第五個 agent，職責是批判裁決方、不附和。它判 `needs_more` 並抓到裁決方的真實偏頗（見「五、R4 校正」）。

**兩處對框架的誠實適配（與 `direction.md` 原文的差異）：**
1. **R2「成員互相 SendMessage 逐條反駁」→ 改為「把另兩方 R1 餵入該方 R2、各自讀檔逐條表態」**。效果等價（逐條 refute/agree/qualify 不變）且更具確定性；三方仍互不知道對方的 R2（只見對方 R1），維持反駁而非附和。
2. **一度同時存在兩條執行線**：team-lead 最初以 Workflow 啟動了一條自動串接的背景流程，但環境同時以具名 teammate（research-defender / minimalist / combat-skeptic）跑起 SendMessage 協調式 team。確認 team 才是會寫實體報告檔、合 `direction.md` 本意的執行線後，**背景重複 workflow 已停掉**（它只回傳結構化資料、不寫 `review/` 檔，停掉不影響 team、純省 token）。本審查的所有報告檔均出自 team。

## 四、三大產出總結

### G1 — 推導鏈與風險缺口
- **推導鏈**：10 條代表規則 → **兌現 4**（SG2 方向、MCP 最小權限、tag 分流、git diff 防篡改）／**牽強 4**／**斷鏈 2**——主線最具體的兩條規則（「禁掃 code」與研究 §2 相反、resume 走衍生文件踩 §1 漂移路徑）恰恰與研究結論相反。
- **11 風險覆蓋**：✅充分 4／🟡部分 6／**❌缺口 2（§3 模型選擇、§7 附和）**。不對稱結構：**§9 規格博弈防得最紮實（5 條規則），§7 附和防得最空（拆了寫審分離）**。
- **README「非經驗談／每條規則有量化研究支撐」宣稱：不成立**（主線行內 §引用＝0，依據全是 AR003/SO0062/AR002 實戰事件）——這是「經驗談」。**但這不是缺點，是宣稱用錯字**：feedback 驅動的演化本身健康，問題在 README 自稱「非經驗談」。
- **誠實限度（R4 補入）**：判準尺（11 風險 + Layer 0-3 框架）是作者編輯層綜合整理、**非學術公認模型**，且**對缺陷單向聚焦 by design**（只列風險、不列做對的事）→ **本審查的「風險缺口」判定可能系統性偏多**，缺口＝值得查的方向，非必須補的證據。

### G2 — 缺陷與改進清單（13 條，對齊 `05` E 表）
- **可立即動手（DECIDED，5 項）**：①README 改文案 ②鬆綁「禁掃 code」→「grep 開門 + 輕量指路」 ③補一行模型選擇（§3 是研究列「最重要的決策」、零成本） ④前端 SG3「靜默跳過」改「顯式警告」 ⑤補 /clear 軟提示。
- **核心實驗（DISCUSS）**：補**輕量獨立 review session**（限 `[validator]`/`[processor]` 邏輯類、context 防火牆、明標**零驗證實驗**、與 SG2/hook **同級退場紀律**、不借 eap 重型 review 的信用）補 §7/§8 最重缺口。
- **退場條件原則**：所有 v2.x 未驗證的**攔截型**新防禦（SG2 對照表 / commit-hook / Scope Statement / 輕量 review）一律設「試用 N 專案不達標即拔」——把 `lessons_learned.md` W-05~W-08 從無限期 DISCUSS 變成有截止的實驗。（共用規約抽離 D25 是 DRY 重構、非攔截型，屬正例，不在此列。）

### G3 — 路線決策
| 特化 | 結論 | 要點 |

- **過度工程分區裁決（不一刀切）**：
  - **主線 5 SKILL + `spec-workflow-refs/p3/` ＝適中（反例中的正例）**——refs 漸進揭露、tag 分流瘦身 stop gate。主線的真問題是**過度精簡**（拔掉有研究撐的寫審分離/模型選擇），不是過度工程。

  - **反諷定位**：「對抗 §10 指令飽和的系統，自己指令在膨脹」這個自噬**成立於特化層、不成立於主線**；主線 refs 抽離/tag 分流正是解藥示範。

## 五、R4 校正紀錄（這套對抗設計的價值體現）

R4 critic 親核裁決方點名的全部事實 **100% 屬實、無造假**，但抓到一個事實之外的真問題：

> **「砍冗餘的手很穩，補缺口的手偏鬆」**——裁決方對實戰方質疑的 SG2/hook 用懷疑語氣，對自己新提的「輕量 review session」卻用辯護語氣。

verdict＝`needs_more`（非推翻）。三條必補 + 一處 polish 已全數套回 G1/G2/G3：

| # | 校正 | 處置 |
|---|---|---|
| 必補一 | G1 缺「判準框架限度」聲明 | G1 開頭加聲明：框架非學術公認、單向列險 → 缺口判定可能系統性偏多 |
| 必補二 | CS-09 後半被過度駁回 + 信用遷移 + §8 數字雙標 | CS-09 拆三層（數字層駁/判準尺單向層收/宣稱層收）；**根除信用遷移**（eap 重型 review 有效 ≠ 輕量 review 有效，兩者皆零驗證）；§8 改引「方向」（自我修正<外部修正）而非未驗證的具體數字 |
| 必補三 | G2「借」eap review prompt vs G3「砍」eap 打架 | G3 抽救清單加入 `phase-3-review-agent.md` review prompt（與 2 hook 並列），與 G2-02 對齊；講圓「eap review 階段是災難架構裡唯一 ROI 正、抽其設計、砍其編排」 |
| polish | RD-17「純用 grep」過窄 | 改「grep 開門 + 輕量指路」（即時指一個好範本，不建 canonical-example 清單 apparatus） |

**意義**：若沒有 R4，裁決會帶著「對自己開的新藥寬鬆」的偏頗定案。這正是五角色獨立對抗（而非單一 reviewer）要捕捉的東西。

## 七、一句話總結

**這套 Skills 的「思路」是對的（結構性防禦 > 指令性約束，主線已證可行）；它的問題不是「過度工程」而是兩件分開的事——主線在 §7/§8 寫審分離上「過度精簡」（該補一個零驗證、設退場的輕量 review），特化三套的「存在方式」才是真正的冗餘過度（該砍 serp、砍 eap 架構但救出 hook/prompt、降 lite）。README「非經驗談」是用錯字的過度宣稱（feedback 驅動本身健康，改文案即可）。而本審查自己也有限度：判準尺單向列險，故「缺口」是值得查的方向、不是必須補的證據——任何補法都要再過「實戰證據 + 退場條件」這關。**
