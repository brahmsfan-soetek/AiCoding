### 第四組：風險缺口（RD-13~19）——交鋒核心

> 一句話收束 RD-16:**「主線缺一個審查」我同意;「該用 eap 三 Agent 補回」我駁。正解是一個 context-防火牆式的輕量 review session,只跑邏輯類 task。這既補了辯護方點出的真缺口,又沒掉進它示範的過度工程。**

**RD-13（§1 語義漂移:resume 走 progress/session_log 漂移路徑）→ qualify**

**RD-15（§3 主線完全移除模型選擇規則）→ agree（這條成本極低,該補）**
team-lead 要我分清「一句模型選擇註記」vs「canonical example 機制」。我明確表態:
- **RD-15(模型選擇)→ agree,該補。** 理由:一句「P3 邏輯類 task / 長 context 階段建議用最強模型」的成本**接近零**(一行字、不增流程、不增 stop gate),而 §3 是研究明列「最重要的決策」。這不違反「少即是多」——少即是多反對的是「無價值的堆疊」,不是「一行高價值註記」。03 P1「保持簡潔且普適」恰恰支持這種普適的一行原則。**這是辯護方諸多『往主線加東西』的訴求裡,唯一我全力附議的。**

**RD-17（§5 canonical example 機制在主線缺失）→ refute（與 RD-15 對照,這條成本不低,不該以現在的形式補）**
- **RD-17(canonical example)→ refute(或至少強 qualify)。** 理由:辯護方要的是一整套「指定 canonical example 檔案清單 + 三層優先級(規範/範例/禁推斷)」機制(05 A2)。這成本**不低**:要有人 review 並維護一份範例清單、要解決「現有 code 都不夠好怎麼辦」(05 A2 自己列的未解問題)、且範例清單會 silent drift(範例檔改了清單沒更新)。**這正是我方 MN-01 警告的同類病:又一套要維護的 apparatus。** 更關鍵:RD-17 與 RD-11 自相矛盾——RD-11 才說「研究鼓勵 grep/glob 讀 code」,RD-17 又說「不能讀 sibling code 所以要 canonical example」。**既然 RD-11 已開放 grep 讀 code,canonical example 機制就多餘了**:agent 用 grep 找一個結構好的既有檔當參考即可,不需要一份人工維護的指定清單。**用 RD-11 的藥(開放 grep),解 RD-17 的病(沒範例可仿),不必新增機制。**

> RD-15 vs RD-17 對照結論:**模型選擇一行註記(零成本、高價值)→ 補;canonical example 機制(高維護成本、與 RD-11 衝突)→ 不補,改用 grep。** 這就是「少即是多」的實際操作——不是一律不加,是只加成本/價值比極高的。

**RD-18（§6 待確認暫不實作通則 P2/P3 未明文）→ qualify**
低度缺口,辯護方自己標低。我 qualify:SG2「偏離 STOP」+ Scope Statement 已間接涵蓋,再補一條明文通則邊際價值低,且增指令數(§10)。傾向不補,或併入既有 STOP 規則一句話帶過,不獨立成條。

**RD-19（§2 無主動 /clear / session 長度上限,主線鼓勵「一氣呵成」與研究相左）→ agree（低成本,該補一句軟提示）**

## 三、三組「同病不同藥」的釐清（本 R2 的核心貢獻）

交鋒下來,最值得 R3 裁決方注意的是:三方常常**指出同一個病,卻開相反的藥**。我把最關鍵的三組講透:

### 釐清一:§N 標註 / 「非經驗談」宣稱 —— 辯護方要補真研究,我要砍標註劇場

- **同病:** RD-01/02/03 + CS-09 + 我方 MN-05 都認定「主線規則依據是經驗談、§N 標註/『非經驗談』宣稱有問題」。
- **不同藥:**
  - 辯護方(RD-01/03):主線該**補回研究標註**,讓規則「真的」追溯到研究,兌現宣稱。
  - 我方(MN-05):應**砍掉 §N 標註劇場**,把 README 宣稱改成誠實的「實戰驅動 + 研究佐證」(RD-02 也同意這個改法)。
- **我的裁決建議:** 採 RD-02 的改宣稱(誠實)+ 我方的砍標註(精簡),**不採 RD-01/03 的補標註**。理由:CS-09 已證研究地基本身有悲觀偏誤,補更多標註是往一個有偏誤的地基上加裝飾;而標註的受眾是審查者不是執行 agent(對 agent 是 context 噪音)。**病根是「宣稱與事實不符」,治法是改宣稱,不是補一堆對 agent 無用的編號去硬撐宣稱。**

### 釐清二:風險缺口的解法 —— 辯護方要加機制,我要用已有的 / 只加零成本的

- **同病:** RD-13~19 指出 §1/§2/§3/§5/§7+§8/§11 多個缺口,部分我 agree 確實存在。
- **不同藥:**
  - 辯護方:逐缺口**新增機制**(回讀源頭機制、task 粒度硬規則、canonical example 機制、獨立 Review Agent…)。
  - 我方:**分類處理**——零成本高價值的補(RD-15 模型一行、RD-19 /clear 一句);已有 SSOT 能解的用已有的(RD-13 用 api_contract/current_schema resume);與 RD-11 衝突的不補(RD-17 canonical example,改用 grep);會增 stop gate 的不補(RD-14 硬規則→改提示)。
- **我的裁決建議:** 在 CS-10/A1 已把 stop gate 標為「HOLD 待減」的前提下,**缺口的解法必須優先用「已存在的 SSOT + 零成本註記」,只有 RD-16 邏輯審查缺口值得新增一個輕量機制(context-防火牆 review session)**。其餘加機制的訴求,方向與「stop gate 待減」相反,應駁。

### 釐清三:eap 存廢 —— 實戰方要全砍,我要砍架構留 hook

- **同病:** CS-02/03 + 我方 MN-01/06 都認定 eap 過度(架構災難、test01 廢料、複製骨架)。
- **不同藥:**
  - 實戰方:eap **整個砍**或降為失敗案例存證。
  - 我方:**砍架構(三 Agent + subagent 寫 code + 整合測試)、留 hook(block-test-edit + silent-test-pass 提取為主線範本)**。
- **我的裁決建議:** 採我方。理由:這兩個 hook 是全套標的裡唯一落實 03 P6/P7「確定性控制 + 成功靜默」的範例,主線的 commit-hook 是其延伸但缺「測試唯讀結構強制」。整個砍 eap 會連這個唯一範例一起丟,違反 03「留下有用的」。**砍 eap 是對的,但要先把 hook 救出來。**
