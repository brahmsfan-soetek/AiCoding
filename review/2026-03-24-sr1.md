## 二、問題與建議

### 問題 1（嚴重）：「測試唯讀」靠指令而非結構，與自身原則矛盾

SKILL.md 核心原則寫著「**結構性防禦優於指令性約束**」，§9 數據也顯示「反作弊指令幾乎無效（14/20 仍博弈）」。但 Phase 2 的「Code Agent 不可修改測試檔案」完全靠 prompt 指令執行，沒有結構性防禦。

**Thariq 文章明確提供了解法**：On-Demand Hooks。`/freeze` hook 可以在 skill 啟用期間阻擋對指定目錄的 Edit/Write。

**建議**：在 SKILL.md 或 Phase 2 prompt 中加入 hook 配置，Phase 2 啟動時自動註冊一個 `PreToolUse` hook，阻擋對 `**/test/**` 和 `**/*Test*` 檔案的 Edit/Write。這才是真正的結構性防禦。

```jsonc
// 範例：Phase 2 on-demand hook
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Edit|Write",
      "command": "node scripts/block-test-edit.js \"$TOOL_INPUT\""
    }]
  }
}
```

---

### 問題 2（嚴重）：前端沒有自動化測試，Phase 1 test-first 實質上只覆蓋後端

`tech-stack.md` 明確寫著前端測試配置是 `"test": "echo \"No test specified\" && exit 0"`。但 skill 有三個前端模板（frontend-page、frontend-service、frontend-store），Phase 1-2 的 test-first 流程對前端形同虛設。

**影響**：前端開發完全失去 §8（自我修正盲區）和 §9（規格博弈）的防禦。Phase 2 Code Agent 寫前端代碼時沒有測試約束，等於回到「先寫代碼」的模式。

**建議**：
- 短期：在 SKILL.md 的 "When to Use" 中明確標示前端部分的限制，或在 Phase 1 prompt 中加入前端替代驗證策略（如 TypeScript 型別檢查、ESLint 規則）
- 長期：配置前端測試框架（Vitest），讓 test-first 真正覆蓋全端

---

### 問題 3（中等）：Agent 分離的實作機制未定義

SKILL.md 定義了 Test Agent ≠ Code Agent ≠ Review Agent 的架構，但**沒有說明在 Claude Code 中如何實現**。是用不同的 `Agent` tool 呼叫？不同的 conversation session？Git worktree 隔離？

HumanLayer 文章的核心發現之一：sub-agent 的價值是**context 防火牆**，而非角色扮演。如果三個「agent」跑在同一個 conversation 中，context 會互相污染，Agent 分離的效果大打折扣。

**建議**：在 SKILL.md 的「Agent 分離模型」區塊加入具體實作指引：

```markdown
### 實作方式
- P1 Test Agent → 獨立 Agent tool 呼叫（或新 session）
- P2 Code Agent → 獨立 Agent tool 呼叫 + worktree 隔離
- P3 Review Agent → 獨立 Agent tool 呼叫
- 每個 Agent 只看到自己 Phase 的 prompt + 必要輸入，不看到其他 Phase 的 context
```

---

### 問題 4（中等）：描述欄位（description）作為觸發器不夠精準

目前的 description：
> `serp 專案的規格驅動開發流程。DOCX + HTML 規格輸入，test-first + agent 分離，Phase 0-3 執行。`

Thariq 強調 **description 是給模型看的觸發器，不是給人看的摘要**。目前的 description 缺少常見觸發場景的關鍵詞。

**建議**：
```yaml
description: >
  serp 專案的規格驅動開發流程。DOCX + HTML 規格輸入，
  test-first + agent 分離，Phase 0-3 執行。
  TRIGGER when: 使用者提到 serp 功能開發、規格書實作、DOCX/HTML 規格、
  新增模組、新增功能頁面、新增 API。
  DO NOT TRIGGER when: bug 修正、單檔小改、重構、無規格書的探索。
```

### 問題 5（中等）：Phase prompt 中 §3 位置偏見未落實

01摘要 §3 的對策寫「關鍵指令放前段」，但各 Phase prompt 的結構是：步驟 → 約束 → 產出 → Gotchas。**最關鍵的約束被放在中段**。

例如 Phase 2 的「不可修改測試檔案」是最重要的規則，但它在 prompt 的第 50+ 行，被埋在約束表格中。

**建議**：在每個 Phase prompt 的最頂部（步驟之前）加一個「硬性約束」區塊，只放 2-3 條最關鍵的不可違反規則：

```markdown
## ⛔ 硬性約束（最高優先級）
1. 不可修改測試檔案 (§9)
2. 不可從前一 Task 繼承 context (§1+§2)
3. 不確定時記錄而非假設 (§6)

---
## 步驟
...
```

### 問題 6（中等）：Phase 0 只查跨文件矛盾，漏掉文件內部矛盾

§4 寫「矛盾消除必須在 AI 介入之前完成」，但 Phase 0 只比對 DOCX ↔ HTML 矛盾。單一文件內部的自相矛盾（如同一份 DOCX 前段說「軟刪除」後段說「可永久刪除」）不在檢查範圍內。

§4 數據顯示「自相矛盾句子比例 17.7%」，這不只發生在跨文件場景。

**建議**：在 Phase 0 Step 1 加入「文件內部一致性檢查」步驟，或至少在 Gotchas 中提醒。

---

### 問題 7（輕微）：Wait 機制定義模糊

P2 和 P3 都提到「Wait」但沒有具體的操作定義。01摘要說「Wait 提示可降低盲區率 89.3%」，但 skill 中的 Wait 只寫「暫停，重新從統一規格理解」。

**建議**：定義 Wait 的具體動作：
```markdown
### Wait Protocol
1. 停止當前修正動作
2. 重新讀取統一規格中該 Task 的段落（從源頭，不從記憶）
3. 重新讀取測試的斷言內容
4. 比對步驟 2 和 3，確認自己對業務規則的理解是否正確
5. 才開始修正
```

---

### 問題 8（輕微）：P2 缺少反附和機制

P1 有明確的「反附和檢查」（列出拆解方案的 2-3 個潛在問題），但 P2 Code Agent 沒有類似機制。Code Agent 在面對「測試失敗可能是測試問題」的判斷時，最容易觸發附和偏見（附和自己的代碼）。

**建議**：在 P2 的失敗處理中加入反附和步驟：
```
測試失敗
  ├─ 反附和檢查：列出「這是代碼問題」的 2 個理由 + 「這是測試問題」的 2 個理由
  ├─ 代碼問題的理由更充分 → 修正代碼
  └─ 測試問題的理由更充分 → 記到 review_notes.md（仍不修改測試）
```

---

### 問題 9（輕微）：WHY 解釋消耗 context，可考慮外部化

Thariq P1 原則：「Don't State the Obvious — 專注在把 Claude 推離正常思考方式的資訊」。

目前每個 Phase prompt 的約束表格都帶有「原因」欄位（如「§9：回饋可見性 30.4% vs 0.7% 博弈率」）。這些解釋對**人類維護者**有價值，但對**模型執行**是噪音 — Claude 不需要被說服，它需要被指示。

**建議**：將原因欄位移到 traceability-appendix.md，Phase prompt 中只保留規則本身。既減少 context 消耗（§2 + §10），又保持維護者的可追溯性。

---

### 問題 10（輕微）：缺少 Build/Typecheck 驗證步驟

HumanLayer 強調「back-pressure（驗證反饋）是最高槓桿的投資」，並建議在 agent 停止時跑 formatter + typecheck。

P2 只提到「執行測試」，但沒有提到 `./mvnw compile` 或 TypeScript 型別檢查。編譯失敗比測試失敗更基礎，應該先於測試執行。

**建議**：在 P2 Step 3 中加入編譯/型別檢查前置步驟：
```
Step 3：驗證
  1. 編譯（./mvnw compile 或 tsc --noEmit）— 必須通過才跑測試
  2. 執行該 Task 的測試
```
