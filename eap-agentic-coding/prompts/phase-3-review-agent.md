# Phase 3｜交叉驗證

**執行者**：Review Agent（與 Phase 2 的 Code Agent 不同）

**目的**：獨立驗證代碼是否正確實現統一規格，識別 Phase 1-2 可能遺漏的問題。

## 輸入

- Phase 0 產出的統一規格
- Phase 1 產出的 `test_spec_map.md`
- Phase 1 產出的測試檔案
- Phase 2 產出的代碼
- Phase 2 產出的 `review_notes.md`

## 步驟

### Step 1：規格覆蓋率檢查

逐條比對統一規格的功能點，確認每一條都有對應的代碼實作和測試案例。

產出格式：

```markdown
## 規格覆蓋率

| 規格條目 | 對應代碼 | 對應測試 | 狀態 |
|----------|---------|---------|------|
| §2.1 建立單據 | src/api/doc.ts:L30 | tests/doc.test.ts:L15 | ✅ 覆蓋 |
| §2.3 刪除權限 | — | — | ❌ 未實作 |
```

### Step 2：Review Notes 處理

針對 Code Agent 在 `review_notes.md` 中提出的疑問逐一裁決：
- 測試問題 → 判斷是測試問題還是代碼問題，提出修正建議
- 規格模糊處 → 標記需要使用者確認
- 實作疑問 → 提供獨立判斷

### Step 3：代碼與規格一致性審查

- 代碼中的業務邏輯是否與統一規格一致？
- 是否有代碼實作了規格中**不存在**的功能？(§6 自信填補)
- 是否有硬編碼的值應該是可配置的？
- 代碼是否遵循 `templates/` 模板定義的結構？偏離處是否合理？

### Step 4：測試品質審查

- 測試是否真的在驗證業務規則，還是只在驗證代碼結構？
- 是否有測試的斷言過於寬鬆（永遠通過）？
- 是否有遺漏的邊界條件或異常路徑？

## 約束

| 約束 | 原因 |
|------|------|
| **必須主動找問題，不可只確認「看起來沒問題」** | §7：95% 偏好附和。明確要求找問題才能抵消 |
| **審查意見必須引用具體的規格條目或代碼行號** | §6：模糊的「整體看起來 OK」可能是幻覺 |
| **發現的每個問題都要分類嚴重等級** | 幫助使用者決定哪些必須修、哪些可後續處理 |
| **每個審查步驟完成後 Wait，重新比對規格再繼續** | §8：Wait 提示降低盲區率 89.3%。防止審查慣性（越看越覺得沒問題） |

嚴重等級定義：
- **MUST-FIX**：業務邏輯錯誤、規格未實作、安全漏洞 → 阻擋交付
- **SHOULD-FIX**：程式碼品質、一致性偏離、效能疑慮 → 建議修正
- **NOTE**：觀察、改進建議、風格偏好 → 僅供參考

## 產出

`review_report.md` — 審查報告，含：
- 規格覆蓋率表
- 問題清單（含嚴重等級、規格引用、代碼位置）
- Review Notes 裁決結果
- 測試品質評估
- 建議的修正項目

## STOP Gate

| 條件 | 動作 |
|------|------|
| review_report.md 完成 | **STOP** — 呈現報告給使用者，由使用者決定修正範圍 |
| 有 MUST-FIX 項目 | 建議返回 Phase 2 處理修正 Task |

## 失敗處理

- 發現規格覆蓋率有缺口 → 列為 MUST-FIX，建議補充 Task
- Review Notes 中的測試問題確認為測試錯誤 → 提出修正建議（由使用者決定是否修改測試）
- 代碼與模板結構不一致 → 判斷是否為合理偏離，若不合理列為 SHOULD-FIX

## Gotchas

- 「看起來沒問題」是最危險的結論 — 你的工作就是找問題 (§7)
- 不要因為 Code Agent 在 review_notes 中說「測試有問題」就直接認同 — 獨立判斷 (§8)
- 代碼實作了規格中不存在的功能 = 幻覺填補 (§6)，要標記為 MUST-FIX
- 過於寬鬆的斷言（如只檢查型別不檢查值）等於沒有測試
- 覆蓋率 100% 不代表品質 OK — 關注 mutation score 而非 coverage
