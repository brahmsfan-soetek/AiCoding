# Phase 0｜規格衝突檢查

**執行者**：任何 Agent（或人工）

**目的**：在 AI 開始任何開發工作之前，消除 DOCX 與 HTML 之間的矛盾。
矛盾 context → LLM 100% 靜默混合而非報錯 (§4)。矛盾消除必須在 AI 介入之前完成。

## 輸入

- 規格書（DOCX）
- 對應的 HTML

## 步驟

### Step 1：逐項比對

將 DOCX 的每個功能描述、欄位定義、狀態值、業務規則與 HTML 對應部分比對。
比對維度：資料表名稱、欄位名稱與型別、狀態碼與列舉值、業務規則邏輯、權限定義、UI 流程。

### Step 2：產出衝突清單

格式見 [examples/conflict-list-example.md](../examples/conflict-list-example.md)。

每個衝突項必須包含：
- 項目名稱
- DOCX 描述（原文引用）
- HTML 描述（原文引用）
- 建議（哪個版本為準、或需要 SA 釐清）
- 狀態：⏳ 待人工裁決

### Step 3：標記 SA 待確認項目

規格本身模糊或缺漏的部分（非 DOCX vs HTML 矛盾，而是 SA 本身沒寫清楚），獨立列出。

每個待確認項必須包含：
- 問題描述
- 出處（DOCX 或 HTML 的具體段落）
- 影響範圍（影響哪些功能/API/資料表）
- SA 回覆：（待填）

### Step 4：等待人工裁決

所有衝突項和 SA 待確認項都解決後，產出**統一規格**（以裁決結果為準的單一版本）。

## 約束

| 約束 | 原因 |
|------|------|
| **不可猜測裁決結果** | §6：資訊不足時 AI 會編造合理但錯誤的內容 |
| **不可跳過此 Phase** | 即使 DOCX 和 HTML 看起來一致，仍需跑完比對流程確認 |
| **不可合併衝突項和 SA 待確認項** | 性質不同：衝突 = 兩份文件矛盾；SA 待確認 = 單份文件不清楚 |

## 產出

- `conflicts.md` — 衝突清單 + 裁決結果
- `sa_pending.md` — SA 待確認清單 + 回覆
- 統一規格 — 衝突已解決的版本，後續 Phase 的**唯一輸入源**

## STOP Gate

| 條件 | 動作 |
|------|------|
| 衝突=0 且 SA待確認=0 | 自動進入 Phase 1（通知使用者） |
| 任何項目仍為 ⏳ | **STOP** — 呈現清單給使用者，等待裁決 |

## Gotchas

- 「DOCX 和 HTML 看起來一樣」不等於「沒有衝突」— 細微差異（如列舉值順序、欄位型別）很容易忽略
- 欄位名稱大小寫不一致也是衝突（如 `IM_DOC_DETAIL` vs `im_doc_line`）
- 狀態碼的數量差異特別重要 — 多出或缺少的狀態碼影響整個流程設計
- 不要把「SA 沒寫」歸類為衝突 — 這是 SA 待確認，性質不同
