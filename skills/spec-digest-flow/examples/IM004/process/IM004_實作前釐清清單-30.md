## 一、無法實作的項目（編碼規則 / 來源不明 / 邏輯歧義）

### 1-1 🔴 INV_NO（庫存異動單號）編碼規則未定義

**問題：** 規格統計已標記 INV_NO 編碼規則為「規格未明確定義，需確認」。無 PREFIX 組成、無序號表、無格式範例。

**影響：** IM_DOC_HEADER 的主鍵無法產生，新增功能完全無法實作。

**待確認：**
- PREFIX 格式？是否類似 JE_HEADER_ID 用 `IM` + COMPANY_CODE + PERIOD_YEAR？
- 序號來源是否也用 AGN（AU_GENERAL_NO）表？
- 長度與格式範例？

---

### 1-2 🔴 TXN_ID（庫存交易檔主鍵）編碼規則未定義

**問題：** MM_TRANSACTIONS.TXN_ID 僅標註「系統自編」，無任何編碼規則說明。

**影響：** 過帳流程步驟一（建立 MM_TRANSACTIONS）無法實作。

**待確認：**
- 是否也走 AGN 取號？PREFIX 格式？
- 或為 DB auto-increment？

---

### 1-3 🔴 D02/D03 表頭額外顯示欄位疑似對調

**問題：** 原始規格定義：
- D02（成本中心領用）→ 顯示「成本中心」+「**報廢原因**」
- D03（存貨報廢處理）→ 顯示「成本中心」+「**領用申請**」

語義明顯不匹配——領用單不應填報廢原因，報廢單不應填領用申請。規格統計已標記為不一致 #3。

**影響：** 前端條件欄位顯示邏輯無法確定，表頭 H.REF_DOC_NO / H.DESCRIPTION 寫入對象搞反。

**待確認：** 是否應對調為：
- D02 → 成本中心 + 領用申請（H.REF_DOC_NO）
- D03 → 成本中心 + 報廢原因（H.DESCRIPTION）

### 1-4 🔴 利潤中心（PROFIT_CODE）欄位的顯示條件未明確

**問題：** 表頭欄位表中，利潤中心寫「欄位顯示判斷參考如下 C2.1 章節說明」，但規格中找不到 C2.1 章節的具體定義。規格統計標註「規格未明確列出，需確認」。

**影響：** 無法確定哪些異動類型需要顯示利潤中心欄位。

**待確認：**
- 利潤中心是否在所有類型都顯示？還是只在 D02/D03（同成本中心）？
- D04/D05 是否需要利潤中心？
- D01 是否需要利潤中心？

---

### 1-5 🔴 科目邏輯步驟 3.2 的 TXN_KEY 值不一致

**問題：** 取得費用科目的 SQL 範例寫 `WHERE atk.TXN_KEY = 'INP'`，但邏輯說明寫「傳入步驟 2 取得之 TXN_KEY」= `'OEI'`。兩者衝突。規格統計已標記為不一致 #5。

**影響：** 用 'INP' 與用 'OEI' 查 AU_TXN_KEY 會得到不同的科目，導致傳票借貸科目錯誤。

**待確認：** 費用科目查詢的 TXN_KEY 應為 `'OEI'`？（根據邏輯說明應是 'OEI'，SQL 範例寫錯）

---

### 1-6 🔴 AGN 表格的實際表名與存取方式

**問題：** 規格僅寫 `AGN.PREFIX`、`AGN.CURRENT_NUMBER`，未給出完整表格名稱。規格統計推測為 `AU_GENERAL_NO`。

**影響：** 傳票編碼（JE_HEADER_ID）與可能的 INV_NO/TXN_ID 取號邏輯無法實作。

**待確認：**
- 實際表名是否為 `AU_GENERAL_NO`？
- 表結構（欄位清單、主鍵）為何？
- 取號是否需要鎖定（防止併發重複號）？既有系統是否已有取號 utility 可呼叫？

---

### 1-7 🟡 修改頁面的可修改欄位範圍不明

**問題：** 表頭與表身欄位表中「修改頁可修改」欄幾乎全為空白（無勾選），但空白的語義不明確——是「不可修改」還是「未定義」？

若修改頁面所有欄位都不可修改，那「編輯」功能形同只能新增明細行（如果連明細都不能改，那編輯頁面等於只能看）。

**待確認：**
- 修改頁面，表頭的異動類型、日期等是否可修改？
- 修改頁面，既有明細行的數量、工廠、庫位等是否可修改？
- 修改頁面是否只能「新增 / 刪除明細行」但不能修改既有行？

## 二、規格定義不足需補充

### 2-1 🔴 庫位 LOV 2.3 缺少 PLANT_CODE 篩選條件

**問題：** LOV 2.3 SQL 只篩選 COMPANY_CODE 和 MATERIAL_CODE，但連動性欄位表明確寫「選擇工廠 LOV (2.2) → 更新庫位 LOV (2.3) 篩選條件」。目前 SQL 未包含 `AND mms.PLANT_CODE = :PLANT_CODE`。

**影響：** 不加工廠篩選，使用者選了 P001 工廠卻看到 P002 工廠的庫位，造成資料錯誤。

**建議：** LOV 2.3 SQL 補加 `AND mms.PLANT_CODE = :PLANT_CODE`。

---

### 2-2 🟡 LOV 2.3 的 AVL_QTY > 0 條件對入庫類型不合理

**問題：** LOV 2.3 篩選條件 `AND mms.AVL_QTY > 0` 對出庫類型（D01/D02/D03/D05）合理，但對入庫類型 D04（雜收）不合理——雜收的目標庫位可能現有庫存為 0。

**影響：** D04 若選到庫存為 0 的庫位，LOV 查不到任何選項，無法操作。更嚴重的是，若該物料從未在某庫位存放過（MM_MATERIAL_STORAGE 無記錄），也無法選擇。

**待確認：**
- D04 的庫位 LOV 是否應改用 IM_SLOC（LOV 2.4 的結構）而非 MM_MATERIAL_STORAGE？
- 或者 D04 的 LOV 2.3 SQL 移除 `AVL_QTY > 0` 條件？
- D04 過帳時若 MM_MATERIAL_STORAGE 無記錄，是否需要 INSERT 新記錄？

---

### 2-3 🔴 D01 調撥——目標端庫存記錄不存在時的處理

**問題：** D01 過帳需要更新目標端的 MM_MATERIAL_STORAGE（增加 AVL_QTY）。但如果該物料從未在目標工廠/庫位存放過，MM_MATERIAL_STORAGE 無記錄。

**影響：** 直接 UPDATE 會影響 0 筆記錄，庫存沒有增加。

**待確認：**
- 是否需要「UPDATE or INSERT」邏輯（UPSERT）？
- 若需要 INSERT，新記錄的其他欄位預設值為何？
- MM_MATERIAL_BATCH 同理——目標端無批次記錄時是否需要新建？

---

### 2-4 🟡 D01 調撥——目標端批次號的處理

**問題：** 規格確認 #2 定義「出庫時→LOV 挑選、入庫時→手輸」。D01 同時有出庫端和入庫端。但 UI 表身中只有一個「批次號」欄位，D01 的目標端沒有獨立的「目標批次號」欄位。

**影響：** D01 批次物料調撥時，來源批號由 LOV 挑選，但目標端批號寫什麼？

**待確認：**
- D01 是否沿用來源批號到目標端？（即出入庫用同一個批次號）
- 或者目標端需要獨立輸入批號？（則 UI 需增加「目標批次號」欄位）

---

### 2-5 🔴 併發控制的實作機制未定義

**問題：** 最新規格確認 #6 明確要求「後面的人不能正常打開單據」，但未定義技術實作方式。

**影響：** 實作方式不同，複雜度和可靠性差異極大。

**待確認：**
- 使用資料庫鎖（DB row lock）？前端輪詢？Redis 分散式鎖？IM_DOC_HEADER 增加 LOCKED_BY / LOCK_TIME 欄位？
- 鎖的釋放時機：儲存時釋放？離開頁面時釋放？超時自動釋放？
- 如果使用者瀏覽器崩潰 / 斷網，鎖如何釋放？（心跳機制？鎖超時？）
- 既有系統是否已有類似的併發控制元件可複用？

---

### 2-6 🟡 D01 來源與目標相同時是否卡控

**問題：** 規格未定義 D01 調撥時「來源工廠 + 庫位」與「目標工廠 + 庫位」是否可以相同。

**影響：** 若允許相同，會產生無意義的交易記錄（自己調給自己）。

**待確認：** 是否需要前端或後端卡控「來源 ≠ 目標」？

---

### 2-7 🟡 刪除方式：物理刪除 or 邏輯刪除

**問題：** 刪除流程寫「刪除 IM_DOC_DETAIL + IM_DOC_HEADER」，暗示物理刪除。但許多 ERP 系統偏好邏輯刪除（標記狀態 = 'D'）以保留稽核軌跡。

**待確認：**
- 使用 DELETE 物理刪除？或增加刪除狀態（如 DOC_STATUS = 'D'）？
- 是否需要記錄刪除操作的日誌（誰在何時刪了什麼）？

---

### 2-8 🟡 過帳流程——庫位/批次庫存更新的步驟順序

**問題：** 規格統計步驟二-附指出「主規格未明確列出庫位/批次庫存更新步驟」。邏輯上必要，但執行順序影響正確性：
- 若先更新 MM_MATERIAL_STORAGE 再呼叫 usp_UpdateMovingAverageCost → 後端庫存充足檢查用哪個值？
- 成本程序（CO003-1）使用 MM_MATERIAL_VALUATION 的 STOCK_QTY（總量），與 MM_MATERIAL_STORAGE 的 AVL_QTY（庫位量）是分開管理的。

**待確認：**
- 庫位/批次庫存更新應在呼叫成本程序之前還是之後？
- 後端庫存充足檢查是檢查 MM_MATERIAL_STORAGE.AVL_QTY（庫位級）還是 MM_MATERIAL_VALUATION.STOCK_QTY（物料級）？還是兩者都檢查？

---

### 2-9 🟡 傳票金額是整單合計還是逐明細行

**問題：** GL_JOURNAL_DETAIL 的 ENTER_AMT / ACCOUNT_AMT 定義為 `SUM(D.TOTAL_AMOUNT) GROUP BY D.INV_NO`，表示整張單據所有明細行的金額加總後作為一筆分錄金額。

這意味著不管有多少明細行，一張傳票只產生 **2 筆** JE Line（存貨 + 費用）。但如果不同明細行的物料有不同的存貨科目（雖然評價類別一致性檢查確保了 VAL_CLASS 相同）...

**待確認：**
- 確認一張異動單只產生一張傳票、兩筆分錄（一借一貸）？
- 不會出現按明細行逐筆產生分錄的情況？

---

### 2-10 🟡 驗證鈕的檢查範圍

**問題：** 規格提到「驗證鈕：針對過帳鈕寫入傳票前的資料取得有問題時，顯示錯誤訊息」，但未列出具體檢查項目清單。

**待確認：** 驗證鈕是否等同於「執行過帳的所有前置檢查 + 科目邏輯查詢，但不實際寫入」？具體是否包含：
- 會計期間開放檢查（GPS.STATUS = 'O'）？
- 後端庫存充足檢查？
- 科目查詢（AU_MVTXN_MAPPING → AU_TXN_KEY）能否成功取得？
- LEDGER_GROUP_CODE / LEDGER_CODE 是否存在？
- POSTING_KEY 是否能查到？
- AGN 序號是否能取得？

---

### 2-11 🟢 查詢頁面的排序與預設行為

**問題：** 規格要求「後端查詢全都不做分頁和排序」，但查詢頁面未定義：
- 預設排序方式（按文件日期倒排？按文件編號？）
- 篩選條件是否全部為選填（全空時查全部？）
- 查詢結果數量上限？（不分頁的情況下，如果有上萬筆會有效能問題）

**建議：** SA 補充預設排序邏輯及查詢上限。

### 2-12 🟢 成本計算摘要區塊的欄位定義模糊

**問題：** 規格統計中「成本計算摘要」的「成本來源」欄位描述為「評價類別描述（例：'1 - 目標庫位 (1)'）」，但此描述語義不清——「目標庫位 (1)」不像是評價類別的描述。

從畫面截圖看，成本計算摘要區塊顯示：

**待確認：** 「成本來源」的顯示格式與資料來源。是 `CONCAT(adp.PF_VALUE, ' - ', mmv.VAL_CLASS)` 嗎？

## 四、其他需對齊事項

### 4-1 🟡 過帳步驟的順序確認

**問題：** 規格統計整理的過帳步驟順序為：
1. 前置檢查 + DOC_STATUS → 'P'
2. 建立 MM_TRANSACTIONS
3. 更新庫位/批次庫存（MM_MATERIAL_STORAGE / MM_MATERIAL_BATCH）
4. 呼叫 usp_UpdateMovingAverageCost（更新 MM_MATERIAL_VALUATION + 寫日誌）
5. 產生傳票（GL_JOURNAL_HEADER + GL_JOURNAL_DETAIL）
6. 更新 AGN 序號
7. COMMIT / ROLLBACK

**待確認：** 此順序是否正確？特別是：
- 步驟 3（庫位庫存更新）應在步驟 4（成本計算）之前還是之後？
- 步驟 1 的 DOC_STATUS 更新是否應放在最後（所有寫入都成功後才改狀態）？

---

### 4-2 🟡 D04 雜收的 @TRANSACTION_VALUE 傳入值

**問題：** 成本程序參數對照寫 `@TRANSACTION_VALUE = 0`，註明「使用當前成本型，由程序內部計算」。CO003-1 的 MIS_REC 行為為 `StockValue + (qty × AvgPrice)`，AVG_PRICE 不變。

但前端 LOV 帶入的單價就是 AVG_PRICE，金額 = qty × AVG_PRICE。如果使用者修改了數量，前端會重算金額。此時傳 0 給成本程序是合理的（讓程序用當前 AVG_PRICE 自行計算）。

**但如果 D04 允許使用者手動輸入不同的單價**（例如雜收的實際成本與帳面移動平均不同），則應改用 `@TRANSACTION_VALUE = 前端金額` 且 TransactionType 不應是 MIS_REC 而應類似 GR。

**待確認：** D04 的單價是否鎖定為 AVG_PRICE（不可修改）？或使用者可輸入不同單價？

---

### 4-3 🟡 JE_LINE_ID 格式範例不一致

**問題：** 規格統計已標記不一致 #6：規格範例 `MP100020250000012647001` 以 MP 開頭，但定義說「取 JE_HEADER_ID + 三碼序號」，JE_HEADER_ID 以 JE 開頭。

**建議：** 改為 `JE10002025000012674001`。待 SA 確認。
