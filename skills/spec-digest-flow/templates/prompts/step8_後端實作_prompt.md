# Step 8：後端實作 Prompt 模板

## 用途說明

**何時使用：** 當 Step 6 前端實作完成、Step 7 的後端任務清單（`{程式編號}_backend_tasks.md`）確認完畢後，在專案 repo 中開啟新的 AI Session 時使用此模板。

**如何使用：**
1. 在專案 repo 中開啟全新 AI Session（不延續前一步驟的對話）。
2. 將下方 Prompt 複製到新的 AI Session。
3. 將所有 `{佔位符}` 替換為實際的程式編號或說明。
4. AI 會先閱讀 `spec/` 目錄下所有文件，再依任務清單一次性完成全部後端實作。

**前置條件：**
- `spec/` 目錄已包含規格統計文件、SA 釐清回覆及後端任務清單等所有文件。
- 後端任務清單（`{程式編號}_backend_tasks.md`）已完成審閱並確認無誤。
- Step 6 前端實作已完成，API 介面規格（路由、請求／回應格式）已確定。

**注意事項：**
- 前後端分開實作，本步驟**僅處理後端**，不修改前端程式碼。
- 規格統計文件為唯一權威來源，如規格統計與其他文件有衝突，以規格統計為準。
- UI 截圖僅供版面排版參考，不得依截圖自行推斷規格統計中未定義的欄位或邏輯。

---

## Prompt 模板

```
請先完整閱讀 spec/ 目錄下的所有文件，再依照後端任務清單實作全部後端程式碼。

【輸入檔案】
請讀取 spec/ 目錄下的所有文件，包含但不限於：
- {程式編號}_規格統計.md（規格統計，唯一權威來源）
- {程式編號}_backend_tasks.md（後端任務清單，本次實作依據）
- {程式編號}_test_cases.md（測試清單，供實作時參考驗證規則）
{以下視情況增減}
- {其他參考文件}（{用途說明}）

【UI 截圖 / HTML 參考說明】
若 spec/ 目錄中包含 UI 截圖或 HTML 原型，僅供參考版面排版與佈局。
欄位名稱、欄位行為及所有功能邏輯均以規格統計文件為準，
不得依據截圖自行推斷規格統計中未明確定義的欄位或業務邏輯。

【實作指令】
依照 {程式編號}_backend_tasks.md 中的任務清單，一次性完整實作所有後端項目。
請自行判斷各任務的相依順序後依序完成，不需等待確認。

【約束條件】
1. 規格統計（{程式編號}_規格統計.md）為唯一權威來源。
   - 如規格統計與其他文件有衝突，一律以規格統計為準。
   - 如規格統計與 UI 截圖有衝突，以規格統計為準。
2. 遵循現有專案的程式碼風格、命名慣例與目錄結構，不自行引入新的架構模式。
3. SQL 查詢（含 LOV 查詢）的篩選條件、回傳欄位及資料來源，依規格統計中各 LOV 章節的定義撰寫，不自行假設。
4. 過帳、核銷、狀態變更等核心業務邏輯，依規格統計「核心業務邏輯」章節的定義實作，不得依個人判斷增減步驟。
5. 涉及多表寫入或跨資源異動的操作，須遵循 LUW（Logical Unit of Work）原則進行交易管理：
   - 同一業務操作的所有寫入須在同一交易內完成。
   - 任一步驟失敗須完整 Rollback，確保資料一致性。
6. 錯誤處理須明確區分業務規則錯誤（回傳可識別的錯誤碼或訊息）與系統例外（記錄 Log 並回傳通用錯誤），回傳格式依現有專案慣例定義。
7. 欄位寫入值（含固定值、系統帶入值、來源欄位對應）依規格統計的欄位對照表實作，不自行增補或省略。
8. 如遇任務清單中有模糊之處，以規格統計的定義為準，並在程式碼中加上說明注解。

【完成後自檢（請逐項確認）】
1. 任務清單中每一個項目是否都已實作完成，無遺漏？
2. 所有 SQL / ORM 查詢的欄位與篩選條件是否依規格統計中的 LOV 定義撰寫？
3. 核心業務邏輯的執行步驟是否與規格統計「核心業務邏輯」章節完全一致？
4. 所有多表寫入操作是否都在同一 LUW 交易內，且有完整的 Rollback 處理？
5. 欄位寫入值是否依規格統計的欄位對照表，沒有自行增補或遺漏欄位？
6. 新增的檔案路徑與命名是否符合現有專案的目錄結構慣例？
```
