# 任務清單 Prompt 模板

## 用途說明

**何時使用：** 已完成 `spec-digest-flow` S1–S4、取得最終版規格統計，並將規格與 UI 截圖搬入專案 repo 後，於專案目錄內執行。

**如何使用：**
1. 執行前 AI 必須先掃描專案既有程式碼與規範（見下方「專案 Context 建立」）。
2. 將所有 `{佔位符}` 替換為實際檔名、程式編號或說明。輸入檔案直接從工作目錄讀取。
3. AI 產出三份任務清單，輸出至使用者指定位置。

**注意事項：**
- 規格統計為本次實作的**唯一權威來源（Single Source of Truth）**。
- UI 截圖僅供參考排版佈局，不得依截圖自行推斷規格統計中未明確定義的欄位或邏輯。
- 任務清單中的 component 名稱、API 命名、目錄結構、測試寫法**必須貼合專案既有慣例**，不得憑空發明。

---

## 專案 Context 建立（執行 prompt 前必做）

在產出任務清單之前，AI 必須先完成以下掃描並將結論融入清單：

1. 讀取專案根 `CLAUDE.md`（若存在），理解專案整體慣例、禁止事項、技術棧。
2. 掃描前端 component 目錄，歸納：
   - 查詢頁、表單頁、Modal、Table 的既有 component 命名與結構
   - 共用元件（如 LOV 下拉、日期選擇、金額欄位）的封裝方式
   - 路由與 i18n 的擺放位置
3. 掃描後端目錄，歸納：
   - Entity / Domain / Processor / LOV Processor 的既有 pattern 與命名
   - 核心業務邏輯（過帳、檢核）的寫法
4. 掃描測試目錄，歸納既有測試案例的撰寫格式。

將掃描結論作為輸出任務清單時的**命名與結構依據**。遇到規格統計要求與專案慣例衝突時，以專案慣例為準，並在清單中註記。

---

## Prompt 模板

```
請先完成【專案 Context 建立】的四項掃描，再閱讀附件的最終版規格統計，依照前端任務清單模板、後端任務清單模板、測試清單模板的結構，同時產出以下三份可執行的任務清單文件：

  1. {程式編號}_frontend_tasks.md
  2. {程式編號}_backend_tasks.md
  3. {程式編號}_test_cases.md

【輸入檔案】
- {最終版規格統計檔名}（最終版規格統計，唯一權威來源）
{以下視情況增減}
- {UI 截圖 PNG 檔名或說明}（UI 截圖，僅供排版版面參考）

【輸入優先級說明】
規格統計為唯一權威來源，所有欄位名稱、欄位行為及業務邏輯均以規格統計文字為準。
UI 截圖（PNG）僅供參考版面排版與佈局，不得依截圖自行推斷規格統計中未明確定義的欄位或邏輯。
兩者有衝突時，以規格統計為準。

【專案慣例優先級說明】
任務清單中所有 component 名稱、檔案路徑、API 命名、目錄結構、測試寫法，必須與專案既有慣例一致。
若規格統計的要求無法直接對應到專案既有 pattern，應在任務清單中明確標記「⚠️ 與專案慣例差異，待確認」，而非自行發明新結構。

【輸出要求 — 前端任務清單】
1. 依前端任務清單模板的 A~F 六大區塊輸出。
2. 區塊 A：TypeScript 類型定義與 API 服務層檔案（路徑須符合專案既有 service 層位置）。
3. 區塊 B：查詢頁面所有元件，含篩選條件欄位明細與查詢結果表格欄位（沿用專案既有查詢頁 component pattern）。
4. 區塊 C：新增/編輯/檢視頁面，含表頭與表身的條件欄位對照表（格式依模板）。
5. 區塊 D：四欄連動邏輯表格（# / 項目 / 觸發動作 / 連動結果）。
6. 區塊 E：前端驗證規則。
7. 區塊 F：i18n 語系檔與路由設定（路徑須符合專案既有擺放位置）。

【輸出要求 — 後端任務清單】
1. 依後端任務清單模板的 A~E 五大區塊輸出。
2. 區塊 A：Entity / Domain 層，對應目標資料庫表（命名沿用專案既有 Entity pattern）。
3. 區塊 B：Processor 層 API 端點，含查詢、取得、新增、修改、刪除、驗證、核心業務動作（如過帳）、下拉。
4. 區塊 C：LOV Processor 層，每個 LOV 獨立列出。
5. 區塊 D：核心業務邏輯，以子編號拆分（如 {Bxx}-1、{Bxx}-2 …），逐步說明前置檢查至提交的完整流程。
6. 區塊 E：SP 替換評估（可選），評估是否可改呼叫資料庫預存程序。

【輸出要求 — 測試清單】
1. 依測試清單模板的 A~I 大區塊輸出。
2. 每個測試案例格式：| # | 測試案例 | 預期結果 |
3. 涵蓋：基礎 CRUD、編碼取號、驗證規則、業務流程各步驟、交易一致性/併發、LOV/連動邏輯。
4. 測試案例需足夠具體，可直接作為測試工程師的執行依據。

【完成後自檢（請逐項確認）】
1. 三份文件是否已全部產出，且命名符合 {程式編號}_*.md 格式？
2. 規格統計中每個 API 端點（查詢、取得、新增、修改、刪除、業務動作），後端清單是否都有對應的 Processor？
3. 規格統計中每個 LOV / 下拉，後端清單是否都有對應的 LOV Processor？
4. 核心業務邏輯（如過帳）的子步驟，是否與規格統計中描述的完整流程一致，無跳躍或遺漏？
5. 前端連動邏輯（LOV 串接、欄位聯動），測試清單是否有對應的 LOV/連動邏輯測試案例？
6. 是否有任何欄位或業務邏輯只出現在 PNG 截圖而未出現在規格統計中？若有，應標記「⚠️ 規格統計未定義，待確認」而非直接實作。
7. 清單中所有檔案路徑、component 命名、API 命名是否都與專案既有慣例一致？若有差異是否已標記「⚠️ 與專案慣例差異，待確認」？
```
